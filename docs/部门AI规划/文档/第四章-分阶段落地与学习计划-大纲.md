# 四、分阶段落地与学习计划（大纲）

> 本章将"学什么"和"做什么"整合到同一推进路径中，每阶段明确：目标、学习内容、投入、产出。
>
> **说明**：
> - 本章不设定具体时间节点——推进节奏应由部门负责人根据团队情况和资源决定
> - 本章涉及的能力（Rule/Skill/Memory/提示工程）已在第二章详细说明，此处聚焦**先做什么、后做什么**，不重复概念解释
> - 第三章的风险分析应作为本章的前置阅读——了解风险后再规划推进

---

## 设计思路

部门每位开发者的编码风格、常用场景、对 AI 的接受程度各有不同。强行要求全员以统一节奏推进，效果往往不好——有人觉得太快跟不上，有人觉得太慢没意思。

因此，本章的设计思路是：**以开发者个人为单位推进，而非以团队为单位统一推进**。除第一阶段的安全红线需要团队共同遵守外，后续阶段的 Rule、Skill、Memory 均由开发者个人按自己的节奏和需求来。唯一需要团队层面统一的，是部分影响面大的项目级 Rule。

```
                 ┌──────────────────────────────────────────┐
                 │          项目级 Rule（团队统一）            │
                 │     安全规范  ·  接口格式  ·  技术栈约束     │
                 └──────────────────────────────────────────┘
                                    ↑
                    多个开发者有相同需求时，合并为项目级
                                    ↑
  ┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐
  │   开发者 A        │  │   开发者 B        │  │   开发者 C        │
  │                  │  │                  │  │                  │
  │ 个人 Rule        │  │ 个人 Rule        │  │ 个人 Rule        │
  │ 个人 Skill       │  │ 个人 Skill       │  │ 个人 Skill       │
  │ 个人 Memory      │  │ 个人 Memory      │  │ （按需）          │
  └──────────────────┘  └──────────────────┘  └──────────────────┘
```

**推进原则**：

- 第一阶段（安全红线 + 提示习惯）全员参与，这是共同基础
- 第二至第四阶段，每个开发者按自己的节奏推进，不强制同步
- 当某个 Rule 被多人证明有价值时，才合并为项目级 Rule
- 每个人的 Skill 和 Memory 是个人工具，不强制共享，但鼓励交流

---

## 阶段总览

```
第一阶段              第二阶段              第三阶段              第四阶段
建立基础              个人规则整理            个人工作流整合          个人能力扩展
──────────────────────────────────────────────────────────────────────────
安全红线 + 提示习惯     整理自己的开发规范       固化自己的重复流程       沉淀自己的项目认知
（全员共识，共同遵守）    （每个人梳理自己的）      （每个人整合自己的）      （每个人按需扩展）
```

## 第一阶段：建立基础

**定位**：全员参与，建立安全使用 AI 的共识和基本的提示词习惯。这是所有后续阶段的基础。

**目标**：知道什么不能做（安全红线），学会怎么问（提示习惯）。

### 学习内容

| 内容 | 说明 | 方式 |
|------|------|------|
| 三条安全红线 | 敏感信息脱敏、AI 代码必 Review、关键决策标注 | 团队一次共识即可，不需要反复强调 |
| 提示工程四要素 | 背景（Context）、角色（Role）、目标（Goal）、约束（Constraints） | 在每天的实际使用中有意识地练习，不需要专门培训 |
| AI 的基本行为特点 | 幻觉、上下文丢失、知识截止——什么时候该信、什么时候该验证 | 阅读第三章 3.1 节，边用边体会 |

### 核心动作

1. **建立安全红线共识**：三条红线在团队内明确并执行。这是团队层面唯一需要共同遵守的规则
2. **每个人整理自己的常用提示模板**：从自己最高频的场景开始——修 Bug 的整理修 Bug 模板，写接口的整理写接口模板。模板是自己的，风格自己定

### 投入

零额外成本。Qoder 账号已有。

### 产出

- 安全红线团队共识
- 每个人能稳定写出包含"背景+目标+约束"的结构化提示词

### 阶段完成的标志

你不再"随口问"AI，而是习惯性地交代背景和约束——并且能感受到产出质量的明显差异。

## 第二阶段：个人规则整理

**定位**：每个人梳理自己的编码习惯和常见返工点，把"每次都要手动提醒 AI"的约束写成个人 Rule，让 AI 自动遵守。

**前置条件**：已经习惯了结构化提示词（第一阶段），对自己常犯的"低级错误"有了感知。

**目标**：AI 在跟你对话时，自动使用你的技术栈、命名习惯、代码风格。不用每次都手动交代"我用 Java 8""接口返回 `{code, data, msg}`"。

### 核心思路：从"自己的返工点"出发

每个人写代码的习惯不同，容易犯的"低级错误"也不同。Rule 应该从每个人的实际返工点出发，而不是照搬别人的配置。

怎么找自己的返工点？回顾你最近被 Code Review 指出过的问题：

- 后端：SQL 拼接？忘记事务注解？金额用了 double？接口格式不统一？
- 前端：组件命名方式不一致？API 调用没有错误处理？表单校验遗漏？

**反复被指出的问题 = 最适合写成个人 Rule 的候选。**

### 学习内容

| 内容 | 说明 | 方式 |
|------|------|---------|
| Rule 的概念 | Rule 是什么、和提示词里手动写约束有什么区别 | 阅读第二章 2.3.1 |
| 个人 Rule 的配置 | 在 Qoder 中如何配置用户级 Rule | 参考 Qoder 官方文档 |
| 怎么判断"该写成 Rule" | 高频出现的、明确无争议的、不需要每次灵活判断的 = 适合写成 Rule | 从自己的 Code Review 记录中筛选 |

### 核心动作

1. **回顾自己的返工记录**：翻最近 2-3 个月 Code Review 中自己被指出的问题，找出重复出现的
2. **先写 2-3 条个人 Rule**：从最频繁的返工点开始，别贪多。Rule 写多了反而互相干扰
3. **用 2 周，看效果**：哪些 Rule 确实让你的返工减少了？哪些没感觉？没感觉的就删掉或调整

### 项目级 Rule：什么时候从"个人"变成"团队"

大部分 Rule 是个人的。但当同一个 Rule 被团队里多个人独立写了，说明这是团队的共性问题，应该合并为项目级 Rule（所有人自动生效）。

| 类型 | 范围 | 示例 | 谁来维护 |
|------|------|------|---------|
| **个人 Rule** | 只对自己生效 | "我的接口一律用 `{code, data, msg}` 格式" | 自己 |
| **项目级 Rule** | 项目所有人自动生效 | "禁止拼接 SQL 字符串，必须使用参数化查询" | 由发现共性问题的人提议，团队确认后合并 |

合并为项目级 Rule 的条件：
- 团队中 ≥2 人独立写了类似的 Rule
- 或者某条规则涉及**安全**（如 SQL 注入），不需要等多人验证，直接设项目级

### 投入

每人花 30 分钟回顾自己的 Code Review 记录 + 10 分钟写 2-3 条个人 Rule。

### 产出

- 每人 2-5 条个人 Rule（按需增减）
- 如有共性问题，产出 3-5 条项目级 Rule

### 阶段完成的标志

你不再需要在每次对话中手动写"用 Java 8""返回 `{code, data, msg}`""金额用 BigDecimal"——AI 自动就知道了。

## 第三阶段：个人工作流整合

**定位**：每个人把自己工作中最重复、最机械的流程，固化为自己的 Skill。这是个人的效率工具，不是团队任务。

**前置条件**：个人 Rule 已稳定运行一段时间，对 AI 的交互模式有了基本感知。你自己觉得"有些重复流程可以自动化了"——而不是被要求去做。

**目标**：把 1-2 个自己最常遇到的重复性工作流程固化为 Skill。

### 核心思路：先问自己三个问题

在动手写 Skill 之前，先想清楚：

1. **我有什么工作是一遍一遍重复做的？**（比如每次新模块都要从零写 Controller/Service/Mapper）
2. **这个流程的步骤是固定的吗？**（如果每次都有大量变化和判断，不适合做成 Skill）
3. **花 1-2 周写这个 Skill，值得吗？**（如果一年只用两三次，不值得；每周都用，值得）

**如果这三个问题你都能明确回答——这个场景适合做成 Skill。**

### 学习内容

| 内容 | 说明 | 方式 |
|------|------|---------|
| Skill 的概念 | Skill 是什么、和普通对话的区别 | 阅读第二章 2.3.2 |
| Skill 的设计思路 | 定义输入输出、设计执行步骤、内建质量检查 | 参考 Qoder 官方文档，自己动手试 |
| 参考已有的 Skill 结构 | 看看别人怎么设计的（不一定要用，了解思路即可） | Qoder 社区或团队内部分享 |

### 核心动作

1. **找出自己的 1 个高频重复场景**：后端最常见的是标准 CRUD 接口生成，前端最常见的是根据接口文档生成列表/表单页面。选一个自己最需要的
2. **自己试着写**：不需要第一次就完美。先写一个能跑通的版本，用几次，发现哪里不好再改
3. **自己用就行**：这个 Skill 是你自己的工具，不需要给别人用，不需要写文档，不需要团队评审。好用就继续用，不好用就改或扔掉

### 投入

完全自主决定。有兴趣就投入业余时间试试，没兴趣就保持第二阶段的状态，不影响日常工作。

### 产出

- 1-2 个你自己在用的 Skill（不需要团队共享，除非你自己想分享）

### 阶段完成的标志

你发现自己做某类重复任务时，第一反应是"用我的 Skill"，而不是手写或每次重新描述。

> **注意**：
> - Skill 是**个人效率工具**，不是团队考核指标。没有人必须写 Skill
> - 如果你觉得现在的使用方式已经够好了，停在第二阶段完全没问题
> - 如果你写了 Skill 但觉得不好用，说明这个场景可能不适合固化——这不是失败，是排除了一个选项

## 第四阶段：个人能力扩展

**定位**：在 Rule 和 Skill 都稳定后，按自己的兴趣和需求进一步扩展。同样以个人为单位，不强制、不同步。

**前置条件**：自己觉得前三个阶段已经够用了，但对更深入的能力有兴趣——而不是被要求去做。

**目标**：按自己的方向深化 AI 使用能力。

### 可以扩展的方向（按兴趣选择）

| 方向 | 做什么 | 适合谁 |
|------|--------|--------|
| **沉淀项目 Memory** | 把项目里自己经常需要反复交代的背景信息（表结构、业务规则、历史约定）记下来，让 AI 自动了解项目上下文 | 参与长周期项目的开发者 |
| **写更多 Skill** | 把更多重复场景固化为 Skill，或者把多个 Skill 串联成流水线（一个的输出直接喂给下一个） | 对自动化有兴趣的开发者 |
| **了解多 Agent 协作** | 了解多个 Agent 怎么分工作业——比如一个 Agent 生成代码，另一个 Agent 审查代码 | 对 Agent 技术本身感兴趣的开发者 |
| **尝试新的 AI 工具** | Qoder 之外的其他 AI 开发工具，看看有没有更适合自己的 | 喜欢尝鲜的开发者 |
| **保持现状** | 继续用好前三个阶段的习惯，不做新尝试 | **完全合理，这是大多数人的选择** |

### 投入

完全自主。Memory 可以和写文档同步（顺手的事），新 Skill 需要投入时间学习。

### 产出

因人而异，没有统一要求。

### 阶段完成的标志

不需要"完成标志"——第四阶段没有终点，是持续的自我迭代。

---

> **本章小结**
>
> 四个阶段的核心是**以个人为单位的渐进式推进**——不强制同步、不搞运动式推广：
>
> 1. 先用好手头的工具（第一阶段：全员基础，安全红线 + 提示习惯）
> 2. 再整理自己的规范（第二阶段：从自己的返工点出发，写个人 Rule）
> 3. 有需要再固化自己的工作流（第三阶段：选自己最高频的场景，写个 Skill 自己用）
> 4. 有兴趣再扩展（第四阶段：按自己的方向深化，不扩展也完全合理）
>
> 每个开发者走多快、走多远，由自己决定。推进的责任不在个人开发者，而在部门负责人——负责人需要做的是提供环境、清除障碍、认可价值。
