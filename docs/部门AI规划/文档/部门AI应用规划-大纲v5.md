## 引言

### 为什么写这份文档

AI 工具正在快速改变软件开发的方式。公司已为研发部门开通了 Qoder 账号，团队成员也在日常工作中开始使用 AI 辅助编码。但"用起来"和"用得好"之间还有很大距离——很多同事把 AI 当成"高级搜索引擎"，没有充分发挥其潜力；也有人对 AI 的能力边界认识不清，存在过度信任或过度排斥的情况。

这份文档的目标是：

1. **梳理场景**：明确研发部门的哪些工作可以结合 AI 提升效率，以及具体怎么做
2. **提供方法**：从基础的提示工程到进阶的 Agent 能力，给出可落地的使用方法
3. **规划路径**：按阶段推进，让团队在每个阶段都有明确的目标、动作和学习内容
4. **认识风险**：对 AI 的局限性保持清醒，避免盲目信任或过度依赖

## 二、AI 工具与使用方法

> 本章回答"使用哪些 AI 工具，如何使用"。结构为：先划红线（规范）→ 再讲当前怎么用好（IDE 对话 + 提示工程）→ 再讲进阶怎么自动化（Agent 能力）。
>
> **说明**：第二章侧重"是什么、怎么用"的静态说明。团队推进的时间节奏和具体动作，详见第三章"分阶段落地与学习计划"。

### 2.1 使用规范建议

> 工具用起来之前，先把红线划清楚。

- **不要直接贴敏感代码**：数据库密码、API Key、服务器 IP 等先用占位符替换
- **AI 生成的代码必须 Review**：AI 是辅助，责任在写代码的人。尤其是后端涉及数据库操作、金额计算、权限校验的代码
- **AI 生成内容标注**：关键决策点标注是否使用了 AI 辅助，便于追溯

### 2.2 IDE 内代码辅助 — 当前阶段的使用方式

> 部门开发同事目前使用 Qoder 的 VS Code / IDEA 插件，核心方式是在 IDE 内通过自然语言对话让 AI 辅助编码。

#### 2.2.1 当前的核心使用方式

当前阶段的核心使用方式就是 **IDE 内自然语言对话**：选中代码或在对话框中描述需求，AI 返回代码建议、解释或修改方案。无需额外工具，公司已提供的 Qoder 即可覆盖。

#### 2.2.2 提示工程（Prompt Engineering）— 从"随便问"到"有目的地问"

**为什么提示词是核心能力**

同样用 AI，好的提示词和随口提问，产出质量差距巨大。提示工程就是学会**给 AI 交代背景、明确目标、设定约束、指定输出格式**。

Agent 的高级能力（Skill、Rule、Memory）本质上是把好的提示词**固化下来复用**。提示词是这一切的基石。

**提示词的四个要素**

| 要素 | 作用 | 示例 |
|------|------|------|
| **背景（Context）** | 告诉 AI 项目环境、技术栈、现状 | "这是一个已上线的 ERP 系统，Java 8 + Spring Boot，不要引入 Java 9+ 语法" |
| **角色（Role）** | 让 AI 以特定身份思考 | "你是一个有后端开发经验的程序员" |
| **目标（Goal）** | 明确要产出什么 | "为出库单模块新增批量导出 Excel 接口" |
| **约束（Constraints）** | 限制范围、格式、禁区 | "不要修改现有表结构，导出字段含单号/日期/状态/操作人，用 EasyExcel" |

**部门典型提示词模板**（后端示例，前端同理替换技术栈即可）

- 修 Bug：`【背景】【问题】【相关代码】【要求】分析根因，最小改动【注意】事务边界`
- 新接口：`【背景】【需求】【约束】【输出】Controller/Service/Mapper 完整代码`
- SQL 优化：`【背景】【慢 SQL + EXPLAIN】【要求】优先索引优化，其次改写 SQL`

**团队落地方式**

- 建立部门 Prompt 模板库（共享文档即可）
- 好的提示词及时共享沉淀

**参考文档**

- Qoder 官方使用指南
- [OpenAI Prompt Engineering Guide](https://platform.openai.com/docs/guides/prompt-engineering)
- [Anthropic 提示词工程指南](https://docs.anthropic.com/zh-CN/docs/build-with-claude/prompt-engineering/overview)

### 2.3 AI 智能体进阶使用

> IDE 内自然语言对话只是起点。当基础对话不能满足"持续理解项目、自动遵循规范、复用固定流程"的需求时，就需要引入 Agent 的三个核心能力：Rule、Skill、Memory。

#### 2.3.1 Rule（规则）— 让 AI 始终遵循部门规范

**是什么**：项目级或用户级的持久化指令，配置后 AI 在每次对话中自动加载。配一次，一直生效。

**部门应用举例**

| 规则类型 | 内容示例 | 效果 |
|---------|---------|------|
| 安全规范 | "数据库操作必须使用参数化查询，禁止拼接 SQL 字符串" | 从源头杜绝 SQL 注入 |
| 接口约定 | "RESTful 接口统一返回 `{code, data, msg}`" | 前后端接口风格一致 |
| 技术栈约束 | "仅使用 Java 8 语法，不引入 Lombok 以外的注解处理" | 避免 AI 引入不兼容语法 |
| 项目约定 | "实体类字段使用下划线命名，驼峰映射" | 新老代码风格统一 |
| 业务规则 | "金额字段用 BigDecimal，禁止 float/double" | 避免浮点精度问题 |

**引入时机**：当前即可开始。先梳理 3-5 条最常见返工点，写成 Rule。

#### 2.3.2 Skill（技能）— 可复用的自动化工作流模板

**是什么**：把重复性工作流程固化为"技能模板"，一键触发，AI 按预设步骤执行。

| 对比维度 | 普通对话 | Skill |
|---------|---------|-------|
| 流程定义 | 每次手动描述步骤 | 固化在 Skill 中 |
| 输入输出 | 随意 | 明确定义格式 |
| 质量检查 | 靠人判断 | 内建验证步骤 |
| 可复用性 | 每次从零开始 | 一次编写，反复使用 |

**典型 Skill 场景**（行业通用模式，非特定实现）

| Skill 类型 | 输入 | 输出 | 核心流程 |
|-----------|------|------|---------|
| 需求→设计 | PRD 需求文档 | 设计文档 + 建表 SQL + API 接口文档 | 提取业务实体和状态流转 → 设计表结构 → 生成标准 CRUD 接口定义 |
| 设计→代码 | API 接口文档 + 参考模块 | 完整后端代码（Controller/Service/Mapper/Domain/VO） | 分析接口定义 → 参考现有模块风格 → 按分层生成代码 → 自验证 |
| 代码审查 | 代码变更 | 分类审查评论（安全/规范/性能/逻辑） | 多维度并行审查 → 综合评分 → 过滤低质量评论 |

**关键设计思路**

- **流水线串联**：上一个 Skill 的输出是下一个 Skill 的输入，减少手动传递
- **质量门禁**：关键节点设检查，不达标则阻断或标记
- **可复用**：同一 Skill 换输入即可用于不同模块

**引入时机**：中期（团队成员熟悉提示工程后），由 1-2 人试点编写，验证有效后推广。

#### 2.3.3 Memory（记忆）— AI 记住项目上下文

**是什么**：跨会话持久化的记忆机制，AI 能记住项目架构、业务术语、历史决策，不用每次从零开始。

**什么值得记**

| 类型 | 示例 | 价值 |
|------|------|------|
| 技术约定 | "接口返回格式为 `{code, data, msg}`" | AI 生成代码自动符合规范 |
| 业务术语 | "订单状态流转规则" | AI 理解业务上下文 |
| 架构决策 | "计费模块采用策略模式" | AI 生成方案与已有架构一致 |
| 历史遗留 | "某旧表字段含义与标准不一致，新表不要参考" | 避免历史问题被延续 |

**如何沉淀**：与日常工作同步——写设计文档时提取关键决策，排查问题时沉淀经验，共享即可。

**引入时机**：中期（3-6 个月），与 Rule 和 Skill 配合使用。

#### 2.3.4 三项能力的关系

Rule、Skill、Memory 不是并列的，而是有层次关系：

```
        ┌─────────────────────────────┐
        │        Skill（工作流）        │  ← 固化重复流程
        │   ┌─────────────────────┐   │
        │   │   Rule（规范约束）    │   │  ← 基础约束，配一次一直生效
        │   │  ┌───────────────┐  │   │
        │   │  │ Memory（上下文）│  │   │  ← 持续积累，越用越深
        │   │  └───────────────┘  │   │
        │   └─────────────────────┘   │
        └─────────────────────────────┘
```

- **Rule** 让 AI 遵守规范（基础层，当前可做）
- **Memory** 让 AI 理解上下文（知识层，持续积累）
- **Skill** 让 AI 执行标准化流程（应用层，基于前两者）

三者递进引入，形成完整的 Agent 辅助开发体系。

## 三、AI 的局限性与风险认识

> 在积极推动 AI 应用的同时，也需要对 AI 的局限性有清醒认识。这不是阻碍 AI 转型的理由，而是为了让 AI 应用走得更稳。

### 3.1 大模型本身的局限性

| 风险点 | 表现 | 应对措施 |
|--------|------|---------|
| **幻觉（Hallucination）** | AI 会自信地编造不存在的 API、配置项、历史事实 | 后端代码涉及数据库操作、金额计算时必须人工验证；引用外部信息时要求 AI 注明来源 |
| **奉承/迎合（Sycophancy）** | AI 倾向于同意用户的观点，即使用户是错的 | 不要把 AI 当权威；关键决策点反向提问"这个方案有什么问题？" |
| **上下文丢失** | 长对话中 AI 会"忘记"早期设定的规则和约束 | 重要约束通过 Rule 固化；复杂任务拆分为多个短会话 |
| **知识截止** | AI 的知识有截止日期，不了解最新框架版本和安全漏洞 | Java/Spring Boot 版本升级、安全补丁等需人工核对官方文档 |

### 3.2 Agent 自动化的风险

| 风险点 | 表现 | 应对措施 |
|--------|------|---------|
| **操作不可逆** | Agent 如具备自动执行能力（操作数据库、修改文件），误操作后果严重 | Agent 初期仅限"建议模式"；涉及写操作必须人工确认 |
| **连环错误** | Agent 的错误输出被下一个步骤放大 | 关键环节设人工检查点；不要让 AI 的输出直接成为另一个 AI 的唯一输入 |
| **权限失控** | MCP 连接业务系统后，AI 的权限边界可能模糊 | 按最小权限原则配置；区分"查询"和"操作"，操作类必须二次确认 |

### 3.3 团队层面需注意的问题

- **过度依赖**：AI 辅助不等于 AI 替代，核心业务逻辑的理解和设计仍需亲自掌握
- **代码质量稀释**：AI 生成代码量大但可能风格不一致、存在隐蔽 Bug，必须 Review
- **业务逻辑盲区**：AI 不了解公司的具体业务流程和历史背景，生成的代码可能语法正确但业务上不成立，必须由熟悉业务的开发者把关
- **技能退化**：长期依赖 AI 写代码可能导致排查深层问题的能力下降，复杂 Bug 仍需手工分析能力

## 四、分阶段落地与学习计划

> **本章大纲已独立为单独文件**：[第四章-分阶段落地与学习计划-大纲.md](第四章-分阶段落地与学习计划-大纲.md)
>
> 以下是本章的阶段总览和设计逻辑摘要，完整内容（每阶段的学习内容、核心动作、投入、产出、阶段完成标志）请参阅独立文件。

### 阶段总览

```
第一阶段              第二阶段              第三阶段              第四阶段
规范使用              规则固化              流程试点              能力扩展
──────────────────────────────────────────────────────────────────────────
使用规范 + 提示工程     Rule 配置             单个 Skill 试点        Skill 推广 + Memory
（全员参与，零门槛）     （低门槛，配完即生效）   （1-2人先行验证）       （试点成功后再推广）
```

### 设计逻辑

- 四个阶段按**门槛从低到高**排列：全员可直接参与 → 配置规则 → 试点 Skill → 推广扩展
- 每个阶段是下一阶段的**前置条件**——团队需先习惯写好提示词，才能理解 Rule 的价值；Rule 稳定运行后，Skill 才有可靠基础
- **不设具体时间节点**——推进节奏由部门负责人根据团队情况和资源决定
- **第三阶段是关键决策点**——试点成功后进入第四阶段，试点效果不理想则不推进。这不是失败，是理性判断
- **MCP 和 RAG 不纳入本章**——属于平台级建设，归入第五章"展望"

### 每阶段概要

| 阶段 | 目标 | 学习内容 | 投入 | 产出 |
|------|------|---------|------|------|
| 一、规范使用 | 从"随便问"到"有目的地问" | 使用规范 + 提示工程四要素 + AI 行为特点 | 零额外成本 | Prompt 模板库 + 使用规范共识 |
| 二、规则固化 | AI 代码自动符合部门规范 | Rule 概念 + 配置方法 + 规则筛选原则 | 每条约 5-10 分钟配置 | 3-5 条项目级 Rule 上线 |
| 三、流程试点 | 验证 Skill 模式在团队的可行性 | Skill 概念 + 设计原则 + 编写方法 | 1-2 人试点，每个 Skill 约 1-2 周迭代 | 1 个验证通过的 Skill + 试点评估报告 |
| 四、能力扩展 | 形成团队自己的 AI 辅助开发方法 | Skill 设计进阶 + Memory 沉淀 + 多 Agent 概念 | 按需投入，Memory 零额外成本 | Skill 库（3-5 个）+ Memory 积累 + 经验分享记录 |

## 五、展望——从研发到全公司

> 本章描述更远期的可能性。研发部门作为公司 AI 转型的先行者，验证方法论后可向全公司推广。

### 5.1 近期可期待：Agent 能力的深度应用

研发部门在熟练使用 Rule + Skill + Memory 后，可以逐步形成一套标准化的 AI 辅助开发流程：

- 需求→设计→编码→审查全链路 AI 辅助
- Prompt 模板库 + Skill 库成为部门知识资产，新人上手更快
- Memory 沉淀使 AI 对项目的理解随时间加深

### 5.2 中期可能：知识库与内部问答（RAG）

- 整合公司制度、部门文档、技术规范，搭建**内部知识问答系统**
- 应用场景：新员工入职问答（"公司的请假流程是什么？"）、技术规范查询（"WMS 的出库流程怎么走？"）、业务知识检索
- 技术方式：基于 Dify/FastGPT 等低代码平台 + RAG（检索增强生成）技术
- **说明**：此项涉及知识库基础设施建设，属于平台型工作，需要架构层面支持。当前阶段以了解概念为主，待条件成熟后推进

### 5.3 远期展望：AI 连接业务系统（MCP）

- 通过 **MCP（Model Context Protocol）** 协议，为 ERP/WMS/HRM 等系统扩展 AI 可调用的接口
- 实现"对话即操作"：在 AI 对话框中直接查询订单状态、生成报表、发起审批
- 从"AI 辅助写代码"升级到"AI 成为业务系统的入口"
- **说明**：MCP 连接业务系统属于平台级建设，需要架构层面的统一规划。研发部门当前阶段的目标是熟练掌握 Agent 能力（Rule/Skill/Memory），MCP 作为了解方向即可

### 5.4 从研发推广到全公司

- 研发先行验证 → 形成可复制的 AI 应用方法论 → 推广到运营、仓储、人事等部门
- 各部门的推广路径类似：先用起来（自然语言对话）→ 用好（提示工程）→ 自动化（Agent 能力）

### 5.3 远期展望：AI 连接业务系统

- 通过 MCP 协议，为 ERP/WMS/HRM 等系统扩展 AI 可调用的接口
- 实现"对话即操作"：在 AI 对话框中直接查询订单状态、生成报表、发起审批
- 从"AI 辅助写代码"升级到"AI 成为业务系统的入口"

> **说明**：作为 Java 业务开发部门，当前阶段的目标是熟练掌握 Agent 能力（Rule/Skill/Memory）。MCP 连接业务系统属于平台级建设，需要架构层面支持，暂不作为部门短期规划。

### 5.4 从研发推广到全公司

- 研发先行验证 → 形成可复制的 AI 应用方法论 → 推广到运营、仓储、人事等部门
- 各部门推广路径类似：先用起来（自然语言对话）→ 用好（提示工程）→ 自动化（Agent 能力）

## 总结

### 核心观点

1. **AI 是工具，不是替代**：AI 辅助开发的核心价值在于提效，而非替代开发者。业务理解、架构设计、关键决策仍需人工把关
2. **场景驱动，不是技术驱动**：先明确"哪些工作能结合 AI"，再选择合适的工具和方法，避免为了用 AI 而用 AI
3. **渐进式引入**：从零成本的"有意识使用"开始，到需要规范的 Rule/Skill，再到需要搭建的 MCP/RAG，每一步都有明确的投入产出比
4. **WMS 项目已验证可行性**：需求→设计→编码→审查的全链路 AI 辅助流水线已在实际项目中跑通

### 立即可做的 3 件事

| 序号 | 动作 | 预期效果 | 建议时间 |
|------|------|---------|---------|
| 1 | 建立部门 Prompt 模板库 | 团队提示词质量提升，减少"随便问" | 本周 |
| 2 | 梳理 3-5 条 Rule | 减少常见返工点（SQL 注入、接口格式） | 2 周内 |
| 3 | 选定 1 个 Skill 试点场景 | 验证 Skill 模式效果 | 1 个月内 |

### 预期收益

- **短期（1-3 个月）**：日常编码效率提升 20-30%（Bug 修复、SQL 编写、代码理解等高频场景）
- **中期（3-6 个月）**：新功能开发周期缩短，代码审查质量提升，新人上手时间减少
- **长期（6-12 个月）**：形成标准化的 AI 辅助开发流程，具备向其他部门输出方法论的能力

### 风险提示

- AI 生成的代码必须 Review，尤其是涉及数据库操作、金额计算、权限校验的场景
- 不要过度依赖 AI，核心业务逻辑的理解和设计能力仍需保持
- Agent 能力引入初期采用"建议模式"，不自动执行，避免不可逆的误操作
