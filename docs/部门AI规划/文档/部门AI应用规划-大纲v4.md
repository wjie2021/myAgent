## 引言

### 为什么写这份文档

AI 工具正在快速改变软件开发的方式。公司已为研发部门开通了 Qoder 账号，团队成员也在日常工作中开始使用 AI 辅助编码。但"用起来"和"用得好"之间还有很大距离——很多同事把 AI 当成"高级搜索引擎"，没有充分发挥其潜力；也有人对 AI 的能力边界认识不清，存在过度信任或过度排斥的情况。

这份文档的目标是：

1. **梳理场景**：明确研发部门的哪些工作可以结合 AI 提升效率，以及具体怎么做
2. **提供方法**：从基础的提示工程到进阶的 Agent 能力，给出可落地的使用方法
3. **规划路径**：按阶段推进，让团队在每个阶段都有明确的目标和动作
4. **认识风险**：对 AI 的局限性保持清醒，避免盲目信任或过度依赖

### 1.1 需求理解与任务拆分

> 先理解清楚要做什么，再动手写代码。需求阶段的遗漏是返工的最大来源。

| 场景 | 现状痛点 | AI 如何介入 | 预期效果 | 实际案例（WMS） |
|------|---------|------------|---------|----------------|
| 需求文档理解 | 需求描述模糊、缺少边界条件，开发时需要反复找产品确认 | 贴需求文档 → AI 梳理业务流程 → 列出边界条件和逻辑冲突 → 从开发角度提问 | 开发前发现更多边界问题，减少返工 | `req-to-api` Skill：输入 PRD → 按 12 个维度提取需求 → 输出设计文档 + 建表 SQL + API 接口文档。用于 WP08 操作费需求分析 |

### 本章小结

以上场景在 WMS 费用管理模块的开发中已有实际验证，形成了完整的 AI 辅助开发流水线：

```
PRD 需求 ──→ req-to-api ──→ cross-check-docs ──→ api-to-code ──→ code-review
           (需求分析+设计文档    (6维度文档一致性      (全套Java代码)    (5 agent
            +建表SQL+API文档)     核验+自动修正)                       并行审查)
```

按引入优先级，建议从以下顺序推进：

1. **最先做**（零成本，当前即可）：Bug 修复与定位、旧代码理解、SQL 编写、需求文档理解 —— 团队已经在用 Qoder 做类似的事，只需有意识地系统化
2. **优先做**（少量规范即可）：代码审查、任务拆分、上线检查清单 —— 建立审查检查清单和 Prompt 模板
3. **逐步做**（需要搭建）：监控告警初筛 —— 需要与运维配合接入日志

## 二、AI 工具与使用方法

> 本章回答"使用哪些 AI 工具，如何使用"。结构为：先划红线（规范）→ 再讲当前怎么用好（IDE 对话 + 提示工程）→ 再讲进阶怎么自动化（Agent 能力）。

### 2.1 使用规范建议

> 工具用起来之前，先把红线划清楚，避免好心办坏事。

- **不要直接贴敏感代码**：数据库密码、API Key、服务器 IP 等先用占位符替换
- **AI 生成的代码必须 Review**：AI 是辅助，责任在写代码的人。尤其是后端涉及数据库操作、金额计算、权限校验的代码
- **AI 生成内容标注**：关键决策点标注是否使用了 AI 辅助，便于追溯

### 2.2 IDE 内代码辅助 — 当前阶段的使用方式

> 部门开发同事目前使用 Qoder 的 VS Code / IDEA 插件，主要方式是在 IDE 内通过自然语言对话让 AI 辅助编码。本节说明如何把这个基础用法用好。

#### 使用方式

当前阶段的核心使用方式就是 **IDE 内自然语言对话**：选中代码或在对话框中描述需求，AI 返回代码建议、解释或修改方案。不需要额外工具，公司已提供的 Qoder 即可覆盖。

#### 提示工程（Prompt Engineering）— 从"随便问"到"有目的地问"

同样用 AI，好的提示词和随口的提问，产出质量差距巨大。提示工程就是学会**给 AI 交代背景、明确目标、设定约束、指定输出格式**，让 AI 从"能用"变成"好用"。

**为什么提示词是核心能力**

```
随便问：                结构化提示词：
"帮我写个接口"    →      "我们用的是 Spring Boot + MyBatis，
                         返回格式统一 {code, data, msg}，
                         请为采购单表 PO_HEADER 生成查询接口，
                         Controller/Service/Mapper 三层，带分页和按状态筛选"
                          
结果：能用但得改半天        结果：基本直接可用
```

Agent 的高级能力（Skill、Rule、Memory）本质上是把好的提示词**固化下来复用**。提示词写不好，后面 Skill 和 Rule 的效果都打折扣。

**提示词的四个要素**

| 要素 | 作用 | 示例 |
|------|------|------|
| **背景（Context）** | 告诉 AI 项目环境、技术栈、现状 | "这是一个已上线的 ERP 系统，Java 8 + Spring Boot，不要引入 Java 9+ 语法" |
| **角色（Role）** | 让 AI 以特定身份思考 | "你是一个有 5 年跨境电商后端开发经验的程序员" |
| **目标（Goal）** | 明确要产出什么 | "为 WMS 的出库单模块新增一个批量导出 Excel 的接口" |
| **约束（Constraints）** | 限制范围、格式、禁区 | "不要修改现有表结构，导出字段包含单号/日期/状态/操作人，用 EasyExcel" |

**部门工作中的典型提示词模板示例**

> 以下为后端场景示例。前端场景同理，将【背景】替换为 Vue + Element UI、【约束】替换为"不修改现有组件结构"即可，此处不赘述。

后端修 Bug 场景（高频）：
```
【背景】ERP 系统，Spring Boot，MyBatis，SVN 管理
【问题】销售单审核后库存没扣减，报错日志如下：...

【要求】分析根因并给出最小改动方案，不要大面积重写
【注意】涉及库存扣减逻辑，务必确认事务边界是否正确
```

后端新接口开发：
```
【背景】WMS 系统，Java 8 + Spring Boot，现有表 wms_outbound_order
【需求】新增批量导出 Excel 接口，导出字段：单号/日期/状态/操作人/备注
【约束】用 EasyExcel，GET 请求，参数放在 URL 上，不要改表结构
【输出】Controller/Service/Mapper 完整代码
```

SQL 优化场景：
```
【背景】ERP 报表模块，MySQL 5.7，数据量百万级
【慢 SQL】[贴 SQL 和 EXPLAIN 结果]
【要求】分析慢的原因并给出优化方案（索引/改写 SQL/代码层调整）
```

**团队落地方式**

- **建立部门 Prompt 模板库**：将上面这类常用场景的提示词模板化，团队成员填空即用
- **好的提示词共享**：遇到一次写得好、产出质量高的提示词，沉淀到模板库

### 2.3 AI 智能体进阶使用（分阶段引入）

> IDE 内自然语言对话只是起点。Agent 工具提供了 Skill（技能）、Rule（规则）、Memory（记忆）等能力，可以让 AI 从"一次性问答"升级为"持续理解项目上下文的工作伙伴"。
>
> 以下结合 WMS 项目的实际使用经验说明。

#### Skill（技能）— 可复用的自动化工作流模板

**是什么**：把重复性的工作流程固化为一个"技能"，一键触发，AI 按预定步骤执行。与普通对话的区别是：Skill 有明确的输入输出定义、固定的执行步骤和质量检查机制，而不是每次靠人手动描述流程。

**WMS 项目中的实际 Skill**

在 WMS 费用管理模块的开发中，我们搭建了 4 个 Skill，形成了一条完整的开发流水线：

```
PRD 需求 ──→ req-to-api ──→ cross-check-docs ──→ api-to-code ──→ code-review
           (需求分析+设计文档    (6维度文档一致性      (全套Java代码)    (5 agent
            +建表SQL+API文档)     核验+自动修正)                       并行审查)
```

| Skill | 输入 | 输出 | 执行步骤 |
|-------|------|------|---------|
| **req-to-api**（需求→API） | PRD 需求文档 | 设计文档 + 建表 SQL + API 接口文档 | 按 12 个维度提取需求 → 数据库设计（建表模板+命名规范）→ API 接口设计（标准 14 接口） |
| **cross-check-docs**（文档一致性验证） | 需求+设计+API+DDL 四份文档 | 核验报告 + 自动修正后的设计文档 | 加载文档 → 6 维度验证（筛选条件/字段覆盖/业务流程/状态枚举/SQL 类型等）→ 分类问题（P0/P1/P2）→ 自动修正 → 重新验证 |
| **api-to-code**（API→代码） | API 文档 + 参考模块 | 完整 Java 代码（Controller/Service/Mapper/Domain/Cmd/Query/VO） | 文档分析 → 参考实现研究 → 代码生成（10 步有序过程）→ 日志标准 → 自验证 → 文档交叉核对 → 常见问题修复 |
| **code-review**（代码审查） | 代码变更 | 置信度评分的审查评论 | 5 个 Sonnet agent 并行独立审查（安全性/规范性/性能/业务逻辑/可维护性）→ Haiku agent 评分 → 80 分以上才发布 |

**关键设计思路**：

- **流水线串联**：上一个 Skill 的输出是下一个 Skill 的输入，减少手动复制粘贴
- **质量门禁**：cross-check-docs 发现问题会阻断流程，确保设计文档质量后才进入编码阶段；code-review 用评分过滤低置信度评论
- **可复用**：同一个 req-to-api Skill 可以用于操作费、仓库杂费、仓租规则等不同模块，只需换输入的 PRD

**引入时机**：团队成员熟悉基础 Prompt 后，由 1-2 人试点编写，验证有效后推广。

#### Rule（规则）— 让 AI 始终遵循部门规范

**是什么**：项目级或用户级的持久化指令，AI 在每次对话中自动加载，确保输出始终符合团队约定。配一次，之后每次对话都生效，不需要反复提醒。

**部门应用举例**

| 规则类型 | 内容示例 | 效果 |
|---------|---------|------|
| 编码规范 | "数据库操作必须使用 MyBatis，禁止拼接 SQL 字符串" | AI 生成的代码自动符合安全要求 |
| 接口约定 | "RESTful 接口返回格式统一用 `{code, data, msg}`" | 前后端接口风格一致 |
| 命名规范 | "Vue 组件用 PascalCase，API 路径用 kebab-case" | 减少 Code Review 中的命名争议 |
| 技术栈约束 | "仅使用 Java 8 语法，不引入 Lombok 以外的注解处理" | 避免 AI 引入项目不支持的写法 |

**使用效果对比**

```
没有 Rule：                        有 Rule "禁止拼接SQL字符串"：

AI 生成代码：                       AI 生成代码：
String sql = "SELECT * FROM         @Select("SELECT * FROM po_header
  po_header WHERE status = '"         WHERE status = #{status}")
  + status + "'";                   List<PoHeader> findByStatus(
                                       @Param("status") String status);
//     ↑ SQL 注入风险                 //     ↑ 自动使用参数化查询
```

**引入时机**：当前即可开始。先梳理 3-5 条最常见的返工点，写成 Rule，让大家体验"配一次、一直生效"的便利。

#### Memory（记忆）— AI 记住项目上下文

**是什么**：跨会话持久化的记忆，AI 能记住项目架构、业务知识、历史决策，不用每次重新交代背景。普通对话是"每次从零开始"，有了 Memory 是"AI 已经了解这个项目"。

**WMS 项目中的实际使用**

在 WMS 费用管理模块开发中，我们沉淀了以下 Memory：

| 记忆内容 | 具体记录 | 用途 |
|---------|---------|------|
| 业务术语 | "费用ID前缀：操作费 CZF、仓库杂费 ZF、仓租规则 CKZF + 4位顺序码" | AI 生成代码时自动使用正确的 ID 前缀 |
| 业务规则 | "14 种计费单位，建议用策略模式（BillingUnitCalculator 接口，每种一个实现类）" | AI 在设计和代码生成时自动采用策略模式架构 |
| 历史决策 | "del_flag 字段统一用 0=正常 1=删除，项目旧表的 0/2 写法是历史遗留不要沿用" | AI 生成新表时自动使用正确的 del_flag 规范 |
| 状态机定义 | "操作费 5 种状态：草稿→确认→生效→无效/已过期，可作废" | AI 生成状态操作代码时自动按正确流转逻辑实现 |
| 开发依赖 | "操作费依赖：FeeType ← Customer 改造 ← 计费引擎" | AI 辅助任务拆分时自动标注依赖顺序 |

**引入时机**：中期开始，和知识库建设同步推进。先建项目级 memory，再逐步扩展。

                   + 深度项目理解)

## 三、学习路径与建议

> 从基础到进阶，按知识层次递进。每个层次说明"学什么"和"怎么学"。

### 3.1 基础：自然语言对话与提示工程

**学什么**

- 掌握在 IDE 内通过自然语言对话让 AI 辅助编码的基本操作
- 理解提示词四要素（背景、角色、目标、约束），能写出结构化的提示词
- 了解 AI 的基本行为特点（如幻觉、上下文丢失），知道什么场景该信任、什么场景该验证

**学习建议**

- **不需要专门培训**：团队已经在使用 Qoder 做自然语言对话，提示工程是在实践中"有意识地优化"，不是从零学起
- **建 Prompt 模板库**：每人贡献 1-2 个自己用过效果好的提示词，沉淀为部门模板，新人填空即用
- **参考文档**：Qoder 官方使用指南、[OpenAI Prompt Engineering Guide](https://platform.openai.com/docs/guides/prompt-engineering)、[Anthropic 提示词工程指南](https://docs.anthropic.com/zh-CN/docs/build-with-claude/prompt-engineering/overview)

### 3.2 进阶：Agent 能力（Rule + Skill + Memory）

**学什么**

- **Rule**：学会配置项目级规则，让 AI 每次对话自动遵循部门编码规范
- **Skill**：学会将重复性工作流程固化为可复用的 Skill 模板
- **Memory**：学会沉淀项目架构、业务术语、历史决策等上下文信息

**学习建议**

- **Rule 最先上手**：梳理 3-5 条部门最常见的返工点（如 SQL 注入、接口返回格式），写成 Rule。配置简单，效果立竿见影
- **Skill 从一个场景开始**：不要一口气写 10 个 Skill。先选一个最高频的场景（如后端 CRUD 接口生成），参照 WMS 项目已有的 req-to-api / api-to-code 等 Skill 结构，写一个模板验证效果，再逐步扩展
- **Memory 与项目文档同步**：每次写技术方案或设计文档时，顺手把关键决策和业务规则沉淀到 Memory 中，不需要额外工作量

### 3.3 拓展：了解更广阔的 AI 生态

> 以下内容不要求团队掌握，但作为技术人员应有所了解，便于在合适的时机引入。

- **MCP（Model Context Protocol）**：让 AI 连接外部系统的标准协议。了解其概念即可，目前主要用于平台型工具，业务部门暂不急需
- **RAG（检索增强生成）**：让 AI 基于私有知识库回答问题的技术。适合未来搭建公司内部知识问答系统
- **多 Agent 协作**：多个 AI Agent 分工协作完成复杂任务。WMS 项目的 code-review Skill（5 个 agent 并行审查）就是一个实际案例

## 五、AI 的局限性与风险认识

> 在积极推动 AI 应用的同时，也需要对 AI 的局限性有清醒认识。这不是阻碍 AI 转型的理由，而是为了让 AI 应用走得更稳。

### 5.1 大模型本身的局限性

| 风险点 | 表现 | 应对措施 |
|--------|------|---------|
| **幻觉（Hallucination）** | AI 会自信地编造不存在的 API、配置项、历史事实 | 后端代码涉及数据库操作、金额计算时必须人工验证；引用外部信息时要求 AI 注明来源 |
| **奉承/迎合（Sycophancy）** | AI 倾向于同意用户的观点，即使用户是错的 | 不要把 AI 当权威；关键决策点反向提问"这个方案有什么问题？"让 AI 给出反对意见 |
| **上下文丢失** | 长对话中 AI 会"忘记"早期设定的规则和约束 | 重要约束通过 Rule 固化；复杂任务拆分为多个短会话；定期让 AI 复述当前理解的任务目标 |
| **知识截止** | AI 的知识有截止日期，不了解最新的框架版本和安全漏洞 | Java/Spring Boot 版本升级、安全补丁等不能仅依赖 AI，需人工核对官方文档 |

### 5.2 Agent 自动化的风险

| 风险点 | 表现 | 应对措施 |
|--------|------|---------|
| **操作不可逆** | Agent 如果具备自动执行能力（如操作数据库、修改文件），误操作后果严重 | Agent 初期仅限"建议模式"（只输出方案不自动执行）；涉及数据库写操作、文件删除等必须人工确认 |
| **连环错误** | Agent 的错误输出被下一个 Agent 或步骤放大 | 关键环节设置人工检查点；不要让 AI 的输出直接成为另一个 AI 的唯一输入 |
| **权限失控** | MCP 连接业务系统后，AI 的权限边界可能模糊 | 按最小权限原则配置 MCP 服务；区分"查询"和"操作"两类接口，操作类必须二次确认 |

### 5.3 团队层面需注意的问题

- **过度依赖**：AI 辅助不等于 AI 替代，核心业务逻辑的理解和设计仍需亲自掌握
- **代码质量稀释**：AI 生成代码量大但可能风格不一致、存在隐蔽 Bug，必须 Review
- **业务逻辑盲区**：ERP/WMS 等系统的业务规则复杂、上下游关联紧密，AI 不了解公司的具体业务流程和历史背景，生成的代码可能语法正确但业务上不成立，必须由熟悉业务的开发者把关
- **技能退化**：长期依赖 AI 写代码可能导致排查深层问题的能力下降，复杂 Bug 仍需手工分析能力

## 六、展望——从研发到全公司

> 本章描述更远期的可能性。研发部门作为公司 AI 转型的先行者，验证方法论后可向全公司推广。

### 6.1 近期可期待：Agent 能力的深度应用

研发部门在熟练使用 Rule + Skill + Memory 后，可以逐步形成一套标准化的 AI 辅助开发流程：

- **需求→设计→编码→审查** 全链路 AI 辅助（WMS 项目已验证可行性）
- **Prompt 模板库 + Skill 库** 成为部门知识资产，新人上手更快
- **Memory 沉淀** 使 AI 对项目的理解随时间加深，而非每次从零开始

### 6.2 中期可能：知识库与内部问答

- 整合公司制度、部门文档、技术规范，搭建**内部知识政策问答智能体**
- 新员工入职后可直接向 AI 提问："公司的请假流程是什么？""WMS 的出库流程怎么走？"
- 技术实现方式：基于 Dify/FastGPT 等低代码平台 + RAG（检索增强生成）技术

### 6.3 远期展望：AI 连接业务系统

- 通过 **MCP（Model Context Protocol）** 协议，为 ERP/WMS/HRM 等系统扩展 AI 可调用的接口
- 实现"对话即操作"：在 AI 对话框中直接查询订单状态、生成报表、发起审批
- 从"AI 辅助写代码"升级到"AI 成为业务系统的入口"

> **说明**：作为 Java 业务开发部门，当前阶段的目标是**熟练掌握 Agent 能力（Rule/Skill/Memory）**，这是投入产出比最高的一步。MCP 连接业务系统属于平台级建设，需要架构层面的支持，暂不作为部门短期规划。

### 6.4 从研发推广到全公司

- 研发先行验证 → 形成可复制的 AI 应用方法论 → 推广到运营、仓储、人事等部门
- 各部门的推广路径类似：先用起来（自然语言对话）→ 用好（提示工程）→ 自动化（Agent 能力）

## 总结

### 核心观点

1. **AI 是工具，不是替代**：AI 辅助开发的核心价值在于提效，而非替代开发者。业务理解、架构设计、关键决策仍需人工把关
2. **场景驱动，不是技术驱动**：先明确"哪些工作能结合 AI"，再选择合适的工具和方法，避免为了用 AI 而用 AI
3. **渐进式引入**：从零成本的"有意识使用"开始，到需要规范的 Rule/Skill，再到需要搭建的 MCP/RAG，每一步都有明确的投入产出比
4. **WMS 项目已验证可行性**：需求→设计→编码→审查的全链路 AI 辅助流水线已在实际项目中跑通，可以作为推广的样板

### 立即可做的 3 件事

| 序号 | 动作 | 预期效果 | 负责人 | 时间 |
|------|------|---------|--------|------|
| 1 | 建立部门 Prompt 模板库 | 团队提示词质量提升，减少"随便问"的情况 | 全员贡献，1 人维护 | 本周 |
| 2 | 梳理 3-5 条 Rule | 减少常见返工点（如 SQL 注入、接口格式） | 后端负责人 | 2 周内 |
| 3 | 选定 1 个 Skill 试点场景 | 验证 Skill 模式的效果 | 1-2 人试点 | 1 个月内 |

### 预期收益

- **短期（1-3 个月）**：日常编码效率提升 20-30%（Bug 修复、SQL 编写、代码理解等高频场景）
- **中期（3-6 个月）**：新功能开发周期缩短，代码审查质量提升，新人上手时间减少
- **长期（6-12 个月）**：形成标准化的 AI 辅助开发流程，具备向其他部门输出方法论的能力

### 风险提示

- AI 生成的代码必须 Review，尤其是涉及数据库操作、金额计算、权限校验的场景
- 不要过度依赖 AI，核心业务逻辑的理解和设计能力仍需保持
- Agent 能力引入初期采用"建议模式"，不自动执行，避免不可逆的误操作
