## 四、分阶段落地与学习计划

> 本章将"学什么"与"做什么"整合为一条由浅入深的推进路径，每个阶段明确目标、学习内容、投入和产出。
>
> **阅读建议**：第三章"AI 的局限性与风险认识"是本章的前置阅读——了解风险之后再阅读落地计划，有助于理解每个阶段为何如此设计。
>
> **说明**：本章涉及的具体能力（Rule、Skill、Memory、提示工程）已在第二章中详细说明其概念和配置方法，此处聚焦**先做什么、后做什么、做到什么程度**，不再重复概念解释。

---

### 4.0 设计思路：以个人为推进单位

在制定推进方案之前，需要先明确一个基本前提：部门中每位开发者的编码风格、常用技术栈、日常任务类型以及对新工具的接受程度各不相同。同一份 Rule 配置，对长期维护旧系统的开发者可能完全无用，对频繁开发新模块的开发者则不可或缺。同一种 Skill，后端开发者关心的是接口生成效率，前端开发者关心的则是页面骨架的搭建速度。

如果以团队为单位制定统一的推进时间表和动作清单，必然出现两种情况：部分成员觉得节奏过快、内容与自己无关；部分成员觉得节奏过慢、已经自行探索了更深入的能力。两种感受都合理——因为每个人的起点和需求确实不同。

因此，本章的推进方案遵循以下原则：

**以开发者个人为推进单位。** 除第一阶段中涉及信息安全和代码责任的三条红线需要团队共同遵守外，后续每个阶段的推进节奏、推进深度、是否进入下一阶段，均由开发者个人自行决定。

**以"自下而上"的方式形成团队级规范。** 大部分 Rule 初始为个人配置——每个开发者从自己的实际返工点出发，配置对自己有用的规则。只有当同一条规则被团队中多位成员独立采用时，才将其提升为项目级 Rule，全员共享。这样做的好处是：每一条项目级 Rule 都经过了实际验证，而非自上而下地"规定"。

**Skill 和 Memory 是个人工具，而非团队任务。** 每个开发者根据自己的工作内容决定是否编写 Skill、沉淀哪些 Memory。一个后端开发者编写的"CRUD 接口生成 Skill"对前端开发者没有直接价值，反过来也是如此。这些工具的受益者是编写者本人，因此是否投入时间、做到什么程度，由个人判断。

下图展示了这种"个人为主、团队为辅"的结构关系：

```
                 ┌──────────────────────────────────────┐
                 │       项目级 Rule（团队统一维护）       │
                 │   安全规范  ·  接口格式  ·  技术栈约束   │
                 └──────────────────────────────────────┘
                                    ↑
                    多位开发者独立验证有效后，合并为项目级
                                    ↑
  ┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐
  │   开发者 A        │  │   开发者 B        │  │   开发者 C        │
  │                  │  │                  │  │                  │
  │ 个人 Rule        │  │ 个人 Rule        │  │ 个人 Rule        │
  │ 个人 Skill       │  │ 个人 Skill       │  │ 个人 Skill       │
  │ 个人 Memory      │  │ 个人 Memory      │  │ （按需选择）       │
  └──────────────────┘  └──────────────────┘  └──────────────────┘
```

按此思路，推进路径划分为四个阶段：

```
第一阶段              第二阶段              第三阶段              第四阶段
建立基础              个人规则整理            个人工作流整合          个人能力扩展
──────────────────────────────────────────────────────────────────────────
安全红线 + 提示习惯     整理自己的开发规范       固化自己的重复流程       沉淀自己的项目认知
（全员共识，共同遵守）    （每个人梳理自己的）      （每个人整合自己的）      （每个人按需扩展）
```

四个阶段按门槛从低到高排列。第一阶段全员参与，建立共同基础；第二阶段至第四阶段由开发者个人掌握节奏，不强制同步推进。

---

### 4.1 第一阶段：建立基础

**阶段定位**：全员参与，建立 AI 使用的安全底线和基本的提示词习惯。这是全部后续阶段的基础，所有开发者无论后续是否深入，都至少应完成此阶段。

#### 4.1.1 目标

本阶段解决两个最基本的问题：

1. **安全底线**：知道使用 AI 工具时什么不能做——不贴敏感代码、AI 生成的代码必须 Review、关键决策标注来源
2. **提问习惯**：从"随口问"过渡到"有目的地问"——在提问前习惯性地交代项目背景、技术栈、约束条件

#### 4.1.2 学习内容

**三条安全红线**

这是团队层面需要达成共识的基础规范，不针对个人风格，仅涉及信息安全和代码责任：

- **敏感信息脱敏**：在将代码或配置粘贴到 AI 对话框之前，将数据库密码、API Key、服务器地址等替换为占位符，生成结果后再替换回来。这不影响 AI 对代码逻辑的理解。
- **AI 生成代码必须 Review**：代码的最终责任在提交者。涉及数据库写操作、金额计算和权限校验的代码，必须在充分理解每一行逻辑后人工确认。
- **关键决策标注来源**：对于架构方案选型、核心算法实现等关键决策点，在代码注释中标注 AI 辅助情况。其目的不在于追责，而是便于后续维护时了解代码的"来历"，以及在出现问题时更快定位根因。

这三条红线不增加额外工作量，但为 AI 应用划定了安全边界。建议在团队内部以一次简短的共识会议明确，后续由团队负责人在日常 Code Review 中留意执行情况。

**提示工程四要素**

结构化提示词包含四个要素：背景（Context）、角色（Role）、目标（Goal）、约束（Constraints）。这四个要素不需要每次都凑齐——简单的代码解释可能只需要"背景 + 目标"，但涉及代码生成的场景建议四个要素都写清楚。

核心原则是：向 AI 提供的信息越充分，其产出越精准。以下通过对比说明：

```
× 不充分的提示词：
"帮我写个接口。"

√ 结构化的提示词：
"我们使用的技术栈是 Spring Boot + MyBatis，接口返回格式统一为
 {code, data, msg}。请为出库单表 wms_outbound_order 生成分页查询接口，
 Controller、Service、Mapper 三层，支持按单号和状态筛选。不要修改
 现有表结构。"
```

两者的差异不在于措辞，而在于信息密度：前者花费五秒钟输入，但后续需要大量时间修改 AI 的产出以适配项目实际；后者多投入了半分钟描述背景与约束，但产出基本可以立刻使用。提示词质量的提升，本质上是将调试时间前置为思考时间。

**AI 的基本行为特点**

在正式使用 AI 辅助开发之前，需要对大模型的基本行为特征建立认知。第三章 3.1 节对此有详细分析，此处仅列出与日常使用最相关的三点：

- **幻觉**：AI 会以高度自信的方式输出实际上不存在的信息——包括不存在的 API 方法、不存在的配置属性以及表面上合理但实际上有逻辑漏洞的代码。涉及数据操作和金额计算的代码不应直接信任。
- **上下文丢失**：在长对话中，AI 对早期设定信息的关注度会逐渐衰减。对话开头设定的"使用 Java 8"可能在五十轮后被遗忘。对于重要的技术约束，不应依赖每次手动提醒，而应通过后续阶段中介绍的 Rule 机制来固化。
- **知识截止**：AI 的知识覆盖范围存在时间边界，不了解截止日期之后发布的框架新版本和已知安全漏洞。涉及版本升级和依赖选择时，应以官方文档为准。

#### 4.1.3 核心动作

**一、建立安全红线共识。** 三条红线在团队内明确并执行，这是团队层面唯一需要共同遵守的规则。

**二、每个人整理自己的常用提示模板。** 从自己最高频的场景出发：日常主要修 Bug 的开发者，整理一个用于 Bug 排查的提示模板；主要做新功能开发的开发者，整理一个用于接口开发的提示模板。模板的格式和风格由个人决定——每个人的工作内容不同，模板自然不同。

举例来说，一个典型的后端 Bug 排查提示模板可能包含以下结构：

```
【背景】项目技术栈、涉及的模块和表
【问题】Bug 的现象描述，附报错日志
【相关代码】出问题的方法或调用链
【要求】分析根因，给出最小改动方案
【注意】需要特别关注的业务约束（如事务边界、金额精度）
```

一个典型的新接口开发提示模板可能包含：

```
【背景】Java 8 + Spring Boot，涉及表名
【需求】接口功能描述，入参和返回字段
【约束】技术方案限制（如使用 EasyExcel、不修改现有表结构）
【输出】Controller / Service / Mapper 完整代码
```

模板的价值在于减少每次提问前的"空白页"时间——有模板之后，只需要按结构填入当前场景的具体信息，不需要每次重新组织思路。

#### 4.1.4 投入

本阶段零额外成本。Qoder 账号已由公司提供；提示模板由个人在自己的工作设备上维护；安全红线的共识可利用部门例会传达。

#### 4.1.5 产出

- 安全红线团队共识
- 每位开发者能稳定输出包含"背景 + 目标 + 约束"的结构化提示词

#### 4.1.6 阶段完成的标志

在日常开发中，向 AI 提问前习惯性地交代项目背景和技术约束——而非随口提问。如果感受到同样的需求描述，结构化提示词的产出质量明显高于随口提问，说明第一阶段已经完成。

### 4.2 第二阶段：个人规则整理

**阶段定位**：每位开发者从自己的实际编码习惯和常见返工点出发，将"每次都要手动提醒 AI"的约束写成个人 Rule，让 AI 在对话中自动遵守。这是从"每次都靠提示词约束"到"AI 自动了解你的规范"的关键一步。

**前置条件**：已经习惯于结构化提示词（第一阶段），对自己常被 Code Review 指出的问题有基本感知。

#### 4.2.1 目标

AI 在与你对话时，自动遵循你的技术栈版本、命名习惯和代码风格。不再需要每次手动交代"我用 Java 8""金额用 BigDecimal""接口返回格式为 `{code, data, msg}`"。

#### 4.2.2 核心思路：从自己的返工点出发

每人写代码的风格不同，容易出现的"低级错误"也不同。定义 Rule 的出发点应该是每个人自己的实际返工记录，而非照搬他人的配置清单。

具体方法：回顾近两到三个月 Code Review 中被指出的问题，找出那些**反复出现**的问题。这些问题通常具有以下特征：规则明确无争议（如"金额字段必须使用 BigDecimal"）、不需要灵活判断（如"禁止拼接 SQL 字符串"）、每次都需要手动提醒 AI。这类问题正是最适合固化为 Rule 的候选。

**后端开发者的常见返工点举例**：

| 返工类型 | 具体表现 | 适合写成 Rule 吗？ |
|---------|---------|------------------|
| SQL 安全 | Mapper 中拼接 SQL 字符串而非使用参数化查询（`#{ }`） | 适合——规则明确，无例外 |
| 金额精度 | 实体类中金额字段使用了 `double` 或 `float` 类型 | 适合——规则明确，无例外 |
| 接口返回格式 | 部分接口返回 `{code, data, msg}`，部分返回 `{success, data, message}`，格式不统一 | 适合——规则明确，无例外 |
| 事务注解遗漏 | Service 方法中有多个写操作但缺少 `@Transactional` | 适合——规则明确 |
| 异常处理方式 | 有的用 try-catch 包裹，有的用全局异常处理器 | 不适合——两种方式各有适用场景，需要灵活判断 |
| 类命名风格 | 有的用 `XxxService`，有的用 `XxxManager` | 不适合——存在合理的命名差异 |

**前端开发者的常见返工点举例**：

| 返工类型 | 具体表现 | 适合写成 Rule 吗？ |
|---------|---------|------------------|
| 组件命名 | 同一项目中混用 PascalCase 和 kebab-case 命名 Vue 组件 | 适合——规则明确 |
| API 调用 | 接口调用缺少错误处理（`.catch()` 或 `try-catch`） | 适合——规则明确 |
| 表单校验 | 新增表单时遗漏必填项校验 | 适合——规则明确 |
| CSS 方案选择 | 组件中混用 scoped style 和 CSS Module | 不适合——存在合理的场景差异 |

筛选原则很简洁：**反复被指出的、规则明确无争议的、不需要每次灵活判断的问题，适合写成 Rule。**

#### 4.2.3 学习内容

| 内容 | 说明 | 方式 |
|------|------|------|
| Rule 的概念与作用 | Rule 与在提示词中手动编写约束的本质区别：Rule 配置一次，所有后续对话自动生效 | 阅读第二章 2.3.1 节 |
| 用户级 Rule 的配置方法 | 在 Qoder 中如何配置仅对自己生效的用户级 Rule | 参考 Qoder 官方文档（https://docs.qoder.com/zh/user-guide/rules） |
| 筛选 Rule 候选的判断标准 | 高频 + 明确 + 无争议；非高频或需要灵活判断的不适合 | 从个人 Code Review 记录中筛选 |

#### 4.2.4 核心动作

**一、回顾自己的返工记录。** 翻阅最近两到三个月 Code Review 中被指出的问题，标注出重复出现的条目。如果缺乏系统的 Code Review 记录，可以回想过去几个迭代中，哪些问题被反复提及。

**二、先写两到三条个人 Rule。** 从最高频的返工点开始，不要试图一次性穷举所有可能。Rule 的数量应以"好用"而非"完备"为目标——过多的 Rule 之间可能产生冲突，反而降低使用体验。

以一位后端开发者为例，假设回顾返工记录后发现以下高频问题：

1. 新接口的 Mapper XML 中经常出现 `${}` 拼接（应使用 `#{}` 参数化查询）
2. 实体类的金额字段偶尔被定义为 `Double`（应为 `BigDecimal`）
3. 接口返回格式有时写成 `{code, data, msg}`，有时写成 `{success, data, message}`

对应的三条个人 Rule 可以分别写成：

```
Rule 1（SQL 安全）：
"所有数据库查询必须使用 MyBatis 参数化查询（#{}），
 禁止使用 ${} 拼接 SQL 字符串。动态排序或分组字段
 必须通过白名单校验。"

Rule 2（金额精度）：
"所有涉及金额的实体类字段必须使用 BigDecimal 类型，
 禁止使用 float 或 double。金额计算时必须指定
 RoundingMode，不可使用默认的四舍五入假设。"

Rule 3（接口返回格式）：
"所有 RESTful 接口返回格式统一为：
 { code: Integer, data: Object, msg: String }。
 code 为 200 表示成功；分页查询结果包含
 total（总数）和 rows（数据列表）字段。"
```

**三、使用两周后评估效果。** Rule 配置完成之后，在后续开发中有意识地留意：哪些 Rule 切实减少了 Code Review 中的同类返工？哪些 Rule 几乎没有发挥作用？没有发挥作用的 Rule 有两种可能：一是该问题本身发生频率低，不值得占用 Rule 配置；二是该 Rule 的描述不够精确，AI 执行效果不佳。无论是哪种情况，都应该调整或删除。

#### 4.2.5 个人 Rule 与项目级 Rule 的关系

大部分 Rule 应以个人为单位配置和维护。但当同一条 Rule 被团队中多位成员独立采用时，意味着这不是某个人的编码习惯问题，而是团队的共性需求。此时应将该 Rule 提升为项目级，对所有成员自动生效。

下表说明了两种 Rule 的区别：

| 维度 | 个人 Rule | 项目级 Rule |
|------|----------|------------|
| **生效范围** | 仅对该开发者本人的对话生效 | 对该项目的所有开发者生效 |
| **配置位置** | 用户级 Rule 配置 | 项目级 Rule 配置文件 |
| **谁来维护** | 开发者本人 | 提议者负责合并，团队共同维护 |
| **示例** | "我的接口返回格式统一用 `{code, data, msg}`" | "项目内禁止拼接 SQL 字符串" |

合并为项目级 Rule 的条件（满足其一即可）：

- 同一规则已被两位或以上的团队成员独立配置为个人 Rule
- 该规则涉及信息安全或数据安全（如 SQL 注入防护），无需等待多人验证，应直接设为项目级

项目级 Rule 的数量应保持在五条以内，仅覆盖影响面最广、争议最小的规范。过于宽泛或存在争议的规则不应纳入项目级——项目级 Rule 的质量比数量重要。

#### 4.2.6 投入

回顾个人 Code Review 记录约需三十分钟；编写两到三条个人 Rule 约需十分钟。此阶段的主要投入是"思考哪些规范值得固化"，而非编写配置本身。

#### 4.2.7 产出

- 每人两到五条个人 Rule（按需增减）
- 如存在团队共性问题，产出三到五条项目级 Rule

#### 4.2.8 阶段完成的标志

在日常对话中，不再需要在每条提示词中手动追加"使用 Java 8""金额用 BigDecimal""返回 `{code, data, msg}`"等技术约束——AI 已经自动遵循这些规范。

### 4.3 第三阶段：个人工作流整合

**阶段定位**：每位开发者将自身工作中最重复、最机械的流程固化为 Skill 模板。这是个人的效率工具——受益者是自己，决策权也在自己。

**前置条件**：个人 Rule 已稳定运行一段时间，对 AI 的交互模式有了基本认知。进入此阶段的驱动力应来自个人判断——"我觉得某个重复流程值得自动化"，而非外部要求。

#### 4.3.1 目标

将自己工作中一到两个高频且流程固定的重复性任务固化为 Skill。

#### 4.3.2 核心思路：判断一个场景是否值得做成 Skill

在投入时间编写 Skill 之前，建议先向自己确认以下三个问题：

**第一，我有什么工作是反复以相同方式执行的？** 典型的候选场景是"每次新建一个模块，都需要按固定分层生成 Controller、Service、Mapper、Domain、VO 等文件"或者"每次拿到第三方 API 文档，都需要编写结构相似的对接代码"。如果某项工作在三个月内只出现了一两次，则不值得为其编写 Skill。

**第二，该工作的执行步骤是否足够固定？** Skill 的核心价值在于将确定性流程自动化。如果一个流程每次执行时都需要大量人工判断和业务决策（例如"根据业务需求设计数据库表结构"），则不适合做成 Skill。如果一个流程的步骤是清晰的、输入输出是明确的（例如"根据建表 SQL 生成对应的 Domain 实体类"），则适合。

**第三，编写和维护这个 Skill 的投入是否值得？** 一个 Skill 从设计到稳定可用，通常需要一到两周的迭代时间（兼职进行）。如果一个流程每周都会遇到，那么投入是值得的；如果一年只用两到三次，则不值得——此时用结构化提示词手动处理更为经济。

如果以上三个问题都能给出明确肯定的回答，说明该场景是 Skill 的合适候选。

#### 4.3.3 举例：一个后端开发者的 Skill 场景判断

假设一位后端开发者正在考虑是否为"标准 CRUD 接口开发"编写一个 Skill。对照以上三个问题：

1. **重复频率**：在当前的 WMS 项目中，费用管理模块包含操作费、仓库杂费、仓租规则等多个子模块，每个子模块的开发模式高度相似——都需要创建 Controller、Service、Mapper、Domain、Cmd、Query、VO 等类。这是一个极高频率的场景。
2. **步骤固定性**：从建表 SQL 到可运行的 REST 接口，执行步骤是明确固定的：分析表结构 → 生成 Domain 实体类 → 生成 Mapper 接口和 XML → 生成 Service 层 → 生成 Controller 层 → 生成 Cmd/Query/VO 等辅助类。每个步骤的输入和输出都有清晰定义。
3. **投入回报比**：手工完成一个子模块的全部代码文件约需半天；如果 Skill 可以将模板代码的生成时间压缩到几分钟，在一个包含五个以上子模块的项目中，投入产出比是可观的。

结论：该场景适合做成 Skill。

#### 4.3.4 学习内容

| 内容 | 说明 | 方式 |
|------|------|------|
| Skill 的概念与设计原则 | Skill 与普通对话的区别；流水线串联、质量门禁、可复用三条设计原则 | 阅读第二章 2.3.2 节 |
| Skill 的编写方法 | 如何定义输入输出格式、设计执行步骤、内建质量检查环节 | 参考 Qoder 官方文档（https://docs.qoder.com/zh/extensions/skills） |
| 参考现有 Skill 的结构 | 了解已有 Skill 的设计思路作为参考（不一定要直接使用，重点在理解其结构） | Qoder 社区或 Qoder 官方示例 |

#### 4.3.5 核心动作

**一、选定一个高频重复场景。** 后端开发者最典型的场景是标准 CRUD 接口生成——流程固定、频率极高、成功标准明确（生成的代码能编译通过且符合项目规范）。前端开发者的典型场景是根据 Swagger 接口文档生成调用该接口的 Vue 列表或表单页面骨架。选择自己工作中最需要的场景，而非选择"看起来最厉害"的场景。

**二、动手编写第一版。** 不需要在第一次就做到完美。先写一个能跑通基本流程的版本，在实际任务中使用两到三次，记录使用中发现的问题——哪里生成的代码总是需要修改？哪个步骤的产出质量不稳定？然后根据实际使用反馈迭代改进。

**三、自己使用即可。** Skill 是个人的效率工具，不需要向团队演示、不需要编写使用文档、不需要经过团队评审。好用就继续用并持续改进；使用一段时间后发现效果不如预期，可以搁置或删除——这不是失败，而是验证了这个场景可能不适合以 Skill 的方式处理。

#### 4.3.6 投入

由开发者个人自主决定。有兴趣则投入时间尝试，没有兴趣继续保持在第二阶段的使用状态即可。一个 Skill 从初版到稳定可用，通常需要一到两周的迭代（利用业余或碎片时间进行），且并非每个开发者都需要进入此阶段。

#### 4.3.7 产出

- 一到两个开发者在实际工作中使用的个人 Skill

#### 4.3.8 阶段完成的标志

在执行某类重复性开发任务时，第一反应是"使用我的 Skill 来处理"，而不是手工编写或每次重新向 AI 描述流程。

#### 4.3.9 补充说明

Skill 是个人效率工具，而非团队考核指标。以下几点值得明确：

- 第三阶段是**可选**的。如果当前的 AI 使用方式已经满足日常需求，保持在第二阶段完全合理。
- 如果编写了 Skill 但使用效果不理想，说明该场景可能不适合以 Skill 方式固化。这不是个人的失败，而是对场景边界的有效验证。
- 如果试点成功、自己对该模式建立了信心，可以考虑进入第四阶段进一步扩展。如果觉得"一个 Skill 已经够用"，停留在此阶段也完全可以。

### 4.4 第四阶段：个人能力扩展

**阶段定位**：在前三个阶段的基础上，按自己的兴趣和需求选择方向深化。此阶段没有统一的推进路径——每位开发者选择的方向、投入的精力、达到的深度均不相同。

**前置条件**：个人判断前三个阶段已经满足日常工作需求，但对某些进阶方向有兴趣——而不是因为被要求推进。

#### 4.4.1 目标

按自己选择的方向，深化 AI 在开发工作中的应用能力。

#### 4.4.2 可选方向

以下列出五个方向，每位开发者按自身情况选择——可以选一个，可以选多个，也可以一个都不选。

**方向一：沉淀项目 Memory**

在同一个项目中长期工作，每次对话都需要向 AI 重新交代项目背景信息——"这张表的名字是什么""这个字段的业务含义是什么""这个模块采用了哪种架构模式"。Memory 的作用是将这些信息持久化存储，让 AI 在后续对话中自动关联。

适合写入 Memory 的信息通常包括以下几类：

| 类型 | 示例 | 价值 |
|------|------|------|
| 技术约定 | 接口返回格式、命名规范、技术栈版本限制 | AI 生成的代码自动符合团队规范 |
| 业务术语 | 核心实体的业务含义、ID 编码规则、状态流转逻辑 | AI 在讨论业务问题时能理解上下文 |
| 架构决策 | 某个模块采用策略模式而非 if-else 的原因、分库分表的规则 | AI 在设计建议中与已有架构保持一致 |
| 历史遗留 | 某张表存在与现行规范不一致的特殊写法及其历史原因 | 避免 AI 将历史特例误认为应当遵循的规范 |

Memory 的沉淀方式可以与日常工作同步进行——在设计文档中做了关键决策时，顺手将决策理由提炼为一到两段话写入 Memory；在排查线上问题时，将沉淀的经验和教训写入 Memory。不需要专门安排大块时间。

此方向适合参与长周期项目的开发者——项目的持续时间越长，Memory 的累积价值越大。短期或一次性任务中，Memory 的投入产出比较低。

**方向二：编写更多 Skill**

在第三阶段已经成功将一到两个场景固化之后，如果对自动化模式有兴趣，可以将更多适合自己的重复性流程编写为 Skill。

更进一步地，可以将多个 Skill 串联为流水线——上一个 Skill 的输出直接作为下一个 Skill 的输入。例如，一个"需求分析 Skill"产出的 API 接口文档，可以直接作为"接口代码生成 Skill"的输入，中间不需要手动格式转换或信息复制。这种串联的价值在于减少手工传递信息时的遗漏和错误。

但需要注意：流水线的环节越多，对每个环节输出质量的依赖越强——上游的一个小错误可能在下游被放大。因此，流水线中的每个 Skill 都应该先独立验证稳定之后，再考虑串联使用。同时建议在关键节点之间保留人工检查环节（详见第三章 3.2.2 关于"连环错误"的分析）。

**方向三：了解多 Agent 协作**

在 Skill 内部可以设计多个 Agent 并行协同工作。例如，一个"代码审查 Skill"可以让多个 Agent 同时从不同维度（安全性、代码规范、性能、业务逻辑、可维护性）独立审查同一段代码变更，再由一个汇总 Agent 综合各维度的审查结果、过滤低置信度意见后输出最终的审查报告。

这种模式的核心价值在于：多个 Agent 各自聚焦一个维度，相比单个 Agent 审查全部维度，覆盖更全面；同时，低质量的审查意见可以通过评分机制被自动过滤，审查者看到的都是经过筛选的高质量反馈。

需要明确的是，多 Agent 协作增加了系统的复杂度——需要设计各 Agent 之间的分工方式、信息传递格式和质量评估标准。因此在团队中，这一方向适合对 Agent 技术机制本身有兴趣的开发者探索，而非全员推广的要求。

**方向四：尝试新的 AI 开发工具**

Qoder 是当前阶段部门统一使用的 AI 开发工具，但市场中 AI 编码辅助工具迭代迅速。在不影响团队统一工具使用的前提下，如果个人有兴趣，可以了解和尝试其他工具，保持对行业动态的感知。需要注意：涉及公司代码的新工具使用，应先在隔离环境中测试，确保信息安全。

**方向五：保持现状**

在完成前三个阶段（或前两个阶段）的基础上，继续以当前方式使用 AI，不做新的尝试。这是完全合理的选择——而且很可能是大多数开发者的实际状态。AI 工具的价值在于辅助开发，而非成为额外的学习负担。当前的使用方式已经足够支撑日常开发提效，不需要为了"进阶"而进阶。

#### 4.4.3 投入

完全由个人自主决定。Memory 的沉淀可以与日常文档工作同步进行，基本无额外时间投入。新增 Skill 或多 Agent 协作的学习需要投入一定业余时间，投入程度取决于个人兴趣。

#### 4.4.4 产出

因方向选择不同而异，无统一要求。

#### 4.4.5 阶段完成的标志

第四阶段无需定义"完成标志"——它是一个持续的自我迭代过程，而非有终点的任务。每个开发者可以随时进入、随时暂停、随时调整方向。

### 4.5 不纳入本章的内容

以下三项能力在第二章中有概念性介绍，但明确不纳入本章的推进计划：

| 能力 | 不纳入原因 |
|------|-----------|
| **MCP（Model Context Protocol）** | MCP 的目标是让 AI 连接外部业务系统（如 ERP、WMS、HRM），属于平台级建设。其实施需要架构层面的统一规划和接口标准化工作，超出了业务开发部门及个人开发者的推进范畴。 |
| **RAG（检索增强生成）** | RAG 需要搭建向量数据库、文档切片管道和检索服务等知识库基础设施，属于平台型工作。当前阶段宜以了解概念为主，实际搭建留待条件成熟时由架构层面推动。 |
| **多 Agent 协作系统** | 独立的 Multi-Agent 系统编排（多个 Agent 之间的任务分配、状态同步、结果汇总）涉及较复杂的系统工程。本章第四阶段方向三中提到的 Skill 内部多 Agent 并行协作（如多维代码审查）属于此概念的局部应用，更系统的 Multi-Agent 架构属于远期探索方向。 |

以上内容将在第五章"展望"中作为远期方向进行描述。

> **本章小结**
>
> 四个阶段的设计遵循"渐进式推进、个人自主决策"的原则：
>
> 1. **第一阶段（建立基础）**：全员参与。确立安全红线和结构化提示词习惯，这是所有后续阶段的前提。
> 2. **第二阶段（个人规则整理）**：从自己的返工点出发，将最高频的编码规范固化为个人 Rule。这是投入最低、见效最快的提效手段——几条 Rule 即可显著减少日常对话中的重复性交代。
> 3. **第三阶段（个人工作流整合）**：选择自己工作中最高频的重复性流程，编写 Skill 固化。此阶段为可选——Skill 是个人效率工具，没有人被要求进入此阶段。
> 4. **第四阶段（个人能力扩展）**：按自己的兴趣选择深化方向——Memory、更多 Skill、多 Agent 协作、新工具，或保持现状。不扩展完全合理。
>
> 每位开发者走多快、走多远，由自己决定。部门负责人的职责不在于督促每个人进入下一阶段，而在于确保安全红线被遵守、为有兴趣深入的成员提供学习资源、以及在项目级 Rule 需要合并时协调共识。
