[0][Heading 1] 引言
[1][Normal] 目前市面AI技术在不断迭代，从早期的纯对话工具不断发展到目前可以支持调用各类工具的AI智能体，为了跟上AI时代的步伐，结合部门实际情况撰写此文档，本文主要以应用与学习两个大的方面展开，另附部分AI工具的使用说明。
[2][Heading 1] 应用场景分析
[3][Heading 2] 需求分析
[4][Normal] 部门主要的开发工作都是由业务驱动，故对业务需求的理解是一切的基础，无论是开发、测试都需要对相关需求由详细的理解，一个边界条件没考虑到，上线后就是一个 Bug。AI 在这个阶段可以充当"第二双眼睛"，帮助开发在动手之前把需求想得更完整。
[5][Heading 3] 需求文档理解
[6][Heading 4] 现状痛点
[7][Normal] 跨境电商的业务规则复杂，WMS 的费用管理模块涉及 14 种计费单位、5 种费用状态流转、汇率折算、核销规则等，需求文档往往只描述了主流程，边界情况需要开发自己补。开发时经常需要反复找产品确认："如果客户选了'全部'怎么处理？""汇率为 0 怎么办？""状态从草稿直接到生效允许吗？
[8][Heading 4] AI介入方式
[9][Normal] 将需求文档贴给 AI，附加项目背景说明，AI 可以：
[10][Normal] 梳理完整的业务流程（含异常分支）
[11][Normal] 列出需求文档中没有明确提到的边界条件
[12][Normal] 从数据一致性角度提问（如"扣减库存时并发怎么办？"）
[13][Normal] 从开发角度提出实现层面的疑问
[14][Heading 4] 实际案例
[15][Normal] 在WMS项目费用模块中，使用AI工具协助分析，AI 按 12 个维度提取需求要素，输出设计文档 + 建表 SQL + API 接口文档。
[16][Normal] 以 WP08 操作费模块为例，需求说明中描述了"新建操作费时需要选择计费仓库和客户"，AI 补充提出了：客户代码支持多选时是否需要"全部"选项、有效时间段是否有最小单位限制、重复规则校验的维度组合是什么等问题，这些在后续开发中都被证实是必须明确的边界条件。
[17][Heading 3] 需求文档一致性验证
[18][Heading 4] 现状痛点
[19][Normal] 一个完整的需求从 PRD 到落地，会经过多份文档：需求文档、设计文档、API 接口文档、建表 SQL。这几份文档分别在不同阶段产出，很容易出现不一致——需求里有 10 个筛选条件，设计文档漏了 2 个，API 文档又多了 1 个需求里没有的字段。这种不一致往往到编码甚至测试阶段才发现，修复成本很高
[20][Heading 4] AI介入方式
[21][Normal] 将相关文档同时提供给 AI，让它交叉比对，可以做到：
[22][Normal] 筛选/查询条件完整性检查
[23][Normal] 列表字段和导出字段覆盖检查
[24][Normal] 详情/明细字段覆盖检查
[25][Normal] 业务流程完整性检查
[26][Normal] 状态枚举和业务规则一致性检查
[27][Normal] SQL 字段类型/精度/索引 vs 需求检查
[28][Heading 4] 实际案例
[29][Normal] 在 WMS 项目中，使用AI工具进行 6 维度文档一致性核验。问题按严重程度分级：P0（阻塞开发，必须立即修正）、P1（需修正，不阻塞但上线前必须解决）、P2（优化建议）。AI 不仅能发现问题，还能自动修正设计文档中与需求不一致的部分。在WMS操作费开发过程中发现过操作费的有效时间段校验逻辑在需求文档和设计文档中的描述不一致，提前修正避免了返工。
[30][Heading 3] 任务拆分
[31][Heading 4] 现状痛点
[32][Normal] 拿到需求后需要拆分为可执行的开发子任务。对业务理解不深时容易遗漏步骤（比如忘了"先建基础数据表再建业务表"），或者不清楚模块间的依赖关系（比如"操作费依赖费用类型和计费引擎，不能先做"）。此时可以选择使用AI工具进行协助任务拆分。
[33][Heading 4] AI介入方式
[34][Normal] 描述需求和现有系统结构，让 AI 生成开发任务清单，标注依赖关系和执行顺序。
[35][Heading 4] 实际案例
[36][Normal] 在 WMS 项目中，WP08 费用管理模块的需求涉及多个子模块，AI 辅助拆分出的依赖链为：
[37][Normal] FeeType（费用类型）← 无依赖，先做
[38][Normal]     ↓
[39][Normal] Customer 实体改造 ← 新增 settle_currency 字段
[40][Normal]     ↓
[41][Normal] 计费引擎 + 汇率工具 ← 核心计算逻辑
[42][Normal]     ↓
[43][Normal] OperationFee（操作费）← 依赖以上三项
[44][Normal]     ↓
[45][Normal] WarehouseMiscFee（仓库杂费）← 同上
[46][Normal]     ↓
[47][Normal] WarehouseRentRule（仓租规则）← 同上
[48][Normal] 
[49][Heading 2] 已上线系统维护
[50][Normal] 已上线系统的维护工作有几个特点：代码可能几经人手、文档不全、改动必须谨慎避免影响现有功能。AI 在以下环节可以提供有效辅助。
[51][Heading 3] 旧代码理解
[52][Heading 4] 现状痛点
[53][Normal] 接手别人写的或几年前的代码，缺少文档，不知道"为什么这样写"。直接改怕踩坑，不改又无法推进。
[54][Heading 4] AI介入方式
[55][Normal] 选中代码，让 AI 解释逻辑、梳理调用链。AI 可以逐行解释代码意图，并标注可能的"坑"。
[56][Heading 4] 实际案例
[57][Normal] WMS 部分旧表（如 b_bank_card）的 del_flag 字段用了 0 代表存在、2 代表删除，而非项目通用的 0/1 规范。通过 AI 分析旧代码和表结构，确认这是历史遗留写法，新模块统一采用 0/1 规范。这类信息如果没有 AI 辅助梳理，新人很容易沿用旧写法导致数据混乱。
[58][Heading 3] BUG修复
[59][Heading 4] 现状痛点
[60][Normal] 这是日常工作中频率最高的场景。收到 Bug 反馈后，需要查看日志、跟踪调用链、定位根因。如果代码经过多人维护、没有人完全熟悉全貌，排查耗时会更长。以 WMS 出库模块为例，一个出库完成后的操作费计费异常，调用链涉及 OutboundOrderService → 计费引擎 → fee_transaction 表，跨越多个 Service 和 Mapper。
[61][Heading 4] AI介入方式
[62][Normal] 将报错日志 + 相关代码贴给 AI，它可以：
[63][Normal] 解析异常堆栈，定位到具体的类和方法
[64][Normal] 结合代码逻辑分析可能的根因
[65][Normal] 给出最小改动方案（不是大面积重写）
[66][Heading 4] 实际案例
[67][Normal] 在WMS操作费开发过程中发现过操作费的有效时间段校验逻辑在需求文档和设计文档中的描述不一致，提前修正避免了返工WMS 费用模块出库完成触发操作费计费时偶发异常。将异常堆栈 + OutboundOrderService + 计费引擎代码贴给 AI，它分析了从出库完成事件到 fee_transaction 写入的完整调用链，定位到问题出在计费引擎处理某些特殊计费单位时的空值判断缺失。AI 给出的修复方案仅涉及 2 行代码的空值判断补充，而非重写整个计费逻辑。
[68][Normal] 
[69][Heading 3] 功能优化/小需求
[70][Heading 4] 现状痛点
[71][Normal] 在现有逻辑上加字段、改业务规则，最大的风险是"改了一处、影响另一处"。尤其是实体类被多个模块引用的情况下，新增一个字段需要评估所有关联代码。
[72][Heading 4] AI介入方式
[73][Normal] 将现有代码贴给 AI，说明要做的改动，AI 会输出增量修改方案，并标注可能受影响的关联代码。
[74][Heading 4] 实际案例
[75][Normal] 在WMS开发中需要在 Customer 实体新增 settle_currency（结算币种）字段。AI 分析了 Customer 实体被引用的所有位置（Controller、Service、Mapper、VO），列出需要同步修改的文件清单，并生成了每个文件的增量代码，避免了遗漏。
[76][Normal] 
[77][Normal] 
[78][Heading 2] 新功能/新系统开发
[79][Normal] 新功能开发的特点是模板代码多、重复性高。AI 在减少机械敲码方面效果明显。
[80][Heading 3] 新功能开发
[81][Heading 4] 现状痛点
[82][Normal] Spring Boot 项目的接口开发模式高度固定：Controller 接收参数 → Service 处理业务逻辑 → Mapper 操作数据库。从表结构到可用的 REST 接口，中间是大量的模板代码。尤其是 WMS 费用模块，操作费有 14 种计费单位、仓库杂费有 5 种杂费类型、仓租规则有 2 种计费类型，每个模块都是 Controller + Service + Mapper + Domain + Cmd + Query + VO 一整套。
[83][Normal] 
[84][Heading 4] AI介入方式
[85][Normal] 提供 API 接口文档 + 参考模块代码，AI 按照现有项目的代码风格和架构分层，生成完整的接口代码。
[86][Normal] 
[87][Heading 4] 实际案例
[88][Normal] 在 WMS 项目中，输入 API 文档 + 参考模块后，AI 按 7 个阶段生成代码——文档分析、参考实现研究、代码生成、日志标准、自验证、文档交叉核对、常见问题修复。
[89][Normal] 以 WP08 操作费模块为例，AI 生成的代码包括：
[90][Normal] Controller：标准 CRUD + 5 个状态操作（确认、发布、作废、无效、复制）+ 批量更新有效期 + 日志查询 + 导出，共 14 个接口
[91][Normal] Service：业务逻辑 + 事务控制 + 重复规则校验（计费仓库 + 客户代码 + 有效时间段不允许相同计费规则）
[92][Normal] Mapper：动态 SQL + 分页查询 + 仓库权限过滤（Query warehouseIds + Mapper XML IN 过滤）
[93][Normal] Domain / Cmd / Query / VO：实体类、命令对象、查询对象、视图对象
[94][Normal] 开发者需要做的是补充业务特有的校验逻辑和计费引擎的策略模式实现，而非从零写起。
[95][Normal] 
[96][Normal] 
[97][Heading 2] 代码质量控制
[98][Heading 3] 代码审查
[99][Heading 4] 现状痛点
[100][Normal] 代码审查的质量取决于审查者的精力和经验。人工审查容易遗漏安全问题（如 SQL 注入、空指针），尤其是代码量大、改动文件多的情况下。部门不建议大面积重构老代码，但新代码的安全性和规范性需要保障。
[101][Normal] 
[102][Normal] 
[103][Heading 4] AI介入方式
[104][Normal] AI 可以作为"第一道审查"，扫描代码变更并标注风险点：
[105][Normal] SQL 注入检查：扫描 Mapper 层，标记拼接 SQL 的位置
[106][Normal] 空指针风险：标记未做 null 判断的链式调用（如 dto.getXxx().getYyy()）
[107][Normal] 异常处理遗漏：检查 Service 方法是否有 try-catch 包裹关键操作
[108][Normal] 事务边界：标记缺少 @Transactional 注解的写操作组合
[109][Normal] AI 不替代人工审查，而是让审查者把精力集中在业务逻辑的正确性上，安全性和代码规范层面的问题交给 AI 初筛。
[110][Normal] 
[111][Normal] 
[112][Heading 4] 实际案例
[113][Normal] 在 WMS 项目中，我们搭建了 code-review Skill：5 个 Sonnet agent 并行独立审查代码变更，每个 agent 从不同维度（安全性、规范性、性能、业务逻辑、可维护性）给出审查意见。审查结果由 Haiku agent 评分，仅 80 分以上的评论才会发布。该技能已用于 WP08 操作费+仓库杂费、仓租费用、仓租规则三个模块的代码审查。
[114][Normal] 
[115][Normal] 这种"多 agent 并行审查 + 评分过滤"的模式相比单次人工审查有两个优势：覆盖维度更全面（5 个 agent 各有侧重），且低置信度的噪音评论被自动过滤，审查者看到的都是高质量反馈。

[120][Heading 1] 工具及使用方法
[121][Heading 2] 使用规范建议
[122][Normal] 在正式推广 AI 工具之前，有几条底线需要团队共同遵守。这不是给 AI 应用设障碍，而是确保 AI 用得安全、用得放心。以下三条规范建议从当前阶段即开始执行：
[123][Heading 3] 不要直接贴敏感代码
[124][Normal] AI 对话工具会将用户输入发送到云端进行处理。虽然主流厂商承诺数据安全，但从信息安全角度出发，不应将敏感信息直接粘贴到对话框中。具体包括：
[125][Normal] 数据库连接密码、API Key、Token 等凭据类信息
[126][Normal] 服务器 IP 地址、内网拓扑等网络信息
[127][Normal] 客户隐私数据、生产环境真实手机号/身份证号等
[128][Normal] 操作方法：粘贴代码前，将上述敏感值替换为占位符（如 DB_PASSWORD、xxx.xxx.xxx.xxx、138****1234），AI 生成结果后再替换回来。这不会影响 AI 对代码逻辑的理解，但能有效保护信息安全。
[129][Heading 3] AI 生成的代码必须 Review
[130][Normal] 这条是底线中的底线。AI 是辅助工具，代码的最终责任在写代码的人。尤其是后端开发中涉及以下场景的代码，必须在充分理解后人工确认才能提交：
[131][Normal] 数据库写操作（INSERT/UPDATE/DELETE）：确认事务边界、数据一致性、WHERE 条件是否正确
[132][Normal] 金额计算：确认精度（BigDecimal 而非 double）、 rounding 策略、币种转换逻辑
[133][Normal] 权限校验：确认接口是否有正确的鉴权逻辑，是否存在越权风险
[134][Normal] 外部接口调用：确认超时设置、重试策略、异常兜底
[135][Normal] 审查时的一个实用技巧：向 AI 反向提问——"这段代码在什么情况下会出问题？""有没有我没考虑到的边界条件？"——让 AI 帮你做第一轮自查，但最终判断仍由人工做出。
[136][Heading 3] AI 生成内容标注
[137][Normal] 对于关键决策点（如架构方案选型、核心算法实现、安全相关逻辑），建议在代码注释或文档中标注是否使用了 AI 辅助。目的不是追责，而是：
[138][Normal] 便于后续维护时了解代码的"来历"
[139][Normal] 出现问题时能更快定位是人工设计缺陷还是 AI 生成偏差
[140][Normal] 积累 AI 辅助开发的经验数据，帮助团队持续优化使用方式
[141][Normal] 标注方式不需要复杂，一行注释即可：
[142][Normal] 
[143][Heading 2]  IDE 内代码辅助
[144][Normal] 当前阶段，团队使用 AI 的核心方式就是 IDE 内自然语言对话——在 VS Code 或 IDEA 中打开 Qoder 对话框，选中一段代码或在对话框中用自然语言描述需求，AI 返回代码建议、解释或修改方案。
[145][Normal] 
[146][Normal] 这种方式的优势在于零门槛、即时反馈。不需要额外安装工具，不需要学习新平台，公司已为全员开通的 Qoder 账号即可覆盖所有场景，此处不多做说明。但它的效果上限取决于使用者的提问质量，本章主要说一下提示词
[147][Heading 2] 结构化提示词
[148][Normal] 同样用 AI，好的提示词和随口提问，产出质量差距巨大。AI代码的准确率跟自己的提示词的准确性有关，提示词描述越清楚AI代码的准确性就越高。
[149][List Paragraph] 随便问： 
[150][Normal] "帮我写个费用流水查询接口"  
[151][Normal] 结果：能用但需要大量修改  
[152][List Paragraph] 结构化提示词：
[153][Normal] "我们用的是 Spring Boot + MyBatis， 返回格式统一 {code, data, msg}， 请为费用流水模块生成查询接口，带分页和按状态筛选，通过Controller/Service/Mapper 三层实现 "
[154][Normal] 结果：基本直接可用
[155][Normal] 以上两者的区别在于：花费 5 秒打字，然后用 20 分钟修改 AI 的产出。还是多花 30 秒描述背景和约束，但省掉了后面的修改时间。提示词质量直接决定了 AI 产出与"直接可用"之间的距离。
[156][Heading 3] 四个要素
[157][Normal] 一个结构化的提示词包含四个要素 ，这四个要素不需要每次都凑齐。简单的代码解释可能只需要"背景 + 目标"，核心原则是：你给 AI 的信息越充分，AI 的产出越精准。
[158][Normal] 
[159][Heading 3] 举例说明
[160][Normal] 场景一：后端修 Bug
[161][Normal] 【背景】ERP 系统，Spring Boot，MyBatis，SVN 管理
[162][Normal] 【问题】[描述 Bug 现象，贴报错日志]
[163][Normal] 【相关代码】[贴 Service 层 + Mapper 层代码，标注出问题的调用链]
[164][Normal] 【要求】分析根因并给出最小改动方案，不要大面积重写
[165][Normal] 【注意】涉及[库存扣减/金额计算/状态流转]逻辑，务必确认事务边界是否正确
[166][Normal] 场景二：后端新接口开发
[167][Normal] 【背景】WMS 系统，Java 8 + Spring Boot，现有表 [表名]
[168][Normal] 【需求】[描述接口功能，列出输入参数和返回字段]
[169][Normal] 【约束】[技术方案约束，如"用 EasyExcel""GET 请求，参数放在 URL 上""不要改表结构"]
[170][Normal] 【输出】Controller/Service/Mapper 完整代码

[184][Normal] 上个章节描述了提示工程相关内容，自然语言对话只是起点。Agent 工具提供了 Rule（规则）、Skill（技能）、Memory（记忆）等能力，可以让 AI 从"一次性问答"升级为"持续理解项目上下文的工作伙伴
[185][Normal] 然后这一切的基础都是提示词
[186][Heading 3] Rule（规则）
[187][Normal] Rule 是项目级或用户级的持久化指令，配置后 AI 在每次对话中自动加载，确保输出始终符合团队约定。
[188][Normal] 与提示词中手动写约束的区别：Rule 配一次，一直生效。
[189][Normal] 对于团队来说，Rule 解决的现实问题是：AI 不知道项目的具体约定——它不知道接口返回格式、不知道命名规范、不知道技术栈限制。没有 Rule，每次都要手动告知；有了 Rule，每次执行指令前AI工具会自动遵守。

[200][Heading 4] 设计思路
[201][Normal] 示例中的 Skill 并非独立设计的，它们之间有意识地遵循了三条设计原则：
[202][Normal] 1. 流水线串联
[203][Normal] 上一个 Skill 的输出直接作为下一个 Skill 的输入。例如 req-to-api 产出的 API 接口文档，直接作为 api-to-code 的输入，中间不需要人工复制粘贴或格式转换。这种设计减少了手动传递信息时的遗漏和错误。
[204][Normal] 2. 质量门禁
[205][Normal] 在关键节点设置质量检查，不达标就阻断流程：
[206][Normal] cross-check-docs 发现 P0 级问题时，输出核验报告并自动修正，修正后重新验证。在文档一致性确认之前，不建议进入编码阶段——因为"文档阶段改一行文字，代码阶段改一堆文件"。
[207][Normal] code-review 用置信度评分机制过滤低质量审查意见，避免"AI 瞎评论导致开发者浪费时间"。
[208][Normal] 3. 可复用
[209][Normal] 同一个 Skill 可以复用到不同业务模块，只需更换输入内容：
[210][Normal] req-to-api 的 需求提取框架，既用于操作费，也用于仓库杂费和仓租规则
[211][Normal] api-to-code 的代码生成流程，不依赖具体业务逻辑，适用于任何标准 CRUD 模块
[212][Normal] code-review 的审查框架，对所有类似模块的代码通用

[214][Normal] Skill的具体配置与使用说明可参考：
[215][Normal] https://docs.qoder.com/zh/extensions/skills
[216][Heading 3] Memory（记忆）
[217][Normal] Memory 是跨会话持久化的记忆机制——AI 能记住项目架构、业务术语、历史决策等信息，在后续对话中自动调用。
[218][Normal] 它与普通对话的区别：
[219][Normal] 普通对话：每次新开会话，AI 对项目一无所知，你需要重新交代"我们项目用 Java 8 + Spring Boot，这张表叫 fee_transaction……"
[220][Normal] 有 Memory：AI 已经"了解"这个项目的技术栈、核心表结构、业务规则和历史决策，新会话中自动关联这些背景知识
[221][Normal] Memory 的价值在长周期项目中尤其明显，WMS 项目是个长线项目，如果每次对话都要重新交代背景，不仅效率低，而且容易遗漏关键信息。有了 Memory，AI 对项目的理解会随着使用逐渐加深。

[228][Heading 3] 小结
[229][Normal] 本章节从使用规范、对话式使用、提示工程、进阶用法介绍了AI工具的使用，同时这个路线也可以作为AI的学习路线。如果理解了本质，其实会发现：Agent 的部分高级能力（Skill、Rule、Memory）本质上是把好的提示词固化下来复用：
[230][Normal] Rule = 每次对话自动注入的背景和约束（不需要每次手动写"我们用 Java 8 + Spring Boot"）
[231][Normal] Skill = 固定的提示词模板 + 执行步骤 + 质量检查（把一次成功的提示词变成可重复触发的流程）
[232][Normal] Memory = 跨会话持久化的上下文（让 AI 记住项目架构，不用每次交代背景）
[233][Heading 3] 拓展
[234][Normal] 智能体还有更为广泛的生态，作为技术人员可以有所了解，可在合适的节点引入。
[235][Normal] 多 Agent 协作：多个 AI Agent 分工协作完成复杂任务。WMS 项目的 code-review Skill（5 个 agent 并行审查）就是一个实际案例
[236][Normal] MCP（模型上下文协议）：让 AI 连接外部系统的标准协议。了解其概念即可，目前主要用于平台型工具，比如可以通过企微或者WPS的MCP服务通过AI协助直接发送消息或者生成文档。
[237][Normal] 未来可为公司的业务系统扩展mcp服务，从而实现通过AI完成实际业务。
[238][Normal] RAG（检索增强生成）：让 AI 基于私有知识库回答问题的技术。适合未来搭建公司内部知识问答系统
