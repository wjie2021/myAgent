## 三、AI 的局限性与风险认识

> 在积极推动 AI 应用的同时，也需要对 AI 的局限性有清醒认识。这不是阻碍 AI 转型的理由，而是为了让 AI 应用走得更稳。
>
> 本章按风险来源从底层到上层分为三个层次：大模型本身的局限 → Agent 自动化的风险 → 团队使用中的注意事项。越往后，越与团队自身的行为习惯相关。

---

### 3.1 大模型本身的局限性

> 以下四个问题是当前所有大模型的共性局限，与使用哪个厂商的模型无关。了解这些局限，才能在合适的场景信任 AI，在不合适的场景保持警惕。

#### 3.1.1 幻觉（Hallucination）— AI 会"自信地编造"

**是什么**

幻觉是指 AI 生成的内容看起来合理、自信，但实际上是虚构的。这不是 AI 在"撒谎"，而是它的工作机制决定了它更擅长生成"看起来像正确答案"的内容，而非"验证后确认正确"的内容。

**具体表现**

| 幻觉类型 | 表现 | 后端开发中的实际风险 |
|---------|------|-------------------|
| **API/库幻觉** | 编造不存在的类名、方法名、配置项 | AI 生成 `@MyBatisSelect` 注解（实际不存在），或使用某版本 Spring Boot 中不存在的配置属性 |
| **逻辑幻觉** | 生成的代码逻辑看起来合理但实际有漏洞 | 生成的分页查询 SQL 在数据量大时出现重复数据，但代码表面上看不出问题 |
| **业务幻觉** | 对业务规则做出错误推断 | 看到 `del_flag` 字段，AI 推断其为"删除标记"，但不知道项目中某些旧表使用了非标准的 0/2 写法 |

**WMS 项目中的教训**

在 WMS 费用模块开发中，曾遇到一个典型案例：AI 生成了一段计费引擎的代码，其中使用了 `BigDecimal.divide()` 但没有指定 `RoundingMode`。代码语法正确、编译通过，但在某些币种汇率转换场景下会抛出 `ArithmeticException`——因为除不尽时没有指定舍入规则。这个问题在本地测试阶段没有被触发，是在测试同事用特定汇率组合测试时才暴露的。

这类问题就是典型的"幻觉型 Bug"——AI 生成的代码在常规情况下运行正常，但在边界条件下出错，而开发者因为代码是 AI 生成的，没有逐行审查，从而漏掉了这个隐患。

**应对措施**

- **涉及数据库操作、金额计算的代码，必须逐行人工验证**。AI 生成的代码只是"初稿"，不是"终稿"
- **引用外部信息时要求 AI 注明来源**（如"这个 API 来自 Spring Boot 哪个版本的文档？"），如果 AI 无法给出确切来源，就需要人工核实
- **对 AI 生成的关键逻辑反向测试**：问 AI "这段代码在什么输入下会出错？"，让 AI 自己找出可能的边界条件

#### 3.1.2 奉承/迎合（Sycophancy）— AI 倾向于顺着你说

**是什么**

Sycophancy 是指 AI 倾向于同意用户的观点，即使用户的答案是错的。你问"这段代码有没有问题？"AI 回答"看起来没问题"；你改问"这段代码会不会有并发问题？"AI 立刻改口"确实存在并发风险"。它在迎合你的提问方向，而非基于代码本身做出独立判断。

**具体表现**

| 场景 | 不恰当的提问 | AI 的迎合回答 | 更好的提问方式 |
|------|------------|-------------|--------------|
| 代码审查 | "这段代码没问题吧？" | "看起来没问题" | "这段代码在什么情况下会出问题？有没有我没考虑到的边界条件？" |
| 方案选择 | "用策略模式是不是最好的？" | "是的，策略模式非常适合" | "这个需求有哪几种实现方案？各自的优缺点是什么？" |
| 调试排查 | "会不会是数据库死锁？" | "很可能是死锁，建议检查锁等待" | "贴完整的异常堆栈和涉及的代码，分析可能的原因有哪些？" |

**WMS 项目中的教训**

在 WMS 费用模块的计费引擎设计中，最初的想法是用 if-else 堆叠 14 种计费单位的计算逻辑。如果当时问 AI "这种 if-else 的实现方式可以吗？"，AI 大概率会顺着说"可以，逻辑清晰"。但实际上 14 种计费单位的 if-else 会导致代码膨胀到难以维护——最终采用的是策略模式（BillingUnitCalculator 接口 + 每种计费单位一个实现类），这个决策是经过仔细分析后做出的，而非依赖 AI 的"认可"。

**应对措施**

- **关键决策点使用反向提问**：不要问"这样做对吗？"，改问"这样做有什么问题？""有没有更好的方案？"，让 AI 给出反对意见
- **不要把 AI 当权威**：AI 的"同意"或"反对"不是决策依据，它只是一个输入角度。最终的方案选择需要基于实际业务需求、团队能力和维护成本综合判断
- **养成"求证"习惯**：AI 说某件事是"最佳实践"时，追问一句"有什么依据？"——很多时候 AI 会承认"这是我的一般性建议"

#### 3.1.3 上下文丢失 — 长对话中 AI 会"健忘"

**是什么**

大模型的上下文窗口虽然越来越大（目前主流模型支持 100K~200K tokens），但在实际使用中，AI 对长对话中早期设定信息的"注意力"会逐渐衰减。你在一开始的对话中设定了"用 Java 8，不要引入 Java 9+ 语法"，但聊了 50 轮之后，AI 可能开始使用 `var` 关键字或 `List.of()`——这些是 Java 9+ 才有的特性。

**具体表现**

- **早期约束被遗忘**：对话开头强调的技术栈限制，在长对话后期被忽略
- **角色设定偏移**：一开始设定的"后端开发专家"角色，在长时间对话后逐渐退化
- **业务规则混淆**：当对话涉及多个模块时，AI 可能把 A 模块的业务规则应用到 B 模块

**WMS 项目中的教训**

在 WMS 费用模块的长时间开发会话中，曾出现过这样的情况：对话开始时明确设定了"接口返回格式统一用 `{code, data, msg}`"，但在生成了多个模块的代码后，AI 在一个接口中生成了 `{success, data, message}` 的格式。这个不一致被 Code Review 阶段发现并修正。

这也是为什么第二章中强调 Rule 的重要性——把"接口返回格式统一用 `{code, data, msg}`"这类约束写成 Rule，让系统自动在每次对话中注入，而不是依赖开发者每次手动设置。

**应对措施**

- **重要约束通过 Rule 固化**（详见第二章 2.3.1）：配一次，一直生效，不依赖人工记忆
- **复杂任务拆分为多个短会话**：一个长会话不如多个聚焦的短会话。例如将"费用模块开发"拆为"操作费接口开发""仓库杂费接口开发""仓租规则接口开发"三个独立会话
- **定期让 AI 复述约束**：在关键代码生成前，先让 AI 复述一次当前的理解——"请先复述你对这个项目的技术约束的理解"，确认无误后再让它生成代码

#### 3.1.4 知识截止 — AI 的知识有时效性

**是什么**

大模型的知识有截止日期（训练数据的截止时间）。对于截止日期之后发布的框架新版本、安全漏洞、API 变更等，AI 不了解。例如 2025 年发布的 Spring Boot 4.0（假设），如果 AI 的知识截止于 2024 年，它就无法给出准确的指导。

**具体表现**

- **建议的 API 可能已废弃**：AI 推荐的某个方法在最新版本中已被标记为 `@Deprecated`
- **不了解新特性**：新版本中新增的便捷 API 或语法，AI 不会主动使用
- **不了解安全漏洞**：某个版本的依赖存在已知安全漏洞，AI 可能不知情

**应对措施**

- **Java/Spring Boot 版本升级、安全补丁等信息，必须人工核对官方文档**，不能仅依赖 AI
- **将 AI 的代码建议视为"参考实现"而非"权威方案"**，涉及关键依赖版本选择时查阅 Maven Central 或官方 Release Notes
- **引入新的第三方库前**，先确认其最新版本和兼容性，不要直接用 AI 建议的版本号

### 3.2 Agent 自动化的风险

> 第二章介绍了 Agent 的三个核心能力（Rule/Skill/Memory）。这些能力让 AI 从"回答工具"升级为"工作伙伴"，但也引入了新的风险——AI 不仅能"说话"，还能"做事"。能力越强，误操作的影响面越大。

#### 3.2.1 操作不可逆 — Agent 能"做"就意味能"做错"

**是什么**

当 Agent 具备自动执行能力时（如自动修改代码文件、自动执行 SQL、自动提交 SVN），一次误操作就可能造成实际损失。与纯对话模式不同——对话模式下 AI 只是"建议"，错了可以忽略；自动模式下 AI 直接操作，错了就是事故。

**风险等级与阶段划分**

| 操作类型 | 风险等级 | 示例 | 建议模式 |
|---------|---------|------|---------|
| 只读操作 | 低 | 代码审查（只输出评论）、代码解释、日志分析 | 可直接自动执行 |
| 生成新文件 | 中 | 生成新接口代码、生成测试 SQL | 自动生成，但需要人工 Review 后才能提交 |
| 修改现有文件 | **高** | 修改已有 Service 逻辑、重构代码 | 建议模式：只输出方案，不自动修改 |
| 数据库写操作 | **极高** | UPDATE/DELETE/ALTER | **禁止自动执行**，必须人工操作 |
| 版本管理操作 | **极高** | SVN 提交、分支合并 | **禁止自动执行**，必须人工操作 |

**WMS 项目中的实践**

在 WMS 费用模块的开发中，code-review Skill 采用了"建议模式"——AI 审查代码后只**输出审查评论**，由开发者决定采纳还是忽略。不会自动修改代码。这个设计原则在初期就确立了：Skill 的定位是"增强开发者的能力"，而非"替代开发者的判断"。

另一个例子：cross-check-docs Skill 会自动修正设计文档中与需求不一致的部分，但这个"自动修正"是有前提的——修正的是**设计文档**（仍在设计阶段，尚未进入编码），且修正后会输出核验报告供人工确认。设计文档改错了可以回退，但如果跨过这个边界去直接修改生产数据库，后果就完全不同了。

**应对措施**

- **Agent 初期仅限"建议模式"**：只输出方案、建议、审查意见，不自动执行操作
- **涉及写操作必须人工确认**：数据库写操作、文件修改、SVN 提交等，必须由开发者亲自执行
- **建立操作分级制度**：按上表的四级风险等级，明确规定哪些操作可以自动、哪些必须人工

#### 3.2.2 连环错误 — 一个 Agent 的错误被下一个放大

**是什么**

在多个 Agent 或 Skill 流水线协作时，上游 Agent 的错误输出会成为下游 Agent 的输入。如果中间没有人工检查点，上游的一个小错误可能在下游被放大为严重问题。

回顾第二章 2.3.2 中提到的 Skill 流水线模式：

```
需求→设计 Skill → 设计→代码 Skill → 代码审查 Skill
```

这个流水线的高效是有前提的——**每个环节的输出需要经过验证**。如果"需求→设计"环节错误地遗漏了一个必填字段，这个错误会一路传递到"设计→代码"生成的代码中，最终到"代码审查"阶段才被发现——而此时返工的成本已经很高了。

**应对措施**

- **关键环节设置人工检查点**：每个 Skill 的输出都需要人工确认后再作为下一个 Skill 的输入。特别是"需求→设计"环节——"文档阶段改一行文字，代码阶段改一堆文件"
- **不要让 AI 的输出直接成为另一个 AI 的唯一输入**：在两个 AI 步骤之间必须有人的判断介入
- **质量门禁机制**：在流水线中内建检查步骤（如 code-review 的评分过滤），不达标的输出不流入下一环节

#### 3.2.3 权限失控 — MCP 连接业务系统后的边界模糊

**是什么**

MCP（Model Context Protocol）可以让 AI 连接外部系统——通过 MCP，AI 可以调用 ERP、WMS 等业务系统的接口。这在远期是很有价值的能力（详见第五章展望），但也引入了权限管理的挑战：AI 能调用哪些接口？以什么权限调用？谁来审批？

**现阶段风险**

目前团队尚未使用 MCP（根据第三章路线图，MCP 属于 6-12 个月的长期探索），但提前建立权限意识很重要：

- AI 调用业务系统接口时，应该以**调用者本人的权限**为准，而非以"AI 的权限"为准
- 查询类接口和操作类接口应区别对待：查询可以宽松，操作必须严格

**应对措施**

- **按最小权限原则配置 MCP 服务**：未来的 MCP 连接，只开放必要的最小权限集
- **区分"查询"和"操作"**：查询类接口可直接调用，操作类接口必须二次人工确认
- **审计日志**：AI 通过 MCP 调用的操作应记录在案，便于追溯

### 3.3 团队层面需注意的问题

> 前两节讨论的是技术层面的风险——大模型本身的局限和 Agent 自动化的风险。这一节讨论的是使用方式层面的风险——与团队的行为习惯直接相关，往往比技术风险更难纠正。

#### 3.3.1 过度依赖 — "AI 能写"不等于"我可以不思考"

**问题描述**

AI 辅助开发的便利性可能导致一种危险的舒适区：需求来了 → 贴给 AI → 拿到代码 → 跑通 → 提交。整个过程中，开发者的思维参与越来越少。

**WMS 项目中的正面案例**

在 WMS 费用模块的开发中，AI 生成的计费引擎代码提供了策略模式的骨架（BillingUnitCalculator 接口 + 各计费单位的实现类框架），但**每个实现类中的具体计费逻辑是由开发者根据业务规则填充的**。AI 负责的是标准化部分（框架、分层、模板代码），人负责的是核心部分（业务规则、边界判断、异常处理）。这种"AI 做骨架，人填灵魂"的分工才是健康的使用模式。

类似的，AI 可以生成标准的 CRUD 代码（Controller → Service → Mapper），但涉及费用计算的金额精度、事务边界、重复规则校验（同一仓库 + 同一客户 + 同一时间段不允许有相同计费规则）这类业务相关的逻辑，必须由开发者理解和把关。

**应对措施**

- **核心业务逻辑必须亲自理解**：AI 生成的代码，你能讲清楚每一行的作用吗？如果讲不清楚，就先不要提交
- **AI 是加速器，不是替代品**：它的价值是帮你省掉机械敲码的时间，让你把精力集中在业务理解和架构设计上
- **如果一段代码你完全看不懂，即使它跑通了，也不要提交**

#### 3.3.2 代码质量稀释 — AI 生成量大但不一定质量高

**问题描述**

AI 生成代码的速度远超手写。一个开发者一天可能要审查上千行 AI 代码。当审查速度跟不上生成速度时，质量就会稀释——风格不一致、隐蔽 Bug、冗余逻辑等问题可能被遗漏。

**具体风险**

| 问题 | 表现 | 示例 |
|------|------|------|
| 风格不一致 | 同模块不同接口，命名、异常处理方式不统一 | 有的接口用 try-catch 包裹，有的用全局异常处理器，风格混搭 |
| 过度生成 | AI 生成了实际不需要的代码 | 生成了"删除"接口，但业务上费用记录是不允许删除的 |
| 隐蔽 Bug | 逻辑在一般情况下正确，边界条件有坑 | 前文提到的 BigDecimal.divide() 未指定 RoundingMode |
| 注释与代码不符 | AI 生成的注释可能描述的是另一种实现方式 | 代码用的是策略模式，注释写的是"通过 if-else 判断" |

**应对措施**

- **AI 生成的代码必须 Review，没有例外**（这条在第二章 2.1 使用规范中已强调，这里再次强化）
- **分批审查**：不要等 AI 生成完整个模块再审查。每生成一个接口，即时审查、即时理解
- **建立 Review 清单**：安全性（SQL 注入、空指针）→ 规范性（命名、分层）→ 业务正确性（金额精度、状态流转、权限校验）

#### 3.3.3 业务逻辑盲区 — AI 不了解你的公司和项目

**问题描述**

AI 的知识来自公开的训练数据。它不了解你们公司的具体业务流程、不了解 WMS 费用模块中"操作费"和"仓库杂费"的业务区别、不了解你们项目中的历史约定（如某张表的 del_flag 为什么用了 0/2 而非 0/1）。AI 生成的代码可能语法完全正确，但业务上不成立。

**WMS 项目中的实际案例**

在 WMS 费用模块中，AI 初次生成的建表 SQL 将 del_flag 字段的标准定义（0=正常，1=删除）用在了新表上。这个用法本身是正确的——整个项目的标准规范就是 0/1。但如果 AI 在分析旧表 `b_bank_card` 时发现其 del_flag 使用了 0/2，AI 无法判断 0/2 到底是"这个项目的另一种规范"还是"历史遗留错误"。没有人的判断介入，AI 可能在两种写法之间摇摆。

再比如：费用模块中 14 种计费单位各有不同的计算规则，这些规则来自跨境电商的业务实践（如体积重换算公式、最小计费单位限制等），AI 的训练数据中可能有类似的通用知识，但公司的具体费率和规则是业务特有的，AI 无法准确生成。

**应对措施**

- **涉及业务规则的代码，必须由熟悉业务的开发者把关**
- **将业务规则沉淀为 Memory**（详见第二章 2.3.3）：越是团队特有的知识，越值得记下来让 AI 了解
- **遇到 AI 生成的代码涉及你不熟悉的业务模块时，先找熟悉该模块的同事确认**

#### 3.3.4 技能退化 — 用进废退

**问题描述**

长期依赖 AI 写代码，可能导致一些核心能力的退化：

- **排查问题的能力**：当 AI 帮你追踪调用链、分析异常堆栈时，你自己的排查肌肉就在萎缩
- **从零设计的能力**：当 AI 帮你生成代码骨架时，你自己设计架构的能力就缺少锻炼
- **读代码的耐心**：习惯了 AI 帮你解读旧代码，自己逐行分析代码的耐心和速度会下降

**这不是危言耸听**

一个现实的场景：线上出现了一个复杂的 Bug，涉及 OutboundOrderService → 计费引擎 → fee_transaction 的多层调用链。平时这类排查依赖 AI 辅助分析，但如果问题出在 AI 不熟悉的业务逻辑上（如某个特定计费单位的计算规则），你就需要自己跟踪代码。如果平时完全没有手工排查过类似问题，这时就会很吃力。

**应对措施**

- **保持手工排查的习惯**：简单的 Bug 可以先自己排查，再用 AI 验证，而不是反过来
- **定期进行"无 AI 练习"**：每隔一段时间，尝试不借助 AI 完成一个小功能，保持对代码的直接掌控感
- **AI 是工具，保持主导权**：你的目标是成为一个"会用 AI 的优秀工程师"，而非"离开 AI 就不会写代码的操作员"

---

### 本章小结

本章从三个层次梳理了 AI 应用中的风险：

```
大模型本身局限               Agent 自动化风险               团队使用方式问题
（技术底层，无法消除）         （能力越强影响面越大）           （行为习惯，最难纠正）
─────────────────────────────────────────────────────────────────────────
幻觉    奉承    上下文丢失      操作不可逆     过度依赖    代码质量稀释
        知识截止                连环错误       业务逻辑盲区    技能退化
                                权限失控
```

**核心原则**：

1. **风险不是拒绝 AI 的理由**——了解风险是为了用得更稳，而非不用
2. **控制能控制的**——大模型本身的局限（幻觉、知识截止）你无法消除，但可以通过 Review 来兜底；Agent 的权限你可以控制（建议模式 vs 自动执行）；团队的使用习惯你可以培养
3. **越往后越重要**——大模型的局限会随着技术进步逐步改善；但团队层面的使用习惯一旦形成就很难纠正。从现在开始建立正确的使用方式，比事后纠偏容易得多

**下一章预告**：了解了 AI 的局限性后，第四章将给出分阶段的落地与学习计划——在每个阶段做什么、学什么、交付什么，同时把本章提到的风险应对措施融入每个阶段的动作中。
