## 二、AI 工具与使用方法

> 本章回答"使用哪些 AI 工具，如何使用"。结构为：先划红线（规范）→ 再讲当前怎么用好（IDE 对话 + 提示工程）→ 再讲进阶怎么自动化（Agent 能力）。

---

### 2.1 使用规范建议

> 工具用起来之前，先把红线划清楚，避免好心办坏事。

在正式推广 AI 工具之前，有几条底线需要团队共同遵守。这不是给 AI 应用设障碍，而是确保 AI 用得安全、用得放心。以下三条规范建议从当前阶段即开始执行：

**一、不要直接贴敏感代码**

AI 对话工具（包括 Qoder）会将用户输入发送到云端进行处理。虽然主流厂商承诺数据安全，但从信息安全角度出发，不应将敏感信息直接粘贴到对话框中。具体包括：

- 数据库连接密码、API Key、Token 等凭据类信息
- 服务器 IP 地址、内网拓扑等网络信息
- 客户隐私数据、生产环境真实手机号/身份证号等

**操作方法**：粘贴代码前，将上述敏感值替换为占位符（如 `DB_PASSWORD`、`xxx.xxx.xxx.xxx`、`138****1234`），AI 生成结果后再替换回来。这不会影响 AI 对代码逻辑的理解，但能有效保护信息安全。

**二、AI 生成的代码必须 Review**

这条是底线中的底线。AI 是辅助工具，代码的最终责任在写代码的人。尤其是后端开发中涉及以下场景的代码，必须在充分理解后人工确认才能提交：

- **数据库写操作**（INSERT/UPDATE/DELETE）：确认事务边界、数据一致性、WHERE 条件是否正确
- **金额计算**：确认精度（BigDecimal 而非 double）、 rounding 策略、币种转换逻辑
- **权限校验**：确认接口是否有正确的鉴权逻辑，是否存在越权风险
- **外部接口调用**：确认超时设置、重试策略、异常兜底

审查时的一个实用技巧：**向 AI 反向提问**——"这段代码在什么情况下会出问题？""有没有我没考虑到的边界条件？"——让 AI 帮你做第一轮自查，但最终判断仍由人工做出。

**三、AI 生成内容标注**

对于关键决策点（如架构方案选型、核心算法实现、安全相关逻辑），建议在代码注释或文档中标注是否使用了 AI 辅助。目的不是追责，而是：

- 便于后续维护时了解代码的"来历"
- 出现问题时能更快定位是人工设计缺陷还是 AI 生成偏差
- 积累 AI 辅助开发的经验数据，帮助团队持续优化使用方式

标注方式不需要复杂，一行注释即可：

```java
// AI-assisted: 计费引擎策略模式骨架由 AI 生成，计费规则细节人工补充
```

---

以上三条规范建议在团队刚开始使用 AI 时就应建立共识。它们不增加多少工作量，但能为后续更大规模的 AI 应用打好安全基础。

### 2.2 IDE 内代码辅助 — 当前阶段的使用方式

> 部门开发同事目前使用 Qoder 的 VS Code / IDEA 插件，主要方式是在 IDE 内通过自然语言对话让 AI 辅助编码。本节说明如何把这个基础用法用好。

#### 2.2.1 当前的核心使用方式

当前阶段，团队使用 AI 的核心方式就是 **IDE 内自然语言对话**——在 VS Code 或 IDEA 中打开 Qoder 对话框，选中一段代码或在对话框中用自然语言描述需求，AI 返回代码建议、解释或修改方案。

这种方式的优势在于零门槛、即时反馈。不需要额外安装工具，不需要学习新平台，公司已为全员开通的 Qoder 账号即可覆盖所有场景。但它的效果上限取决于使用者的提问质量——这正是下一节提示工程要解决的问题。

#### 2.2.2 提示工程（Prompt Engineering）— 从"随便问"到"有目的地问"

##### 为什么提示词是核心能力

同样用 AI，好的提示词和随口提问，产出质量差距巨大。来看一个真实对比：

```
随便问：                           结构化提示词：
"帮我写个接口"              →       "我们用的是 Spring Boot + MyBatis，
                                   返回格式统一 {code, data, msg}，
                                   请为采购单表 PO_HEADER 生成查询接口，
                                   Controller/Service/Mapper 三层，
                                   带分页和按状态筛选"

结果：能用但需要大量修改             结果：基本直接可用
```

左边的提问方式在团队中很常见——花费 5 秒打字，然后用 20 分钟修改 AI 的产出。右边的提问方式多花 30 秒描述背景和约束，但省掉了后面的修改时间。**提示词质量直接决定了 AI 产出与"直接可用"之间的距离。**

更深一层看，Agent 的高级能力（Skill、Rule、Memory）本质上是把好的提示词**固化下来复用**：

- **Rule** = 每次对话自动注入的背景和约束（不需要每次手动写"我们用 Java 8 + Spring Boot"）
- **Skill** = 固定的提示词模板 + 执行步骤 + 质量检查（把一次成功的提示词变成可重复触发的流程）
- **Memory** = 跨会话持久化的上下文（让 AI 记住项目架构，不用每次交代背景）

提示词是这一切的基石。提示词写不好，后面 Skill 和 Rule 的效果都打折扣。所以，团队在 AI 应用上的第一项能力建设，就是学会写好提示词。

##### 提示词的四个要素

一个结构化的提示词包含四个要素，记住这四个要素就能让 AI 产出质量明显提升：

| 要素 | 作用 | 正例 | 反例 |
|------|------|------|------|
| **背景（Context）** | 告诉 AI 项目环境、技术栈、现状 | "这是一个已上线的 ERP 系统，Java 8 + Spring Boot，不要引入 Java 9+ 语法" | 不提技术栈，AI 可能用 Java 17 的写法 |
| **角色（Role）** | 让 AI 以特定身份思考 | "你是一个有 5 年跨境电商后端开发经验的程序员" | 不设角色，AI 给出的方案可能脱离业务实际 |
| **目标（Goal）** | 明确要产出什么 | "为 WMS 的出库单模块新增一个批量导出 Excel 的接口" | "帮我写个导出"——没说导出什么、什么格式 |
| **约束（Constraints）** | 限制范围、格式、禁区 | "不要修改现有表结构，导出字段包含单号/日期/状态/操作人，用 EasyExcel" | 不给约束，AI 可能自作主张加字段、换技术方案 |

这四个要素不需要每次都凑齐。简单的代码解释可能只需要"背景 + 目标"，但涉及代码生成的场景建议四个要素都写上。核心原则是：**你给 AI 的信息越充分，AI 的产出越精准**。

##### 部门工作中的典型提示词模板

以下模板针对部门最高频的三类后端场景设计，团队成员可直接复制、填空使用。前端场景同理，将【背景】替换为 Vue + Element UI、【约束】替换为"不修改现有组件结构"即可，此处不赘述。

**场景一：后端修 Bug（高频场景）**

```
【背景】ERP 系统，Spring Boot，MyBatis，SVN 管理
【问题】[描述 Bug 现象，贴报错日志]
【相关代码】[贴 Service 层 + Mapper 层代码，标注出问题的调用链]
【要求】分析根因并给出最小改动方案，不要大面积重写
【注意】涉及[库存扣减/金额计算/状态流转]逻辑，务必确认事务边界是否正确
```

**场景二：后端新接口开发**

```
【背景】WMS 系统，Java 8 + Spring Boot，现有表 [表名]
【需求】[描述接口功能，列出输入参数和返回字段]
【约束】[技术方案约束，如"用 EasyExcel""GET 请求，参数放在 URL 上""不要改表结构"]
【输出】Controller/Service/Mapper 完整代码
```

**场景三：SQL 优化**

```
【背景】ERP 报表模块，MySQL 5.7，数据量 [X] 万级
【慢 SQL】[贴 SQL 和 EXPLAIN 结果]
【要求】分析慢的原因并给出优化方案（优先索引优化，其次改写 SQL，最后考虑代码层调整）
```

> **使用建议**：模板中的 `[ ]` 是需要填写的内容。刚开始使用时建议严格按模板填写，熟练后可以根据实际情况调整要素顺序和详略。关键是养成"先想清楚背景和约束再提问"的习惯。

##### 团队落地方式

提示工程不是一次培训就能掌握的技能，需要在日常工作中持续练习和优化。建议团队采取以下方式落地：

1. **建立部门 Prompt 模板库**：将上面三类场景的模板作为起点，团队成员在使用过程中将自己遇到的好用的提示词补充进来。模板库不需要复杂工具，一个共享文档即可。格式统一：场景名称 + 适用情况 + 模板内容 + 贡献人 + 使用效果备注。

2. **好的提示词及时共享**：遇到一次写得好的提示词（AI 产出质量明显高于平时），主动沉淀到模板库。特别是那些"没想到这样问效果这么好"的提示词，往往对团队最有价值。

3. **不需要专门培训**：团队已经在日常工作中使用 Qoder 做自然语言对话，提示工程是在现有习惯上的"有意识地优化"，不是从零学一门新技能。重点是让每个人意识到"多写 30 秒提示词，省下 20 分钟修改时间"这个投入产出比。

### 2.3 AI 智能体进阶使用（分阶段引入）

> IDE 内自然语言对话只是起点。Agent 工具提供了 Skill（技能）、Rule（规则）、Memory（记忆）等能力，可以让 AI 从"一次性问答"升级为"持续理解项目上下文的工作伙伴"。
>
> 以下结合 WMS 项目的实际使用经验说明。

#### 2.3.1 Skill（技能）— 可复用的自动化工作流模板

##### 是什么

Skill 是把一组重复性的工作流程固化为一个"技能"，一键触发后 AI 按预定步骤执行。它与普通对话的核心区别在于：

| 对比维度 | 普通对话 | Skill |
|---------|---------|-------|
| 流程定义 | 每次靠人手动描述步骤 | 固化在 Skill 中，无需每次说明 |
| 输入输出 | 随意，没有固定格式 | 明确定义输入格式和输出产物 |
| 质量检查 | 靠人肉眼判断 | 内建验证步骤，自动检查输出质量 |
| 可复用性 | 每次从零开始描述 | 一次编写，反复使用 |

简单来说，普通对话是"每次手动画流程图"，Skill 是"把流程图固化为一个按钮"。

##### WMS 项目中的实际 Skill

在 WMS 费用管理模块的开发中，我们搭建了 4 个 Skill，形成了一条完整的 AI 辅助开发流水线：

```
PRD 需求 ──→ req-to-api ──→ cross-check-docs ──→ api-to-code ──→ code-review
           (需求分析+设计     (6 维度文档一致       (全套 Java 代码)   (5 agent
            文档+建表 SQL     性核验+自动修正)                        并行审查)
            +API 接口文档)
```

四个 Skill 的详细信息如下：

| Skill | 输入 | 输出 | 核心流程 |
|-------|------|------|---------|
| **req-to-api**（需求→设计） | PRD 需求文档 | 设计文档 + 建表 SQL + API 接口文档（标准 14 接口） | 按 12 个维度提取需求（业务背景、数据实体、状态流转、权限模型……）→ 数据库设计（遵循建表模板和命名规范）→ API 接口设计 |
| **cross-check-docs**（文档一致性验证） | 需求文档 + 设计文档 + API 文档 + DDL | 核验报告 + 自动修正后的设计文档 | 加载四份文档 → 6 维度验证（筛选条件完整性、字段覆盖、业务流程、状态枚举一致性、SQL 字段类型匹配、接口参数与表字段对应）→ 问题分级（P0 阻塞开发 / P1 需修正 / P2 优化建议）→ 自动修正 P0/P1 问题 → 重新验证确认修正效果 |
| **api-to-code**（设计→代码） | API 接口文档 + 参考模块 | 完整 Java 代码（Controller / Service / Mapper / Domain / Cmd / Query / VO） | 文档分析 → 研究参考模块的代码风格和项目约定 → 10 步有序代码生成（建表 → Domain → Mapper → Service → Controller → ……）→ 日志标准注入 → 代码自验证 → 与文档交叉核对 → 常见问题自动修复 |
| **code-review**（代码审查） | 代码变更 | 带置信度评分的审查评论 | 5 个 Sonnet agent 并行独立审查（分别负责安全性、编码规范、性能、业务逻辑、可维护性 5 个维度）→ Haiku agent 综合评分 → 置信度 ≥ 80 分的评论才发布，低分评论自动丢弃 |

##### 关键设计思路

这四个 Skill 并非独立设计的，它们之间有意识地遵循了三条设计原则：

**1. 流水线串联**

上一个 Skill 的输出直接作为下一个 Skill 的输入。例如 `req-to-api` 产出的 API 接口文档，直接作为 `api-to-code` 的输入，中间不需要人工复制粘贴或格式转换。这种设计减少了手动传递信息时的遗漏和错误。

**2. 质量门禁**

在关键节点设置质量检查，不达标就阻断流程：
- `cross-check-docs` 发现 P0 级问题时，输出核验报告并自动修正，修正后重新验证。在文档一致性确认之前，不建议进入编码阶段——因为"文档阶段改一行文字，代码阶段改一堆文件"。
- `code-review` 用置信度评分机制过滤低质量审查意见，避免"AI 瞎评论导致开发者浪费时间"。

**3. 可复用**

同一个 Skill 可以复用到不同业务模块，只需更换输入内容：
- `req-to-api` 的 12 维度需求提取框架，既用于操作费，也用于仓库杂费和仓租规则
- `api-to-code` 的 10 步代码生成流程，不依赖具体业务逻辑，适用于任何标准 CRUD 模块
- `code-review` 的 5 维度并行审查框架，对所有后端代码变更通用

##### 引入时机

Skill 的编写和调试需要一定的时间投入（一个 Skill 从设计到稳定可用通常需要 1-2 周迭代），且需要对 Agent 工作机制有基本理解。建议在以下条件满足后开始试点：

- 团队成员已熟悉基础提示工程（能稳定写出结构化的提示词）
- 已选定一个高频且流程固定的场景作为试点（推荐从后端 CRUD 接口生成开始，这是最高频的场景）
- 由 1-2 位对 AI 工具较熟悉的同事先行编写和验证，确认效果后再推广

WMS 项目的经验是：先花 1-2 周把第一个 Skill（req-to-api）打磨好，团队看到实际效果后，后续 Skill 的推广阻力会小很多。

---

#### 2.3.2 Rule（规则）— 让 AI 始终遵循部门规范

##### 是什么

Rule 是项目级或用户级的持久化指令，配置后 AI 在**每次对话中自动加载**，确保输出始终符合团队约定。

它与提示词中的"约束"要素有本质区别：

- **提示词中的约束**：每次手动写，这次写了下次可能忘
- **Rule**：配一次，之后每次对话自动生效，不需要反复提醒

对于团队来说，Rule 解决了一个很现实的问题：AI 不知道我们项目的具体约定。它不知道我们统一用 `{code, data, msg}` 作为返回格式，不知道我们用 SVN 而不是 Git，不知道我们的 del_flag 用 0/1 而不是 0/2。没有 Rule，每次都要手动告知这些信息；有了 Rule，AI 自动遵守。

##### 部门应用举例

| 规则类型 | Rule 内容示例 | 解决的问题 |
|---------|-------------|-----------|
| 安全规范 | "数据库操作必须使用 MyBatis 的参数化查询（#{ }），禁止拼接 SQL 字符串；动态排序/分组必须用白名单校验" | 从源头杜绝 SQL 注入，比 Code Review 阶段再发现更高效 |
| 接口约定 | "所有 RESTful 接口统一返回格式 `{code: Integer, data: Object, msg: String}`，code=200 表示成功，分页结果包含 `total` 和 `rows` 字段" | 前后端接口风格一致，减少联调时的格式争议 |
| 技术栈约束 | "项目使用 Java 8，禁止使用 var、文本块、switch 表达式等 Java 9+ 语法；仅使用 Lombok 作为注解处理器，不引入 MapStruct 等额外注解框架" | 避免 AI 生成项目不支持的语法，减少编译错误 |
| 项目约定 | "实体类 del_flag 字段统一使用 0=正常 1=删除；数据库表名不加前缀，字段名使用下划线命名；实体类属性使用驼峰命名，通过 MyBatis 的 mapUnderscoreToCamelCase 自动映射" | 新代码与现有代码风格一致，减少 Code Review 中的命名争议 |
| 业务规则 | "所有涉及金额的字段使用 BigDecimal 类型，禁止使用 float 或 double；金额计算使用 BigDecimal 的 add/subtract/multiply 方法，指定 MathContext" | 避免浮点精度问题，这类 Bug 测试阶段不易发现，上线后影响严重 |

##### 使用效果对比

以 SQL 安全规范为例，看一条 Rule 的实际效果：

```
没有 Rule "禁止拼接 SQL 字符串"：

AI 生成的代码：
String sql = "SELECT * FROM po_header WHERE status = '"
           + status + "'";
//     ↑ SQL 注入风险，攻击者可通过 status 参数注入恶意 SQL


有 Rule "禁止拼接 SQL 字符串"：

AI 生成的代码：
@Select("SELECT * FROM po_header WHERE status = #{status}")
List<PoHeader> findByStatus(@Param("status") String status);
//     ↑ 自动使用 MyBatis 参数化查询，安全且风格一致
```

差异很明显：没有 Rule 时，每次都要在提示词里写"不要拼接 SQL"，忘了写就可能出问题；有 Rule 后，AI 自动生成安全的写法，不需要每次提醒。

##### 引入时机

**当前即可开始**。Rule 是三个 Agent 能力中上手门槛最低的——无需编写流程逻辑，就是整理几条规则写进配置文件。建议按以下步骤推进：

1. 团队一起梳理最近 3 个月内 Code Review 中最常见的返工点（如 SQL 拼接、接口格式不统一、命名不规范），列出 5-8 条候选
2. 从中选出 3-5 条"频率高 + 影响大 + 规则明确"的，写成 Rule
3. 配置到项目级 Rule 中，让大家在日常使用中体验"配一次、一直生效"的便利
4. 使用 2-4 周后收集反馈，增补或调整 Rule

先少后多，3-5 条高质量的 Rule 比 20 条粗糙的 Rule 效果好得多。

#### 2.3.3 Memory（记忆）— AI 记住项目上下文

##### 是什么

Memory 是跨会话持久化的记忆机制——AI 能记住项目架构、业务术语、历史决策等信息，在后续对话中自动调用。

它与普通对话的区别：

- **普通对话**：每次新开会话，AI 对项目一无所知，你需要重新交代"我们项目用 Java 8 + Spring Boot，这张表叫 fee_transaction……"
- **有 Memory**：AI 已经"了解"这个项目的技术栈、核心表结构、业务规则和历史决策，新会话中自动关联这些背景知识

Memory 的价值在长周期项目中尤其明显——一个 WMS 项目持续数月，如果每次对话都要重新交代背景，不仅效率低，而且容易遗漏关键信息。有了 Memory，AI 对项目的理解会随着使用逐渐加深。

##### WMS 项目中的实际使用

在 WMS 费用管理模块开发过程中，我们沉淀了以下 Memory，每条都解决了特定的实际问题：

| 记忆类型 | 具体示例 | 解决的问题 |
|---------|---------|-----------|
| **业务术语** | 费用 ID 前缀规则：操作费 = CZF + 4 位顺序码，仓库杂费 = ZF + 4 位顺序码，仓租规则 = CKZF + 4 位顺序码 | AI 生成代码时自动使用正确的 ID 前缀，不会出现"费用类型和 ID 前缀对不上"的问题 |
| **架构决策** | 14 种计费单位（按重量/体积/件数/托盘……），采用策略模式设计——BillingUnitCalculator 接口，每种计费单位一个实现类，通过工厂类动态获取 | AI 在设计和代码生成时自动采用策略模式架构，不会建议 if-else 堆叠的方案 |
| **历史约定** | del_flag 字段统一用 0=正常 1=删除；项目中部分旧表（如 b_bank_card）的 del_flag 用了 0/2 是历史遗留，新表不要沿用 | AI 生成建表 SQL 时自动使用 0/1 规范，不会因为看到旧表的 0/2 而误以为是正确写法 |
| **状态机定义** | 操作费 5 种状态及流转规则：草稿 → 确认 → 生效 → 无效 / 已过期；其中"确认"状态下可作废回到"草稿" | AI 生成状态变更代码时自动按正确流转逻辑实现，且会校验非法状态转换 |
| **模块依赖** | WP08 操作费模块的开发依赖链：FeeType（费用类型）← Customer 改造（结算币种）← 计费引擎 ← 操作费 ← 仓库杂费 ← 仓租规则 | AI 辅助任务拆分时自动标注依赖顺序，不会出现"先开发操作费再改造 Customer"的逻辑错误 |

##### 如何沉淀 Memory

Memory 不需要额外花大量时间专门整理。最自然的做法是和日常文档工作同步进行——每次写技术方案、设计文档或做出关键决策时，顺手把值得记忆的信息提炼成 1-2 段话，写入 Memory：

- **写设计文档时** → 把关键架构决策提取为 Memory（如"费用模块采用策略模式"）
- **排查线上问题时** → 把沉淀的经验写入 Memory（如"del_flag 的 0/2 是历史遗留"）
- **新人上手时** → 把新人最常问的问题和答案写入 Memory（如"项目用 SVN 不用 Git"）

##### 引入时机

建议**中期（3-6 个月）**开始，与 Rule 和 Skill 的推广节奏配合：

- Rule 先上手（当前阶段），解决"AI 遵守规范"的问题
- Skill 同步试点（1-3 个月），解决"AI 执行固定流程"的问题
- Memory 跟进（3-6 个月），解决"AI 理解项目上下文"的问题

这个顺序的逻辑是：Memory 的价值在 Rule 和 Skill 就位后才能真正体现——AI 既遵守了规范，又能执行固定流程，再加上项目上下文的理解，才能形成完整的"AI 工作伙伴"体验。

#### 2.3.4 能力引入节奏总结

Rule、Skill、Memory 三项能力的引入不是并行的，而是有先后顺序：

```
当前（0-3 个月）  ────  中期（3-6 个月）  ────  长期（6-12 个月）

  Rule                    Skill + Memory            MCP
  （立即可做）             （可复用工作流             （连接业务系统）
                           + 深度项目理解）
```

**为什么是这个顺序？**

- **Rule 第一**：配置成本最低，效果最直观。3-5 条 Rule 配好，第二天团队就能感受到变化。而且 Rule 能解决最急迫的问题——AI 生成的代码不符合项目规范。
- **Skill 第二**：需要编写和调试投入，但效果可复用。有了 Rule 保证代码规范后，再通过 Skill 让 AI 执行标准化的开发流程，提升的是效率而非仅仅是规范性。
- **Memory 第三**：需要积累过程。Memory 不是一次性配置完成的，而是在项目推进中逐步沉淀的。在 Rule 和 Skill 使用一段时间后，自然积累了足够的记忆素材。
- **MCP 第四**：MCP（Model Context Protocol）连接业务系统属于平台级建设，需要架构层面的支持。作为 Java 业务开发部门，当前阶段的目标是把 Rule/Skill/Memory 三项基本能力用好，MCP 作为远期方向了解即可。
