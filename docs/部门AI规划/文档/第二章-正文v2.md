## 二、AI 工具与使用方法

> 本章回答"使用哪些 AI 工具，如何使用"。结构为：先划红线（规范）→ 再讲当前怎么用好（IDE 对话 + 提示工程）→ 再讲进阶怎么自动化（Agent 能力）。

---

### 2.1 使用规范建议

> 工具用起来之前，先把红线划清楚，避免好心办坏事。

在正式推广 AI 工具之前，有几条底线需要团队共同遵守：

**一、不要直接贴敏感代码**

AI 对话工具会将用户输入发送到云端处理，不应将敏感信息直接粘贴：

- 数据库连接密码、API Key、Token 等凭据
- 服务器 IP 地址、内网拓扑
- 生产环境客户隐私数据

**操作方法**：粘贴前替换为占位符（如 `DB_PASSWORD`、`xxx.xxx.xxx.xxx`），生成结果后再替换回来。

**二、AI 生成的代码必须 Review**

AI 是辅助工具，代码责任在写代码的人。以下场景必须在充分理解后人工确认：

- **数据库写操作**（INSERT/UPDATE/DELETE）：确认事务边界、数据一致性
- **金额计算**：确认精度（BigDecimal 而非 double）、币种转换逻辑
- **权限校验**：确认接口鉴权、越权风险
- **外部接口调用**：确认超时、重试、异常兜底

一个实用技巧：**向 AI 反向提问**——"这段代码在什么情况下会出问题？""有没有我没考虑到的边界条件？"——让 AI 帮你自查，但最终判断由人工做出。

**三、AI 生成内容标注**

对于关键决策点（架构方案选型、核心算法、安全逻辑），在代码注释或文档中标注 AI 辅助情况。目的不是追责，而是：

- 便于后续维护时了解代码来源
- 出现问题时快速定位根因
- 积累 AI 辅助开发的经验数据

标注方式：

```java
// AI-assisted: 计费引擎核心骨架由 AI 生成，业务规则细节人工补充
```

---

以上三条规范在团队刚开始使用 AI 时就应建立共识。不增加多少工作量，但为后续更大规模的 AI 应用打好安全基础。

### 2.2 IDE 内代码辅助 — 当前阶段的使用方式

> 部门开发同事目前使用 Qoder 的 VS Code / IDEA 插件，核心方式是在 IDE 内通过自然语言对话让 AI 辅助编码。本节说明如何把这个基础用法用好。

#### 2.2.1 当前的核心使用方式

当前阶段的核心使用方式就是 **IDE 内自然语言对话**——在 VS Code 或 IDEA 中打开 Qoder 对话框，选中代码或用自然语言描述需求，AI 返回代码建议、解释或修改方案。零门槛、即时反馈，公司已提供 Qoder 账号即可覆盖。

但这种方式的效果上限取决于使用者的提问质量——这正是下一节要解决的问题。

#### 2.2.2 提示工程（Prompt Engineering）— 从"随便问"到"有目的地问"

##### 为什么提示词是核心能力

同样用 AI，好的提示词和随口提问，产出质量差距巨大：

```
随便问：                           结构化提示词：
"帮我写个接口"              →       "我们用的是 Spring Boot + MyBatis，
                                   返回格式统一 {code, data, msg}，
                                   请为采购单表 PO_HEADER 生成查询接口，
                                   Controller/Service/Mapper 三层，
                                   带分页和按状态筛选"

结果：能用但需要大量修改             结果：基本直接可用
```

左边的提问方式花 5 秒打字，然后用 20 分钟修改产出。右边多花 30 秒描述背景和约束，省掉了修改时间。**提示词质量直接决定了 AI 产出与"直接可用"之间的距离。**

更深一层：Agent 的高级能力（Skill、Rule、Memory）本质上是把好的提示词**固化复用**——提示词是这一切的基石。

##### 提示词的四个要素

| 要素 | 作用 | 正例 | 反例 |
|------|------|------|------|
| **背景（Context）** | 告诉 AI 项目环境、技术栈 | "Java 8 + Spring Boot，已上线系统，不要引入新语法" | 不提技术栈，AI 可能用不兼容的写法 |
| **角色（Role）** | 让 AI 以特定身份思考 | "你是一个有后端开发经验的程序员" | 不设角色，方案可能脱离业务实际 |
| **目标（Goal）** | 明确要产出什么 | "为出库单模块新增批量导出 Excel 接口" | "帮我写个导出"——没说导出什么、什么格式 |
| **约束（Constraints）** | 限制范围、格式、禁区 | "不要修改现有表结构，用 EasyExcel" | 不给约束，AI 可能自作主张改表结构 |

核心原则：**给 AI 的信息越充分，AI 的产出越精准。**

##### 部门工作中的典型提示词模板

> 以下为后端场景示例，前端场景同理（将背景替换为 Vue + Element UI 即可）。

**场景一：后端修 Bug（高频）**

```
【背景】ERP 系统，Spring Boot，MyBatis
【问题】[描述 Bug 现象，贴报错日志]
【相关代码】[贴 Service 层 + Mapper 层代码]
【要求】分析根因并给出最小改动方案，不要大面积重写
【注意】涉及[库存扣减/金额计算/状态流转]，确认事务边界是否正确
```

**场景二：后端新接口开发**

```
【背景】Java 8 + Spring Boot，现有表 [表名]
【需求】[描述接口功能，列出输入参数和返回字段]
【约束】[技术方案约束，如"用 EasyExcel""不要改表结构"]
【输出】Controller/Service/Mapper 完整代码
```

**场景三：SQL 优化**

```
【背景】MySQL 5.7，数据量 [X] 万级
【慢 SQL】[贴 SQL 和 EXPLAIN 结果]
【要求】分析慢的原因并给出优化方案（优先索引优化，其次改写 SQL）
```

> **使用建议**：`[ ]` 是需要填写的内容。关键是养成"先想清楚背景和约束再提问"的习惯。

##### 团队落地方式

1. **建 Prompt 模板库**：将上述模板作为起点，团队成员补充自己验证过的好用的提示词，一个共享文档即可
2. **好的提示词共享**：遇到产出质量明显高于平时的情况，沉淀到模板库
3. **不需要专门培训**：团队已在使用 Qoder，提示工程是在现有习惯上的"有意识地优化"，关键是让每个人认识到"多写 30 秒提示词，省下 20 分钟修改时间"

### 2.3 AI 智能体进阶使用（分阶段引入）

> IDE 内自然语言对话只是起点。当基础对话不能满足"持续理解项目、自动遵循规范、复用固定流程"的需求时，就需要引入 Agent 的三个核心能力：Rule（规则）、Skill（技能）、Memory（记忆）。

#### 2.3.1 Rule（规则）— 让 AI 始终遵循部门规范

##### 是什么

Rule 是项目级或用户级的持久化指令，配置后 AI 在**每次对话中自动加载**，确保输出始终符合团队约定。

与提示词中手动写约束的区别：Rule **配一次，一直生效**。

对于团队来说，Rule 解决的现实问题是：AI 不知道项目的具体约定——它不知道接口返回格式、不知道命名规范、不知道技术栈限制。没有 Rule，每次都要手动告知；有了 Rule，AI 自动遵守。

##### 部门应用举例

| 规则类型 | Rule 内容示例 | 解决的问题 |
|---------|-------------|-----------|
| 安全规范 | "数据库操作必须使用参数化查询（#{ }），禁止拼接 SQL 字符串；动态排序/分组必须用白名单校验" | 从源头杜绝 SQL 注入 |
| 接口约定 | "RESTful 接口统一返回 `{code, data, msg}`，分页结果含 `total` 和 `rows`" | 前后端接口风格一致 |
| 技术栈约束 | "项目使用 Java 8，禁止使用 var、文本块等 Java 9+ 语法；仅使用 Lombok" | 避免 AI 生成不兼容语法 |
| 项目约定 | "实体类 del_flag 用 0=正常 1=删除；表字段下划线命名，实体驼峰命名" | 新老代码风格统一 |
| 业务规则 | "金额字段用 BigDecimal，禁止 float/double；计算时指定 MathContext" | 避免浮点精度问题 |

##### 使用效果对比

以 SQL 安全规范为例：

```
没有 Rule：                          有 Rule：

AI 生成：                            AI 生成：
String sql = "SELECT * FROM           @Select("SELECT * FROM po_header
  po_header WHERE status = '"           WHERE status = #{status}")
  + status + "'";                     List<PoHeader> findByStatus(
//     ↑ SQL 注入风险                    @Param("status") String status);
                                      //     ↑ 自动使用参数化查询
```

##### 引入时机

**当前即可开始**。Rule 是三个能力中上手门槛最低的——就是整理几条规则写进配置文件。建议步骤：

1. 梳理最近 3 个月内 Code Review 中最常见的返工点（如 SQL 拼接、接口格式不统一），列出候选
2. 从中选出 3-5 条"频率高 + 影响大"的，写成 Rule
3. 使用 2-4 周后收集反馈，增补调整

3-5 条高质量的 Rule 效果远好于 20 条粗糙的。

#### 2.3.2 Skill（技能）— 可复用的自动化工作流模板

##### 是什么

Skill 是把一组重复性的工作流程固化为一个"技能模板"，一键触发后 AI 按预设步骤执行。与普通对话的核心区别：

| 对比维度 | 普通对话 | Skill |
|---------|---------|-------|
| 流程定义 | 每次手动描述步骤 | 固化在 Skill 中 |
| 输入输出 | 随意，无固定格式 | 明确定义输入格式和输出产物 |
| 质量检查 | 靠人肉眼判断 | 内建验证步骤 |
| 可复用性 | 每次从零开始 | 一次编写，反复使用 |

##### 典型 Skill 场景示例

以下是行业中被广泛验证的 Skill 模式，可按需参考和改造：

**Skill 示例一：需求 → 设计文档**

| 维度 | 说明 |
|------|------|
| **解决什么问题** | 拿到 PRD 后需要产出数据库设计 + API 设计，纯人工编写耗时长、容易遗漏字段 |
| **输入** | PRD 需求文档 |
| **输出** | 数据库设计（建表 SQL）+ API 接口文档 |
| **核心流程** | 提取业务实体和状态流转 → 设计表结构（遵循项目建表模板）→ 生成标准 CRUD 接口定义 |
| **适用场景** | 任何新模块的需求分析和设计阶段 |

**Skill 示例二：设计 → 代码生成**

| 维度 | 说明 |
|------|------|
| **解决什么问题** | Controller/Service/Mapper 三层重复代码，手写效率低、风格不统一 |
| **输入** | API 接口文档 + 参考模块代码 |
| **输出** | 完整 Java 代码（Controller/Service/Mapper/Domain/VO） |
| **核心流程** | 分析接口定义 → 参考现有模块风格 → 按约定分层生成代码 → 自动检查与文档一致性 |
| **适用场景** | 新模块的标准 CRUD 接口开发 |

**Skill 示例三：代码审查**

| 维度 | 说明 |
|------|------|
| **解决什么问题** | 人工审查耗时长、容易漏掉隐蔽问题 |
| **输入** | 代码变更 |
| **输出** | 分类审查评论（安全/规范/性能/逻辑） |
| **核心流程** | 多维度并行审查 → 综合评分 → 过滤低置信度评论 |
| **适用场景** | 任何代码提交前的质量检查 |

##### 关键设计思路

设计 Skill 时有三条通用原则值得关注：

**1. 流水线串联**

以上三个 Skill 可以串联使用，上一个的输出直接作为下一个的输入：

```
PRD 需求 ──→ 需求→设计 Skill ──→ 设计→代码 Skill ──→ 代码审查 Skill
              (输出：设计文档       (输出：全套代码)       (输出：审查评论)
               + API 文档)
```

串联的好处是减少手动传递信息时的遗漏和格式转换。

**2. 质量门禁**

在关键节点设置检查，不达标就阻断或标记：

- 设计文档阶段发现问题 → 修正后再进入编码（文档阶段改一行，代码阶段改一堆）
- 审查阶段过滤低质量评论 → 避免"AI 乱评论浪费开发者时间"

**3. 可复用**

同一个 Skill 改变输入即可用于不同模块：

- "需求→设计"的分析框架可用于任何业务模块
- "设计→代码"的分层生成流程可用于所有标准 CRUD
- "代码审查"的审查维度对所有后端代码通用

##### 引入时机

Skill 的编写和调试需要一定投入（一个 Skill 从设计到稳定通常需要 1-2 周迭代），建议在以下条件满足后试点：

- 团队成员已熟悉基础提示工程
- 已选定一个高频且流程固定的场景（推荐从后端 CRUD 接口生成开始）
- 由 1-2 位对 AI 较熟悉的同事先行编写验证，确认效果后推广

#### 2.3.3 Memory（记忆）— AI 记住项目上下文

##### 是什么

Memory 是跨会话持久化的记忆机制——AI 能记住项目架构、业务术语、历史决策，在后续对话中自动调用。

- **没有 Memory**：每次新开会话，AI 对项目一无所知，需要重新交代"我们用的什么技术栈、这张表叫什么……"
- **有 Memory**：AI 已了解项目的技术栈、核心约定和历史决策，自动关联背景知识

Memory 在长周期项目中的价值尤其明显——项目持续数月，不用每次对话都从零开始。

##### 什么值得记

Memory 不是"什么都记"，以下四类信息沉淀价值最高：

| 类型 | 示例 | 价值 |
|------|------|------|
| **技术约定** | "接口返回格式为 `{code, data, msg}`"，"金额用 BigDecimal" | AI 生成的代码自动符合规范 |
| **业务术语** | "订单状态流转：待审核→已审核→已发货→已完成" | AI 理解业务上下文，不用每次解释 |
| **架构决策** | "费用计算采用策略模式，而非 if-else 堆叠" | AI 生成的设计方案与已有架构一致 |
| **历史遗留** | "某旧表字段含义与标准定义不一致，新表不要参考" | 避免 AI 把历史问题当成规范延续 |

##### 如何沉淀

与日常工作同步进行，不需要专门花大块时间：

- **写设计文档时** → 把关键架构决策提取为 Memory
- **排查线上问题时** → 把沉淀的经验写入 Memory
- **新人上手时** → 把新人最常问的问题和答案写入 Memory

##### 引入时机

建议**中期（3-6 个月）**开始：

- Memory 的价值在 Rule 和 Skill 就位后才能真正体现
- 且需要积累过程，不是一次性配置完成的
- 在 Rule 和 Skill 使用中自然积累记忆素材

---

#### 2.3.4 能力引入节奏总结

三项能力的引入不是并行的，而是有先后顺序：

```
当前（0-3 个月）  ────  中期（3-6 个月）  ────  长期（6-12 个月）

  Rule                    Skill + Memory            MCP
  （立即可做）             （可复用工作流             （连接业务系统）
                           + 深度项目理解）
```

**为什么是这个顺序？**

- **Rule 第一**：配置成本最低，效果最直观。3-5 条配好，第二天就能感到变化。解决最急迫的问题——AI 生成代码不符合项目规范。
- **Skill 第二**：需要编写和调试投入，但效果可复用。有了 Rule 保证规范性后，通过 Skill 提升流程效率。
- **Memory 第三**：需要积累过程。在 Rule 和 Skill 使用一段时间后，自然积累了足够的素材。
- **MCP 第四**：MCP（Model Context Protocol）连接业务系统属于平台级建设，需要架构层面支持。作为业务开发部门，当前阶段目标是先把 Rule/Skill/Memory 三项基本能力用好，MCP 作为远期方向了解即可。

> **本章小结**
>
> 第二章从三个层次回答了"怎么用好 AI"：
>
> 1. **先守底线**（2.1）：三条使用规范——不贴敏感代码、AI 代码必 Review、关键决策标注来源
> 2. **再练基本功**（2.2）：提示工程四要素（背景、角色、目标、约束），让 AI 产出从"能用"到"好用"
> 3. **再上自动化**（2.3）：Rule 让 AI 遵守规范（当前可做），Skill 让 AI 执行固定流程（中期试点），Memory 让 AI 理解项目上下文（持续积累）。三项递进引入，形成完整的 Agent 辅助开发体系
