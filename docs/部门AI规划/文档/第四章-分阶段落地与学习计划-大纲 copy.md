# 四、分阶段落地与学习计划（大纲）

> 本章将"学什么"和"做什么"整合到同一推进路径中，每阶段明确：目标、学习内容、投入、产出。
>
> **说明**：
> - 本章不设定具体时间节点——推进节奏应由部门负责人根据团队情况和资源决定
> - 本章涉及的能力（Rule/Skill/Memory/提示工程）已在第二章详细说明，此处聚焦**先做什么、后做什么**，不重复概念解释
> - 第三章的风险分析应作为本章的前置阅读——了解风险后再规划推进

---

## 阶段总览

```
第一阶段              第二阶段              第三阶段              第四阶段
规范使用              规则固化              流程试点              能力扩展
──────────────────────────────────────────────────────────────────────────
使用规范 + 提示工程     Rule 配置             单个 Skill 试点        Skill 推广 + Memory
（全员参与，零门槛）     （低门槛，配完即生效）   （1-2人先行验证）       （试点成功后再推广）
```

**设计逻辑**：

- 四个阶段按**门槛从低到高**排列。第一阶段全员可直接参与，不需要额外学习；第二阶段只需配置几条规则；第三阶段需要有人投入时间编写 Skill，建议由对 AI 较熟悉的同事先行试点；第四阶段在试点成功后才向全员推广
- 每个阶段都是下一阶段的前置条件——团队需要先习惯写好提示词（第一阶段），才能理解 Rule 为什么有用（第二阶段）；Rule 稳定运行后，Skill 才有可靠的基础（第三阶段）
- MCP、RAG 等内容不列入本章，归入第五章"展望"——这些属于平台级建设，需要架构层面支持，不作为部门当前推进内容

---

## 第一阶段：规范使用

**阶段定位**：在不增加工具和预算的前提下，把已经在用的 Qoder 用得更好。全员参与，零额外投入。

**目标**：从"随便问"到"有目的地问"，建立安全的 AI 使用习惯。

### 学习内容

| 内容 | 说明 | 学习方式 |
|------|------|---------|
| 使用规范 | 敏感信息脱敏、AI 代码必 Review、关键决策标注 | 团队共识，一次宣讲即可 |
| 提示工程四要素 | 背景（Context）、角色（Role）、目标（Goal）、约束（Constraints） | 在日常使用中有意识地优化提问，非专门培训 |
| AI 的基本行为特点 | 幻觉、上下文丢失、知识截止——什么场景该信任、什么场景该验证 | 阅读第三章 3.1 节，结合实际使用体会 |

### 核心动作

1. **建立使用规范共识**：三条红线（不贴敏感代码、AI 代码必 Review、关键决策标注）在团队内明确并执行
2. **建立 Prompt 模板库**：从最高频的三类场景开始——修 Bug、新接口开发、SQL 优化。模板格式：场景名称 + 模板内容 + 使用说明。共享文档即可，不需要专门工具
3. **收集好的提示词**：团队成员在使用中遇到产出质量明显高于平时的情况，主动沉淀到模板库

### 投入

零额外成本。Qoder 账号已有，模板库用共享文档，规范宣导利用例会时间。

### 产出

- Prompt 模板库（共享文档，至少含修 Bug / 新接口 / SQL 优化三类模板）
- 使用规范团队共识
- 团队成员能稳定写出包含"背景+目标+约束"的结构化提示词

### 阶段完成的标志

团队成员在日常工作中**默认使用模板化提示词**而非随口提问，且能感受到产出质量的明显提升。

---

## 第二阶段：规则固化

**阶段定位**：将团队最高频的规范要求固化为 Rule，让 AI 自动遵守。配置成本低，效果立竿见影。

**前置条件**：第一阶段已稳定运行（团队已习惯结构化提示词，使用规范已建立共识）。

**目标**：AI 生成的代码在安全性、接口格式、技术栈方面自动符合部门规范，减少 Code Review 中的低级返工。

### 学习内容

| 内容 | 说明 | 学习方式 |
|------|------|---------|
| Rule 的概念与作用 | Rule 是什么、和提示词中手动写约束的区别 | 阅读第二章 2.3.1 |
| Rule 的配置方法 | Qoder 中如何配置项目级/用户级 Rule | 参考 Qoder 官方文档，由 1 人先行配置后分享经验 |
| 如何识别"值得写成 Rule"的规则 | 高频返工点、明确无争议的规范 = 值得写；偶发问题、有争议的 = 不适合 | 团队讨论，从 Code Review 记录中筛选 |

### 核心动作

1. **梳理候选规则**：回顾近 3 个月的 Code Review 记录，找出最常见的返工点——如 SQL 拼接、接口返回格式不统一、命名不一致、Java 版本语法混用
2. **筛选并编写 3-5 条 Rule**：从候选列表中选出"频率高 + 影响大 + 规则明确无争议"的条目，写成 Rule 配置
3. **试运行并收集反馈**：使用 2-4 周后，收集团队反馈——哪些 Rule 确实减少了返工？哪些需要调整？是否有新增需求？

### 投入

每条约 5-10 分钟编写配置。由 1 人负责梳理和初版编写，全员参与候选规则讨论。

### 产出

- 3-5 条项目级 Rule 上线并稳定运行
- Rule 维护说明（如何新增、修改、删除 Rule）

### 阶段完成的标志

连续 2-4 周 Code Review 中不再出现 Rule 已覆盖的低级问题（如 SQL 拼接、接口格式不一致）。

---

## 第三阶段：流程试点

**阶段定位**：选择 1 个最高频的开发场景，将工作流程固化为 Skill 模板。由 1-2 位对 AI 较熟悉的同事先行试点，验证效果后再考虑推广。

**前置条件**：第二阶段 Rule 已稳定运行（Skill 依赖 Rule 提供基础规范保障）；有 1-2 位同事愿意投入时间试点。

**目标**：验证"将重复性工作流固化"模式在团队中的可行性。

### 学习内容

| 内容 | 说明 | 学习方式 |
|------|------|---------|
| Skill 的概念与设计原则 | Skill 是什么、与普通对话的区别、流水线串联/质量门禁/可复用三条原则 | 阅读第二章 2.3.2 |
| Skill 的编写方法 | 如何定义输入输出、设计执行步骤、内建质量检查 | 参考 Qoder 官方文档，由试点同事先行研究 |
| 试点场景的选择标准 | 流程固定、频率高、输入输出明确、成功标准清晰 | 团队讨论确定 |

### 核心动作

1. **选定试点场景**：推荐从后端标准 CRUD 接口生成开始——这是部门最高频的场景，流程固定（Controller → Service → Mapper），成功标准明确（生成的代码能编译通过且符合项目规范）
2. **1-2 人试点编写**：由对 AI 较熟悉的同事负责编写和调试，预期一个 Skill 从设计到稳定可用约需要 1-2 周迭代
3. **验证效果**：在实际开发任务中使用该 Skill，对比纯手工开发的效率和质量
4. **决定是否推广**：根据试点效果，由团队共同决定是否进入第四阶段

### 投入

试点同事需要投入一定时间学习和调试（1 个 Skill 约 1-2 周，兼职进行）。其他团队成员此阶段无额外投入。

### 产出

- 1 个验证通过的 Skill（建议为后端 CRUD 接口生成）
- 试点效果评估（效率提升数据、遇到的问题、适用场景和不适用场景）
- 经验总结文档（供后续推广参考）

### 阶段完成的标志

试点 Skill 在至少 1 个实际开发任务中成功使用，产出质量得到团队认可，且试点同事认为值得推广。

> **注意**：第三阶段的核心是**验证可行性**，而非追求 Skill 的数量。一个验证通过且团队认可的 Skill，比三个勉强能跑但没人用的 Skill 更有价值。如果试点效果不理想，团队可以决定暂不进入第四阶段——这不是失败，而是基于事实的理性决策。

---

## 第四阶段：能力扩展

**阶段定位**：在试点成功的基础上，将 Skill 模式推广到更多场景，并开始沉淀项目 Memory。此阶段为**可选**——是否推进由团队根据第三阶段结果决定。

**前置条件**：第三阶段试点效果得到团队认可；团队有意愿继续投入。

**目标**：形成一套团队自己的 AI 辅助开发方法，从"能用"到"好用"。

### 学习内容

| 内容 | 说明 | 学习方式 |
|------|------|---------|
| Skill 的设计进阶 | 如何将试点经验复用到其他场景、如何设计 Skill 间的流水线串联 | 以试点 Skill 为模板，举一反三 |
| Memory 的沉淀方法 | 什么值得记（技术约定/业务术语/架构决策/历史遗留）、如何与日常工作同步 | 阅读第二章 2.3.3，在日常文档工作中顺手沉淀 |
| 多 Agent 协作概念 | 了解并行审查、评分过滤等模式的基本概念 | 技术分享或文档阅读 |

### 核心动作

1. **扩展 Skill 库**：从 1 个试点扩展到 3-5 个成熟 Skill。推荐扩展方向：代码审查 Skill、文档一致性验证 Skill、SQL 生成 Skill
2. **沉淀项目 Memory**：在日常开发中，将关键决策、业务规则、历史遗留等信息顺手沉淀为 Memory
3. **建立经验分享机制**：定期（如每月）分享 AI 使用中的好方法和遇到的坑

### 投入

每个新增 Skill 约 1-2 周迭代（由 1-2 人兼职）。Memory 沉淀与文档工作同步，零额外投入。经验分享利用例会时间。

### 产出

- Skill 库（3-5 个验证通过的 Skill）
- 项目级 Memory 积累（技术约定 + 业务知识 + 历史决策）
- 经验分享记录

### 阶段完成的标志

Skill 模式已成为团队的日常工具，新人上手时可以从 Skill 库中直接使用；项目 Memory 开始在日常开发中产生价值（如 AI 自动使用正确的技术约定）。

## 不纳入本章的内容

以下能力在第二章中有概念性介绍，但**不作为本章的推进内容**，归入第五章"展望"：

| 能力 | 不纳入本章的原因 |
|------|----------------|
| **MCP（模型上下文协议）** | 属于平台级建设，需要架构层面支持，超出业务开发部门的推进范畴 |
| **RAG（检索增强生成）** | 需要搭建知识库基础设施，属于平台型工作，当前阶段以了解概念为主 |
| **多 Agent 协作系统** | 第四章仅涉及 Skill 内部的多 Agent 协作（如代码审查的并行审查），独立的 Multi-Agent 系统编排属于远期方向 |

---

> **本章小结**
>
> 四个阶段的核心是**渐进式推进**——不是一口气把 AI 能力全铺开，而是一步一步验证、一步一步扩展：
>
> 1. 先用好手头的工具（第一阶段：零成本，立即开始）
> 2. 再让 AI 自动遵守规范（第二阶段：低投入，效果好）
> 3. 试点一个固定流程的自动化（第三阶段：先验证，再决定）
> 4. 试点成功后逐步扩展（第四阶段：可选，按需推进）
>
> 每一步都有明确的前置条件和阶段完成标志，团队可以根据实际情况灵活调整节奏。推进的责任在于部门负责人，本大纲提供的是方向和框架。
