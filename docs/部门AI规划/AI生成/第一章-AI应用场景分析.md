# 一、AI 应用场景分析

> 本章回答"哪些工作能结合 AI 提升效率"。按后端视角编写，前端内容已预留位置。
>
> 文中实际案例均来自 WMS 项目费用管理模块（WP08 操作费/仓库杂费/仓租规则、WP09 结算管理）的开发实践。在该模块开发中，已使用 Qoder 的 4 个 Skill 形成了一条从需求到代码审查的完整 AI 辅助开发流水线。

## 1.1 需求理解与任务拆分

开发工作中最大的浪费不是写代码慢，而是写了半天发现理解错了需求。需求阶段的遗漏是返工的最大来源——一个边界条件没考虑到，上线后就是一个 Bug。AI 在这个阶段可以充当"第二双眼睛"，帮助开发在动手之前把需求想得更完整。

### 1.1.1 需求文档理解

**现状痛点**

跨境电商的业务规则复杂，WMS 的费用管理模块涉及 14 种计费单位、5 种费用状态流转、汇率折算、核销规则等，需求文档往往只描述了主流程，边界情况需要开发自己补。开发时经常需要反复找产品确认："如果客户选了'全部'怎么处理？""汇率为 0 怎么办？""状态从草稿直接到生效允许吗？"

**AI 如何介入**

将需求文档（或 PRD）贴给 AI，附加项目背景说明，AI 可以：

1. 梳理完整的业务流程（含异常分支）
2. 列出需求文档中没有明确提到的边界条件
3. 从数据一致性角度提问（如"扣减库存时并发怎么办？"）
4. 从开发角度提出实现层面的疑问

这不是替代产品经理，而是让开发在动手之前想得更完整。

**实际案例**

在 WMS 项目中，我们基于 Qoder 搭建了 `req-to-api` Skill：输入 PRD 需求文档后，AI 按 12 个维度提取需求要素，输出设计文档 + 建表 SQL + API 接口文档。以 WP08 操作费模块为例，PRD 中描述了"新建操作费时需要选择计费仓库和客户"，AI 补充提出了：客户代码支持多选时是否需要"全部"选项、有效时间段是否有最小单位限制、重复规则校验的维度组合是什么等问题，这些在后续开发中都被证实是必须明确的边界条件。

### 1.1.2 需求文档一致性验证

**现状痛点**

一个完整的需求从 PRD 到落地，会经过多份文档：需求文档、设计文档、API 接口文档、建表 SQL。这几份文档分别在不同阶段产出，很容易出现不一致——需求里有 10 个筛选条件，设计文档漏了 2 个，API 文档又多了 1 个需求里没有的字段。这种不一致往往到编码甚至测试阶段才发现，修复成本很高。

**AI 如何介入**

将四份文档同时提供给 AI，让它交叉比对，可以做到：

1. 筛选/查询条件完整性检查
2. 列表字段和导出字段覆盖检查
3. 详情/明细字段覆盖检查
4. 业务流程完整性检查
5. 状态枚举和业务规则一致性检查
6. SQL 字段类型/精度/索引 vs 需求检查

**实际案例**

在 WMS 项目中，我们搭建了 `cross-check-docs` Skill 进行 6 维度文档一致性核验。问题按严重程度分级：P0（阻塞开发，必须立即修正）、P1（需修正，不阻塞但上线前必须解决）、P2（优化建议）。AI 不仅能发现问题，还能自动修正设计文档中与需求不一致的部分。在 WP08 开发过程中，该技能发现过操作费的有效时间段校验逻辑在需求文档和设计文档中的描述不一致，提前修正避免了返工。

### 1.1.3 任务拆分

**现状痛点**

拿到需求后需要拆分为可执行的开发子任务。新人容易遗漏步骤（比如忘了"先建基础数据表再建业务表"），或者不清楚模块间的依赖关系（比如"操作费依赖费用类型和计费引擎，不能先做"）。

**AI 如何介入**

描述需求和现有系统结构，让 AI 生成开发任务清单，标注依赖关系和执行顺序。

**实际案例**

WP08 费用管理模块的需求涉及多个子模块，AI 辅助拆分出的依赖链为：

```
FeeType（费用类型）← 无依赖，先做
    ↓
Customer 实体改造 ← 新增 settle_currency 字段
    ↓
计费引擎 + 汇率工具 ← 核心计算逻辑
    ↓
OperationFee（操作费）← 依赖以上三项
    ↓
WarehouseMiscFee（仓库杂费）← 同上
    ↓
WarehouseRentRule（仓租规则）← 同上
```

按这个顺序开发，每个模块的前置依赖都已经就绪，避免了"做到一半发现缺一个基础模块"的情况。

## 1.2 日常编码

### 1.2.1 后端

#### A. 已上线系统维护

已上线系统的维护工作有几个特点：代码可能几经人手、文档不全、改动必须谨慎避免影响现有功能。AI 在以下环节可以提供有效辅助。

### Bug 修复

**现状痛点**

这是日常工作中频率最高的场景。收到 Bug 反馈后，需要查看日志、跟踪调用链、定位根因。如果代码经过多人维护、没有人完全熟悉全貌，排查耗时会更长。以 WMS 出库模块为例，一个出库完成后的操作费计费异常，调用链涉及 OutboundOrderService → 计费引擎 → fee_transaction 表，跨越多个 Service 和 Mapper。

**AI 如何介入**

将报错日志 + 相关代码贴给 AI，它可以：

1. 解析异常堆栈，定位到具体的类和方法
2. 结合代码逻辑分析可能的根因
3. 给出最小改动方案（不是大面积重写）

**实际案例**

WMS 费用模块出库完成触发操作费计费时偶发异常。将异常堆栈 + OutboundOrderService + 计费引擎代码贴给 AI，它分析了从出库完成事件到 fee_transaction 写入的完整调用链，定位到问题出在计费引擎处理某些特殊计费单位时的空值判断缺失。AI 给出的修复方案仅涉及 2 行代码的空值判断补充，而非重写整个计费逻辑。

### 功能优化 / 小需求

**现状痛点**

在现有逻辑上加字段、改业务规则，最大的风险是"改了一处、影响另一处"。尤其是实体类被多个模块引用的情况下，新增一个字段需要评估所有关联代码。

**AI 如何介入**

将现有代码贴给 AI，说明要做的改动，AI 会输出增量修改方案，并标注可能受影响的关联代码。

**实际案例**

WP08 开发中需要在 Customer 实体新增 `settle_currency`（结算币种）字段。AI 分析了 Customer 实体被引用的所有位置（Controller、Service、Mapper、VO），列出需要同步修改的文件清单，并生成了每个文件的增量代码，避免了遗漏。

### 数据修复 / 脚本执行

**现状痛点**

生产环境偶发数据异常——某条费用流水状态卡在中间态、某条账单的核销金额对不上。需要写一次性修复脚本，这类脚本的难点不在于逻辑复杂，而在于"不能出错"——写错一行可能导致二次数据污染。

**AI 如何介入**

描述数据异常的表、字段和预期状态，AI 生成修复 SQL。开发者审核确认后执行。

**实际案例**

费用流水表 fee_transaction 中出现状态异常数据（应为"已出账"但实际为"未出账"）。AI 根据表结构和状态流转规则生成了修复 SQL，并在 WHERE 条件中加了严格的时间范围和状态过滤，确保只修复目标数据。

### 旧代码理解

**现状痛点**

接手别人写的或几年前的代码，缺少文档，不知道"为什么这样写"。直接改怕踩坑，不改又无法推进。

**AI 如何介入**

选中代码，让 AI 解释逻辑、梳理调用链。AI 可以逐行解释代码意图，并标注可能的"坑"。

**实际案例**

WMS 部分旧表（如 b_bank_card）的 del_flag 字段用了 0 代表存在、2 代表删除，而非项目通用的 0/1 规范。通过 AI 分析旧代码和表结构，确认这是历史遗留写法，新模块统一采用 0/1 规范。这类信息如果没有 AI 辅助梳理，新人很容易沿用旧写法导致数据混乱。

#### B. 新功能 / 新系统开发

新功能开发的特点是模板代码多、重复性高。AI 在减少机械敲码方面效果明显。

### 新接口开发

**现状痛点**

Spring Boot 项目的接口开发模式高度固定：Controller 接收参数 → Service 处理业务逻辑 → Mapper 操作数据库。从表结构到可用的 REST 接口，中间是大量的模板代码。尤其是 WMS 费用模块，操作费有 14 种计费单位、仓库杂费有 5 种杂费类型、仓租规则有 2 种计费类型，每个模块都是 Controller + Service + Mapper + Domain + Cmd + Query + VO 一整套。

**AI 如何介入**

提供 API 接口文档 + 参考模块代码，AI 按照现有项目的代码风格和架构分层，生成完整的接口代码。

**实际案例**

在 WMS 项目中，我们搭建了 `api-to-code` Skill：输入 API 文档 + 参考模块后，AI 按 7 个阶段生成代码——文档分析、参考实现研究、代码生成、日志标准、自验证、文档交叉核对、常见问题修复。

以 WP08 操作费模块为例，AI 生成的代码包括：

- **Controller**：标准 CRUD + 5 个状态操作（确认、发布、作废、无效、复制）+ 批量更新有效期 + 日志查询 + 导出，共 14 个接口
- **Service**：业务逻辑 + 事务控制 + 重复规则校验（计费仓库 + 客户代码 + 有效时间段不允许相同计费规则）
- **Mapper**：动态 SQL + 分页查询 + 仓库权限过滤（Query warehouseIds + Mapper XML IN 过滤）
- **Domain / Cmd / Query / VO**：实体类、命令对象、查询对象、视图对象

开发者需要做的是补充业务特有的校验逻辑和计费引擎的策略模式实现，而非从零写起。

### SQL 编写

**现状痛点**

WMS 的费用查询涉及多表关联——fee_transaction（费用流水）、fee_transaction_detail（费用明细）、customer（客户）等，查询条件包括业务类型、费用状态、核销状态、时间范围等多个维度。手写这类动态 SQL 容易遗漏条件或写错联表关系。

**AI 如何介入**

用自然语言描述查询需求（如"查出某仓库下所有未出账的操作费，按客户分组汇总金额"），AI 输出对应的 SQL 或 MyBatis 动态 XML。

**实际案例**

WP09 费用查询模块需要支持 7 种业务类型（订单、入库单、退件、认领、认领销毁、不良品处理、其他），每种业务类型的关联表和查询条件不同。AI 根据业务类型和状态条件生成了动态查询 SQL，包括按币种汇总金额的聚合逻辑。

### 批量数据处理

**现状痛点**

WMS 中常有批量操作需求——操作费批量更改有效期、费用类型批量导入。这类功能逻辑不复杂但代码繁琐，涉及 Excel 解析、数据校验、批量更新。

**AI 如何介入**

提供字段映射和业务规则，AI 基于项目现有的 EasyExcel + ExcelUtil 工具类生成批量处理代码。

**实际案例**

WP08 操作费的"批量更改有效期"功能：选择多条非作废、非已过期的操作费记录，以替换方式更新有效期时间。AI 生成了完整的批量处理代码，包含状态过滤校验和事务控制。费用类型模块也支持批量导入（提供模板下载），AI 基于 EasyExcel 生成了导入解析 + 数据校验逻辑。

### 定时任务开发

**现状痛点**

WMS 中有多个定时任务——仓租日统计、订单状态刷新等。这类任务的模板代码多（任务骨架、异常处理、日志记录），但每个任务的触发条件和业务逻辑不同。手写容易遗漏异常处理，导致任务静默失败。

**AI 如何介入**

描述定时任务的触发条件和业务逻辑，AI 生成任务骨架代码，包含异常处理、日志记录和失败重试机制。

**实际案例**

WP09 仓租日统计定时任务：每天 00:00:00 执行，统计维度为计费仓库 + 客户代码 + 费用发生时间。统计范围包括可用库存、不良品库存、暂存库存。体积计算公式为 `round((∑SKU库存数量 × SKU体积) / 10^6, 4)`，单位 m³。AI 生成了定时任务骨架代码，包含库存数据查询、体积计算、异常处理（如某 SKU 体积数据缺失时的处理）和日志记录。

### 第三方接口对接

**现状痛点**

跨境电商场景下，经常需要对接物流（如 DHL、FedEx）、支付等外部服务的 API。这些 API 文档通常是英文、格式各异，解析文档 + 编写对接代码 + 处理各种异常情况，耗时较长。

**AI 如何介入**

将对方 API 文档（或关键片段）贴给 AI，它可以：

1. 解析请求参数和响应格式
2. 生成 HTTP 调用代码（含异常处理和重试逻辑）
3. 生成对应的 DTO 类

开发者只需核对关键参数和异常分支。

## 1.3 代码质量

> 部门无单元测试（由专门测试岗位进行黑盒测试），且不建议对老代码进行大面积重构。此处聚焦可行性高的事项。

### 代码审查

**现状痛点**

代码审查的质量取决于审查者的精力和经验。人工审查容易遗漏安全问题（如 SQL 注入、空指针），尤其是代码量大、改动文件多的情况下。部门不建议大面积重构老代码，但新代码的安全性和规范性需要保障。

**AI 如何介入**

AI 可以作为"第一道审查"，扫描代码变更并标注风险点：

- **SQL 注入检查**：扫描 Mapper 层，标记拼接 SQL 的位置
- **空指针风险**：标记未做 null 判断的链式调用（如 `dto.getXxx().getYyy()`）
- **异常处理遗漏**：检查 Service 方法是否有 try-catch 包裹关键操作
- **事务边界**：标记缺少 `@Transactional` 注解的写操作组合

AI 不替代人工审查，而是让审查者把精力集中在业务逻辑的正确性上，安全性和代码规范层面的问题交给 AI 初筛。

**实际案例**

在 WMS 项目中，我们搭建了 `code-review` Skill：5 个 Sonnet agent 并行独立审查代码变更，每个 agent 从不同维度（安全性、规范性、性能、业务逻辑、可维护性）给出审查意见。审查结果由 Haiku agent 评分，仅 80 分以上的评论才会发布。该技能已用于 WP08 操作费+仓库杂费、仓租费用、仓租规则三个模块的代码审查。

这种"多 agent 并行审查 + 评分过滤"的模式相比单次人工审查有两个优势：覆盖维度更全面（5 个 agent 各有侧重），且低置信度的噪音评论被自动过滤，审查者看到的都是高质量反馈。

## 1.4 问题排查

### Bug 定位（后端）

**现状痛点**

线上 Bug 的排查往往需要跟踪完整的调用链，尤其涉及多服务、多表操作时。以 WMS 出库模块为例，一个"出库完成但操作费没生成"的 Bug，调用链可能是：出库完成事件 → OutboundOrderService → 计费引擎 → 根据 14 种计费单位匹配规则 → 写入 fee_transaction。任何一个环节出问题，最终表现都一样——"费用没生成"。

**AI 如何介入**

将完整异常堆栈 + 出错的 Service/Mapper 代码一并贴入 AI，它可以：

1. 解析异常堆栈，定位到具体的类和方法
2. 结合代码逻辑分析可能的根因（是空值？是事务未提交？是规则未匹配？）
3. 给出排查方向和最小改动方案

**实际案例**

WMS 出库完成触发操作费计费涉及 OutboundOrderService → 计费引擎 → fee_transaction 多层调用。一次计费异常中，AI 通过分析调用链定位到计费引擎在处理"票总重+件数"这种复合计费单位时，件数参数未正确传递导致计算结果为 null。排查方向从"整个计费流程"缩小到了"复合计费单位的参数传递"这一具体环节。

### 慢接口 / SQL 优化

**现状痛点**

WMS 的报表和查询接口随着数据量增长会越来越慢。传统做法是 DBA 或资深开发手工分析 `EXPLAIN` 结果，门槛较高。

**AI 如何介入**

将 SQL 语句 + `EXPLAIN` 输出贴给 AI，它可以：

1. 判断是全表扫描、索引失效还是联表过多
2. 给出优化建议（加索引、改写 SQL、拆分为多次查询、代码层加缓存）

### 数据库死锁 / 锁等待

**现状痛点**

WMS 中库存扣减、订单状态更新等并发操作偶发死锁。死锁日志（`LATEST DETECTED DEADLOCK`）结构复杂、可读性差。

**AI 如何介入**

将死锁段贴给 AI，它可以解析出：哪两个事务互相等待、等待的是哪张表的哪个锁、建议的事务调整（如缩小事务范围、统一加锁顺序）。

### 环境 / 部署问题

**现状痛点**

新人配环境各种报错，老员工反复被打断。

**AI 如何介入**

将完整报错信息（含操作系统和软件版本）贴给 AI，它给出针对性的解决步骤，而非通用的"重启试试"。

## 1.5 上线支持与日常运维

### 上线检查清单

**现状痛点**

每次上线靠经验回忆检查项——数据库脚本执行了吗？配置文件改了吗？定时任务关了吗？人一忙就容易遗漏。

**AI 如何介入**

将 SVN 提交记录贴给 AI，结合本次变更的文件列表，AI 生成针对性的上线检查清单。比如：改了 Mapper XML → 检查 SQL 是否需要在目标环境执行；改了定时任务类 → 检查任务是否需要暂停/重启。

### 监控告警初筛

**现状痛点**

半夜告警要爬起来查，压力大。很多告警其实是非紧急问题（如磁盘空间告警但离阈值还远），但不敢不看。

**AI 如何介入**

将告警信息和相关日志贴给 AI 做第一轮分析，快速判断严重程度和可能原因，帮助决定是否需要立即处理。

## 1.6 其他日常工作

### 技术调研

**现状痛点**

从搜索引擎到博客到官方文档，比较一个技术方案往往要花一整个下午。

**AI 如何介入**

描述调研需求（如"MyBatis-Plus 和原生 MyBatis 在复杂动态 SQL 场景下哪个更合适"），AI 列出主流方案及优劣对比，省去大量搜索时间。

### 周报 / 日报

**现状痛点**

纯手工回忆本周做了什么，容易遗漏。

**AI 如何介入**

将 SVN 提交记录贴给 AI，整理为周报格式。可以按模块、按功能点分类，比手动回忆更完整。

### 会议纪要

**现状痛点**

开会时记笔记分心，会后追忆容易遗漏关键决策和待办。

**AI 如何介入**

录音转文字后，AI 提取关键决策点和待办事项，分配到具体负责人。

## 本章小结

以上场景在 WMS 费用管理模块的开发中已有实际验证，形成了完整的 AI 辅助开发流水线：

```
PRD 需求 ──→ req-to-api ──→ cross-check-docs ──→ api-to-code ──→ code-review
           (需求分析+设计文档    (6维度文档一致性      (全套Java代码)    (5 agent
            +建表SQL+API文档)     核验+自动修正)                       并行审查)
```

按引入优先级，建议从以下顺序推进：

1. **最先做**（零成本，当前即可）：Bug 修复与定位、旧代码理解、SQL 编写、需求文档理解 —— 团队已经在用 Qoder 做类似的事，只需有意识地系统化
2. **优先做**（少量规范即可）：代码审查、任务拆分、上线检查清单 —— 建立审查检查清单和 Prompt 模板
3. **逐步做**（需要搭建）：监控告警初筛 —— 需要与运维配合接入日志
