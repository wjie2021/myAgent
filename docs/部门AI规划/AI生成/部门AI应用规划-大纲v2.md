## 背景

- 公司为跨境电商企业
- 研发部门负责 ERP、WMS、HRM 等内部业务系统
- 技术栈：Vue（前端）+ Spring Boot（后端），前后端岗位分离
- 版本管理：SVN
- 测试模式：无单元测试，由专门测试岗位进行黑盒测试
- 现状：公司已提供 Qoder 账号，团队成员已在使用 VS Code/IDEA 插件进行 AI 对话和代码修改
- 目标：从研发部门开始，推动公司 AI 转型

## 一、当前工作中有哪些环节可以用 AI 提效（重点）

> 回答领导最关心的问题："哪些工作能结合 AI 提升效率"

### 1.1 日常编码

> 部门前后端岗位分离，且日常工作中已上线系统的维护（修 Bug、功能优化）占比很大，故分开列出。

#### A. 已上线系统维护（占比高，重点）

| 场景 | 岗位 | 现状痛点 | AI 如何提效 | 预期效果 |
|------|------|---------|------------|---------|
| Bug 定位与修复 | 后端 | 看日志、跟踪调用链耗时，老代码不熟悉 | 贴报错日志 + 相关代码 → AI 分析根因并给出修复建议 | 排查时间明显缩短 |
| Bug 定位与修复 | 前端 | 页面异常但难以定位是数据问题还是渲染问题 | 贴截图/报错 + 组件代码 → AI 辅助定位 | 减少前后端扯皮 |
| 功能优化/小需求 | 后端 | 在现有逻辑上加新字段、改业务规则，怕改出问题 | AI 理解现有代码逻辑后给出最小改动方案 | 降低改动风险 |
| 功能优化/小需求 | 前端 | 列表加筛选项、表单加字段等高频小改动 | AI 在现有页面基础上生成增量代码 | 小需求快速交付 |
| 数据修复/脚本执行 | 后端 | 生产数据出问题，需要写临时 SQL 或脚本修复，手写容易出错 | AI 根据问题描述生成数据修复 SQL/脚本，人工审核后执行 | 减少二次事故风险 |
| 旧代码理解 | 通用 | 接手别人写的/几年前的代码，缺少文档 | 选中代码 → AI 解释逻辑、画出调用关系 | 减少上手时间 |

#### B. 新功能 / 新系统开发

| 场景 | 岗位 | 现状痛点 | AI 如何提效 | 预期效果 |
|------|------|---------|------------|---------|
| 新页面开发 | 前端 | Vue 列表页、表单页大量模板代码 | AI 根据需求描述生成页面骨架，人工调样式和逻辑 | 节省 30-50% 模板时间 |
| 新接口开发 | 后端 | Controller/Service/Mapper 三层重复代码 | AI 根据表结构或注释生成全套接口代码 | 减少机械敲码 |
| SQL 编写 | 后端 | 复杂报表查询、多表联查调试耗时长 | 自然语言描述需求 → AI 输出 SQL | 降低 SQL 调试时间 |
| 批量数据处理 | 后端 | ERP/WMS 中常有批量导入导出、数据同步任务，逻辑繁琐 | AI 根据字段映射和业务规则生成处理代码 | 批量类需求开发提速 |
| 定时任务开发 | 后端 | 库存同步、订单状态刷新等定时任务模板代码多 | AI 生成定时任务骨架 + 异常处理 + 日志 | 减少遗漏异常处理 |
| 第三方接口对接 | 后端 | 对接物流、支付等外部 API，文档全是英文/格式不一 | AI 根据对方文档生成对接代码和异常处理 | 减少文档阅读和试错时间 |
| 前端组件开发 | 前端 | 公共组件封装费时，重复写类似的逻辑 | AI 辅助生成可复用组件框架 | 组件开发效率提升 |

### 2.2 使用规范建议

> 工具用起来之前，先把红线划清楚，避免好心办坏事。

- **不要直接贴敏感代码**：数据库密码、API Key、服务器 IP 等先用占位符替换
- **AI 生成的代码必须 Review**：AI 是辅助，责任在写代码的人。尤其是后端涉及数据库操作、金额计算、权限校验的代码
- **AI 生成内容标注**：关键决策点标注是否使用了 AI 辅助，便于追溯

### 2.3 提示工程（Prompt Engineering）— 从"随便问"到"有目的地问"

> 基础对话人人都会，但同样用 AI，好的提示词和随口的提问，产出质量差距巨大。提示工程就是学会**给 AI 交代背景、明确目标、设定约束、指定输出格式**，让 AI 从"能用"变成"好用"。

#### 为什么提示词是核心能力

```
随便问：                结构化提示词：
"帮我写个接口"    →      "我们用的是 Spring Boot + MyBatis，
                         返回格式统一 {code, data, msg}，
                         请为采购单表 PO_HEADER 生成查询接口，
                         Controller/Service/Mapper 三层，带分页和按状态筛选"
                          
结果：能用但得改半天        结果：基本直接可用
```

Agent 的高级能力（Skill、Rule、Memory）本质上是把好的提示词**固化下来复用**。提示词写不好，后面 Skill 和 Rule 的效果都打折扣。

#### 提示词的四个要素

| 要素 | 作用 | 示例 |
|------|------|------|
| **背景（Context）** | 告诉 AI 项目环境、技术栈、现状 | "这是一个已上线的 ERP 系统，Java 8 + Spring Boot，不要引入 Java 9+ 语法" |
| **角色（Role）** | 让 AI 以特定身份思考 | "你是一个有 5 年跨境电商后端开发经验的程序员" |
| **目标（Goal）** | 明确要产出什么 | "为 WMS 的出库单模块新增一个批量导出 Excel 的接口" |
| **约束（Constraints）** | 限制范围、格式、禁区 | "不要修改现有表结构，导出字段包含单号/日期/状态/操作人，用 EasyExcel" |

#### 部门工作中的典型提示词模板示例

**后端修 Bug 场景**（高频）：
```
【背景】ERP 系统，Spring Boot，MyBatis，SVN 管理
【问题】销售单审核后库存没扣减，报错日志如下：...
【相关代码】[贴 Service 层 + Mapper 层代码]
【要求】分析根因并给出最小改动方案，不要大面积重写
【注意】涉及库存扣减逻辑，务必确认事务边界是否正确
```

**后端新接口开发**：
```
【背景】WMS 系统，Java 8 + Spring Boot，现有表 wms_outbound_order
【需求】新增批量导出 Excel 接口，导出字段：单号/日期/状态/操作人/备注
【约束】用 EasyExcel，GET 请求，参数放在 URL 上，不要改表结构
【输出】Controller/Service/Mapper 完整代码
```

**SQL 优化场景**：
```
【背景】ERP 报表模块，MySQL 5.7，数据量百万级
【慢 SQL】[贴 SQL 和 EXPLAIN 结果]
【要求】分析慢的原因并给出优化方案（索引/改写 SQL/代码层调整）
```

#### 团队落地方式

- **建立部门 Prompt 模板库**：将上面这类常用场景的提示词模板化，团队成员填空即用
- **好的提示词共享**：遇到一次写得好、产出质量高的提示词，沉淀到模板库
- **新人先学提示词**：入职先配好 AI 工具，第二件事就是学怎么写提示词

**引入时机**：当前即可开始。提示工程不需要额外工具，是 ROI 最高的一项投入。

### 2.4 Agent 能力进阶使用（分阶段引入）

> 基础使用（对话框、IDE 插件）只是起点。Agent 工具提供了 Skill（技能）、Rule（规则）、Memory（记忆）等能力，可以让 AI 从"一次性问答"升级为"持续理解项目上下文的工作伙伴"。以下按引入顺序说明。

#### Skill（技能）— 可复用的自动化工作流模板

**是什么**：把重复性的工作流程固化为一个"技能"，一键触发，AI 按预定步骤执行。

**部门应用举例**：

| 技能名称 | 触发场景 | AI 自动做什么 |
|---------|---------|-------------|
| 生成 CRUD 接口 | 新增数据表功能（后端） | 根据表名/字段 → 生成 Controller、Service、Mapper 全套代码 |
| 生成 Vue 页面 | 新增列表/表单页面（前端） | 根据需求描述 → 生成页面骨架、组件和路由配置 |
| 代码审查 | 提交代码前 | 按部门编码规范检查安全问题、SQL 注入、空指针、异常处理遗漏 |

**引入时机**：中期（3-6 个月），团队成员熟悉基础 Prompt 后，由 1-2 人试点编写，验证有效后推广。

#### Rule（规则）— 让 AI 始终遵循部门规范

**是什么**：项目级或用户级的持久化指令，AI 在每次对话中自动加载，确保输出始终符合团队约定。

**部门应用举例**：

| 规则类型 | 内容示例 | 效果 |
|---------|---------|------|
| 编码规范 | "数据库操作必须使用 MyBatis，禁止拼接 SQL 字符串" | AI 生成的代码自动符合安全要求 |
| 接口约定 | "RESTful 接口返回格式统一用 `{code, data, msg}`" | 前后端接口风格一致 |
| 命名规范 | "Vue 组件用 PascalCase，API 路径用 kebab-case" | 减少 Code Review 中的命名争议 |
| 技术栈约束 | "仅使用 Java 8 语法，不引入 Lombok 以外的注解处理" | 避免 AI 引入项目不支持的写法 |

**引入时机**：当前即可开始。先梳理 3-5 条最常见的返工点，写成 Rule，让大家体验"配一次、一直生效"的便利。

#### Memory（记忆）— AI 记住项目上下文

**是什么**：跨会话持久化的记忆，AI 能记住项目架构、业务知识、历史决策，不用每次重新交代背景。

**部门应用举例**：

| 记忆内容 | 用途 | 示例 |
|---------|------|------|
| 项目架构 | AI 理解模块划分，给出的建议不偏离现有结构 | "ERP 系统分为基础数据、采购、销售、库存、财务五大模块" |
| 业务术语 | AI 理解公司特有的业务概念 | "PO = 采购单，SO = 销售单，ASN = 预到货通知" |
| 历史决策 | AI 知道之前为什么这样设计，不重复踩坑 | "库存扣减放在发货环节而非下单环节，因为支持部分发货" |
| 同事分工 | AI 知道谁负责什么，提问时能给出针对性的上下文 | （仅内部使用，含敏感信息需控制权限） |

**引入时机**：中期开始，和知识库建设同步推进。先建项目级 memory，再逐步扩展到个人信息。

#### 能力引入节奏

```
当前 ──────── 中期(3-6月) ──────── 长期(6-12月)
   Rule           Skill               Memory + MCP
   (立即可做)      (可复用工作流)       (深度项目理解 + 连接业务系统)
```

## 三、配套学习计划（简要）

> 学习与上述 Agent 能力的引入同步推进：学一个、落地一个、验证一个。

### 第一阶段：基础使用（基本完成）

- 团队已在使用 Qoder IDE 插件进行对话和代码修改，此阶段大部分目标已达成
- 补充动作：提示工程培训（从随便问到结构化提示词）、建立首批 Rule（3-5 条编码/安全规范）、搭建 Prompt 模板库雏形

### 第二阶段（3-6 个月）：Agent 能力落地

- **Skill**：选 1-2 个高频重复场景（如后端 CRUD 生成、前端 Vue 页面生成），编写 Skill 模板并试点
- **Memory**：梳理项目架构和业务术语，写入项目 memory，让 AI 持续理解上下文
- 搭建内部知识库雏形（Dify/FastGPT），与 Memory 互补——知识库管"是什么"，Memory 管"AI 应该知道什么"

### 第三阶段（6-12 个月）：深度集成

- 了解 MCP（让 AI 连接外部系统的协议）
- 试点：AI 通过 MCP 连接内部系统，进行查单、报表生成等操作
- Skill + MCP 组合：例如"查订单状态" Skill → AI 自动调用 ERP 的 MCP 接口 → 返回结果
- 从"用 AI 写代码"升级到"AI 是系统的入口"

## 五、AI 的局限性与风险认识

> 在积极推动 AI 应用的同时，也需要对 AI 的局限性有清醒认识。这不是阻碍 AI 转型的理由，而是为了让 AI 应用走得更稳。

### 5.1 大模型本身的局限性

| 风险点 | 表现 | 应对措施 |
|--------|------|---------|
| **幻觉（Hallucination）** | AI 会自信地编造不存在的 API、配置项、历史事实 | 后端代码涉及数据库操作、金额计算时必须人工验证；引用外部信息时要求 AI 注明来源 |
| **奉承/迎合（Sycophancy）** | AI 倾向于同意用户的观点，即使用户是错的 | 不要把 AI 当权威；关键决策点反向提问"这个方案有什么问题？"让 AI 给出反对意见 |
| **上下文丢失** | 长对话中 AI 会"忘记"早期设定的规则和约束 | 重要约束通过 Rule 固化；复杂任务拆分为多个短会话；定期让 AI 复述当前理解的任务目标 |
| **知识截止** | AI 的知识有截止日期，不了解最新的框架版本和安全漏洞 | Java/Spring Boot 版本升级、安全补丁等不能仅依赖 AI，需人工核对官方文档 |

### 5.2 Agent 自动化的风险

| 风险点 | 表现 | 应对措施 |
|--------|------|---------|
| **操作不可逆** | Agent 如果具备自动执行能力（如操作数据库、修改文件），误操作后果严重 | Agent 初期仅限"建议模式"（只输出方案不自动执行）；涉及数据库写操作、文件删除等必须人工确认 |
| **连环错误** | Agent 的错误输出被下一个 Agent 或步骤放大 | 关键环节设置人工检查点；不要让 AI 的输出直接成为另一个 AI 的唯一输入 |
| **权限失控** | MCP 连接业务系统后，AI 的权限边界可能模糊 | 按最小权限原则配置 MCP 服务；区分"查询"和"操作"两类接口，操作类必须二次确认 |

### 5.3 团队层面需注意的问题

- **过度依赖**：AI 辅助不等于 AI 替代，核心业务逻辑的理解和设计仍需亲自掌握
- **代码质量稀释**：AI 生成代码量大但可能风格不一致、存在隐蔽 Bug，必须 Review
- **技能退化**：长期依赖 AI 写代码可能导致排查深层问题的能力下降，复杂 Bug 仍需手工分析能力

## 六、展望——从研发到全公司

1. **内部知识政策问答智能体**：整合公司制度、部门文档、技术规范，员工自然语言提问即答
2. **业务系统 AI 入口**：为 ERP/WMS/HRM 等系统扩展 MCP 服务，AI 智能体可直连业务系统进行查询、审批、报表等操作
3. **从研发推广到全公司**：研发先行验证 → 形成可复制方案 → 推广到运营、仓储、人事等部门
