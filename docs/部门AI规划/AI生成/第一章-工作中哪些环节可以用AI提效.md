# 一、当前工作中有哪些环节可以用 AI 提效

> 本章回答"哪些工作能结合 AI 提升效率"——结合研发部门现有技术栈（Vue + Spring Boot）和业务系统（ERP/WMS/HRM），按日常工作场景逐一梳理 AI 可以介入的环节。

---

## 1.1 日常编码

部门前后端岗位分离，且日常工作中已上线系统的维护（修 Bug、功能优化）的占比较高，新功能/新系统开发相对较少，故分开列出。

> **当前基础**：团队成员已在使用 Qoder 的 VS Code / IDEA 插件进行代码对话和修改。以下场景均基于此基础展开。

### A. 已上线系统维护（占比高，重点）

已上线系统的维护工作有几个特点：代码可能几经人手、文档不全、改动必须谨慎避免影响现有功能。AI 在以下环节可以提供有效辅助。

| 场景 | 岗位 | 现状痛点 | AI 如何介入 | 使用方式 |
|------|------|---------|------------|---------|
| Bug 修复 | 后端 | 看日志、跟踪调用链耗时，老代码不熟悉 | 贴报错日志 + 相关代码 → AI 分析根因并给出修复建议 | IDE 插件中选中报错方法和日志，直接对话 |
| Bug 修复 | 前端 | 页面异常但难以定位是数据问题还是渲染问题 | 贴截图描述现象 + 组件代码 → AI 辅助定位 | 将页面异常截图和描述粘贴到 AI 对话框 |
| 功能优化/小需求 | 后端 | 在现有逻辑上加新字段、改业务规则，怕改出问题 | AI 理解现有代码逻辑后给出最小改动方案 | 把现有代码贴给 AI，说明"在这个基础上增加/修改什么" |
| 功能优化/小需求 | 前端 | 列表加筛选项、表单加字段等高频小改动 | AI 在现有页面基础上生成增量代码 | 贴现有组件代码，描述要新增的筛选项或表单字段 |
| 数据修复/脚本执行 | 后端 | 生产数据出问题，需要写临时 SQL 或脚本修复，手写容易出错 | AI 根据问题描述生成数据修复 SQL/脚本，人工审核后执行 | 描述数据异常的表和字段，说明修复目标，AI 输出 SQL |
| 旧代码理解 | 通用 | 接手别人写的/几年前的代码，缺少文档 | 选中代码 → AI 解释逻辑、画出调用关系 | IDE 中选中代码块，让 AI 解释"这段代码做了什么" |

**重点场景说明**：

**Bug 修复（后端）** — 这是日常工作中频率最高的场景。以 ERP 系统为例，典型流程是：

1. 收到 bug 反馈（如"销售单审核后库存没扣减"）
2. 查看日志定位到相关 Service 方法
3. 将报错日志 + Service 层代码贴给 AI
4. AI 分析调用链和事务边界，指出可能的根因
5. 开发者根据分析结果决定修复方案

这里 AI 的价值不在于"自动修 Bug"，而在于快速缩小排查范围——尤其是代码经过多人维护、没有人完全熟悉全貌的情况下。

**数据修复/脚本执行** — 后端特有的高频场景。生产环境偶发数据异常（如某单据状态卡在中间态），需要写一次性修复脚本。这类脚本的难点不在于逻辑复杂，而在于"不能出错"——写错一行可能导致二次数据污染。让 AI 生成脚本初稿、人工审核确认后执行，可以降低手写的出错概率。

### B. 新功能 / 新系统开发

新功能开发的特点是模板代码多、重复性高。AI 在减少机械敲码方面效果明显。

| 场景 | 岗位 | 现状痛点 | AI 如何介入 | 使用方式 |
|------|------|---------|------------|---------|
| 新页面开发 | 前端 | Vue 列表页、表单页大量模板代码 | AI 根据需求描述生成页面骨架，人工调样式和逻辑 | 描述页面功能（如"带分页和搜索的列表页，字段有a/b/c"），AI 生成 Vue 组件 |
| 新接口开发 | 后端 | Controller/Service/Mapper 三层重复代码 | AI 根据表结构或注释生成全套接口代码 | 提供表名和字段，说明"生成带分页查询+新增+修改的 REST 接口" |
| SQL 编写 | 后端 | 复杂报表查询、多表联查调试耗时长 | 自然语言描述需求 → AI 输出 SQL | 描述查询需求（"查出上月各供应商的采购金额汇总，按金额降序"）|
| 批量数据处理 | 后端 | ERP/WMS 中常有批量导入导出、数据同步任务，逻辑繁琐 | AI 根据字段映射和业务规则生成处理代码 | 提供 Excel 模板字段和数据库表字段的对应关系 |
| 定时任务开发 | 后端 | 库存同步、订单状态刷新等定时任务模板代码多 | AI 生成定时任务骨架 + 异常处理 + 日志 | 描述定时任务的触发条件和业务逻辑 |
| 第三方接口对接 | 后端 | 对接物流、支付等外部 API，文档全是英文/格式不一 | AI 根据对方文档生成对接代码和异常处理 | 将对方 API 文档贴给 AI，说明需要对接哪些接口 |
| 前端组件开发 | 前端 | 公共组件封装费时，重复写类似的逻辑 | AI 辅助生成可复用组件框架 | 描述组件的输入输出和行为，AI 生成组件骨架 |

**重点场景说明**：

**新接口开发（后端）** — 对于一个新数据表，从 Controller → Service → Mapper 三层代码的编写高度模板化。以 WMS 出库单为例，只需告诉 AI：

- 表名：`wms_outbound_order`
- 主要字段：`order_no`, `status`, `customer_name`, `warehouse_code`, `create_time`
- 需要：分页查询 + 按状态筛选 + 新增 + 修改状态

AI 即可生成 Controller（`@RestController` + 参数校验）、Service（业务逻辑 + 事务）、Mapper（`@Select` + 动态 SQL）的完整代码。开发者需要做的是补充业务特有的校验逻辑。

**第三方接口对接（后端）** — 跨境电商场景下，经常需要对接物流（如 DHL、FedEx）、支付等外部服务的 API。这些 API 文档通常是英文、格式各异。将文档片段贴给 AI，它能：

- 解析请求参数和响应格式
- 生成 HTTP 调用代码（含异常处理和重试逻辑）
- 生成对应的 DTO 类

开发者只需核对关键参数和异常分支。

## 1.2 代码质量

> 部门无单元测试（由专门测试岗位进行黑盒测试），且不建议对老代码进行大面积重构。此处聚焦可行性高的事项。

| 场景 | 现状痛点 | AI 如何介入 | 使用方式 |
|------|---------|------------|---------|
| 代码审查 | 人工审查耗时长，容易漏掉安全问题 | AI 审查代码变更，标注潜在 Bug、SQL 注入、空指针、异常处理遗漏等风险点 | 提交代码前，将有变更的文件贴给 AI，让其按检查清单审查 |

**重点场景说明**：

代码审查是目前最容易落地且收益直接的场景。当前部门代码审查依赖人工，审查者的精力和经验决定审查质量。AI 可以作为"第一道审查"：

- **SQL 注入检查**：扫描所有 Mapper 层，标记拼接 SQL 的位置（这是人工审查最容易漏的）
- **空指针风险**：标记未做 null 判断的链式调用（如 `dto.getXxx().getYyy()`）
- **异常处理遗漏**：检查 Service 方法是否有 try-catch 包裹关键操作（如数据库写操作后的库存同步）
- **事务边界**：标记缺少 `@Transactional` 注解的写操作组合

AI 不替代人工审查，而是让审查者把精力集中在业务逻辑的正确性上，安全性和代码规范层面的问题交给 AI 初筛。

## 1.3 问题排查

| 场景 | 岗位 | 现状痛点 | AI 如何介入 | 使用方式 |
|------|------|---------|------------|---------|
| Bug 定位 | 后端 | 看日志、跟踪调用链耗时，尤其涉及多服务/多表操作时 | 贴报错日志 + 相关代码 → AI 分析根因并给出修复建议 | 将完整异常堆栈 + 出错的 Service/Mapper 代码一并贴入 |
| Bug 定位 | 前端 | 页面异常但难以判断是数据问题还是渲染问题 | 贴截图描述现象 + 组件代码 → AI 辅助定位 | 描述操作步骤 + 截图 + 浏览器控制台报错 |
| 慢接口/SQL 优化 | 后端 | 接口越来越慢，不知道是代码问题还是 SQL 问题 | 贴 SQL 执行计划 + 接口代码 → AI 分析瓶颈并给出优化方案 | 贴 `EXPLAIN` 结果 + Mapper XML/注解 SQL |
| 数据库死锁/锁等待 | 后端 | 并发场景下偶发死锁，日志难读 | 贴死锁日志 → AI 解析锁冲突链路 → 给出事务调整建议 | 贴 MySQL `SHOW ENGINE INNODB STATUS` 的死锁段 |
| 环境/部署问题 | 通用 | 新人配环境各种报错 | AI 结合报错信息给出针对性解决步骤 | 贴完整报错信息，说明操作系统和软件版本 |

**重点场景说明**：

**慢接口/SQL 优化** — ERP 报表和查询接口随着数据量增长会越来越慢。传统做法是 DBA 或者资深开发手工分析 `EXPLAIN` 结果。AI 可以：

1. 接收 SQL 语句 + `EXPLAIN` 输出
2. 判断是全表扫描、索引失效还是联表过多
3. 给出优化建议（加索引、改写 SQL、拆分为多次查询、代码层加缓存）

**数据库死锁** — 跨境电商系统中，库存扣减、订单状态更新等并发操作偶发死锁。死锁日志（`LATEST DETECTED DEADLOCK`）结构复杂、可读性差。将死锁段贴给 AI，它可以解析出：
- 哪两个事务互相等待
- 等待的是哪张表的哪个锁
- 建议的事务调整（如缩小事务范围、统一加锁顺序）

## 1.4 需求理解与上线支持

> CI/CD 脚本编写频率低，不是日常高频场景。此处聚焦更常见的需求分析和上线相关环节。

| 场景 | 现状痛点 | AI 如何介入 | 使用方式 |
|------|---------|------------|---------|
| 需求文档理解 | 需求文档描述模糊、缺少边界条件说明，开发时需要反复找产品确认 | AI 通读需求文档 → 梳理业务流程 → 列出潜在的边界情况和逻辑冲突 | 将需求文档或 PRD 贴给 AI，让它从开发角度提问 |
| 任务拆分 | 拿到需求后需要拆分为可执行的开发子任务，新人容易遗漏步骤 | AI 根据需求生成前后端任务清单，标注依赖关系 | 描述需求和现有系统结构，让 AI 输出拆分方案 |
| 上线检查清单 | 每次上线靠经验回忆检查项，容易遗漏 | AI 根据本次变更内容生成针对性的上线检查清单 | 贴出 SVN 提交记录，让 AI 生成检查项 |
| 监控告警初筛 | 半夜告警要爬起来查，压力大 | AI 先做第一轮分析，给出初步判断 | 将告警信息和相关日志贴给 AI 做初筛 |

**重点场景说明**：

**需求文档理解** — 跨境电商业务规则复杂（如不同国家的税率、物流商规则、库存计算方式），需求文档往往只写了主流程，边界情况需要开发自己补。将需求文档贴给 AI，附加"这是跨境电商 ERP/WMS 系统的一个需求"，AI 可以：

- 梳理完整的业务流程（含异常分支）
- 列出需求文档中没有明确提到的边界条件
- 从数据一致性角度提问（如"如果扣减库存时并发怎么办？"）

这不是替代产品经理，而是让开发在动手之前想得更完整。

## 1.5 其他日常工作

| 场景 | 现状痛点 | AI 如何介入 | 使用方式 |
|------|---------|------------|---------|
| 技术调研 | 搜索引擎 → 看博客 → 比较方案，一下午没了 | AI 辅助搜索、对比、生成调研摘要 | 描述调研需求，让 AI 列出主流方案及优劣对比 |
| 周报/日报 | 纯手工回忆写了什么 | 结合 SVN 提交记录 + AI 润色 | 贴 SVN 提交记录，让 AI 整理为周报格式 |
| 会议纪要 | 开会时记笔记分心 | 录音 → AI 转文字 → 自动提取待办事项 | 录音转文字工具 + AI 提取关键决策和待办 |

## 本章小结

以上场景覆盖了研发部门日常工作的主要环节，共 5 个大类：

| 章节 | 覆盖内容 | 关联岗位 |
|------|---------|---------|
| 1.1 日常编码 | 维护期 Bug 修复/功能优化 + 新功能开发 | 后端 / 前端 |
| 1.2 代码质量 | AI 辅助代码审查 | 通用 |
| 1.3 问题排查 | Bug 定位、慢 SQL 优化、死锁分析、环境问题 | 后端为主 |
| 1.4 需求理解与上线支持 | 需求文档分析、任务拆分、上线检查、告警初筛 | 通用 |
| 1.5 其他日常工作 | 技术调研、周报、会议纪要 | 通用 |

按引入优先级，建议从以下顺序推进：

1. **最先做**（零成本，当前即可）：Bug 修复与定位、旧代码理解、SQL 编写、需求文档理解 —— 团队已经在用 Qoder 做类似的事，只需有意识地系统化
2. **优先做**（少量规范即可）：代码审查、任务拆分、上线检查清单 —— 建立审查检查清单和 Prompt 模板
3. **逐步做**（需要搭建）：监控告警初筛 —— 需要与运维配合接入日志
