# myAgent

这是我学习 Agent 的历程，从一个聊天机器人 demo 开始。

以下是我自己的一些理解：

---

## 智能体 Agent

其实就是由所有不需要智能的部分构成。

> **❌ 错误标注**
>
> "由所有不需要智能的部分构成"这个定义不准确。Agent 的核心恰恰是需要 LLM 提供"智能"来驱动。
>
> ✅ **正确理解：** Agent 是由 **LLM（大脑）+ 工具（手脚）+ 编排逻辑（决策循环）** 组成的系统。它本身不是"不需要智能"，而是把 LLM 的智能与外部工具能力结合起来，让 LLM 能"动手做事"。

### LLM

大语言模型，无状态的函数调用（输入 → 输出），只能输入和输出，本质上就是一句一句的词语接龙。但是为了尽量让模型回答的内容符合自己的需求，所以就会尽量在一次提问中包含 LLM 需要用到的所有内容，这就是**提示词**。

> **⚠️ 小修正**
>
> "词语接龙"这个比喻基本可用，但不够精确。LLM 做的是 **token 预测**——根据前面的所有 token 预测下一个最可能的 token，然后不断重复。它不是像接龙那样"接上一个词"，而是基于概率分布生成下一个词。

### Prompt（提示词）

比如：我在南京旅游，但是预算只有 10 块钱了，帮我规划一个旅游路线。

可以区分为**背景信息**、还有**最终的指示**。于是，又一个词出现了。

### Context（上下文）

之前说过，LLM 本身是无状态的函数调用（输入 → 输出）。那如何能进行追问呢？那么就把每次历史内容放到上下文部分，作为上下文信息，然后再提示最终问题，于是就伪装成了多轮对话，就实现了追问。这些特殊的上下文信息，就是

没有"对话"概念。多轮对话是上层应用把历史消息拼进 prompt 模拟出来的，LLM 本身不记得任何事。每次调用都是独立的。

> **❌ 错误标注**
>
> 第 25 行"这些特殊的上下文信息，就是"这句话断了，没有说完，缺少结论。另外第 27 行的"没有'对话'概念"开头突兀，与上文缺少过渡。
>
> ✅ **正确理解：** 应该是"这些特殊的上下文信息，就是 **Context（上下文）**。"并且 LLM 本身确实没有"对话"概念——多轮对话是上层应用把历史消息拼进 prompt 模拟出来的，LLM 本身不记得任何事，每次调用都是独立的。

### Memory（记忆）

这些记忆可以再次调用 LLM 对上下文进行总结，从而对记忆进行压缩，从而减少上下文的长度。

---

## 从工具到智能体

现在就是一个可以进行追问的工具了。但是有个新问题，就是无法上网。

LLM 只能进行词语接龙，无法操作电脑，所以可以告诉 LLM 需要上网的时候告知，用户查阅后再告诉 LLM。这样有点慢。

于是为了加快效率，可以把帮忙查资料的过程写成一段程序，当遇到需要搜索请求的部分由这个程序与 LLM 进行交互，但是在用户看来，还是在一问一答，而且还能查阅资料了。这个程序就叫**智能体 Agent**，这就是早期的智能体。

```text
用户 ──"今天有哪些新闻"──> Agent ──> LLM
                                LLM ──"需要上网查"──> Agent
                                          Agent ──上网搜索──> 拿到结果
                                Agent ──搜索结果──> LLM
                                LLM ──整理好的新闻──> Agent
用户 <──"今天的新闻是……"── Agent
```

---

## RAG（检索增强生成）

智能体不仅能查询网上资料，还可以调用本地知识库，从而丰富提示词，这就是 **Retrieval-Augmented Generation（检索增强生成）**。

> **⚠️ 小修正**
>
> RAG 不仅仅是"丰富提示词"，它的核心价值是让 LLM 能基于 **私有知识/实时数据** 生成回答，解决 LLM 训练数据过时和无法访问私有数据的问题。

从知识库中检索相关文档片段 → 拼入 prompt → 让 LLM 基于这些片段生成回答：

1. 有一个知识库（你的文档、PDF、笔记等，提前切片 + 向量化存储）
2. 用语义检索找到相关片段（不是关键词匹配，是理解语义后找最相关的）
3. 把检索到的片段拼入 prompt，让 LLM 基于这些内容回答

RAG 是 Agent 调用一个特定的工具（知识库语义检索），把检索结果拼入 prompt，让 LLM 基于你的私有知识回答问题。

---

## 人与 Agent 的关系

```text
人 - Agent - LLM
```

Agent 可以用来减少人直接与 LLM 沟通的次数，完成一些 LLM 无法做到的事情，比如上网，或者工具的调用。

---

## Function Calling

但是聚焦于大模型与 Agent 对话来看，这部分的内容肯定不全是自然语言进行沟通，否则程序的实现太复杂。所以这种 Agent 与 LLM 之间的对话就约定了一种格式，让 LLM 以指定的、死板的格式来回复，类似于前后端开发约定的接口文档。

这种 Agent 和 LLM 之间关于工具调用约定的格式叫做 **Function Calling**。

所以到目前为止，智能体的含义大概就是：

```text
人 <-自然语言-> 智能体[搜索，文件读取、脚本执行、PDF转换…] <-Function Calling/自然语言-> LLM
```

---

## MCP（模型上下文协议）

再看看这部分 `智能体[搜索，文件读取、脚本执行、PDF转换…]`，里面的各种工具现在是写在主程序里面的。如果把这部分内容进行解耦——把工具的部分单独写一个服务，再由 Agent 调用这些服务，比如用 `tools/list` 等方法获取工具列表，`tools/call` 来调用工具。也就是一套约定：

```text
[搜索，文件读取、脚本执行、PDF转换…] <-tools/list;tools/call-> 智能体 <-Function Calling/自然语言-> LLM
```

那么这些约定就是：**MCP（Model Context Protocol，模型上下文协议）**，其实就是能提供各种工具的程序集。这个时候中间的 Agent 就是个传话筒。

> **❌ 错误标注**
>
> "其实就是能提供各种工具的程序集"这个描述不准确。MCP 是一个 **通信协议/标准**，不是"程序集"。程序集是一组代码的集合，而协议是一套通信规范。
>
> ✅ **正确理解：** MCP 是一套 **标准化的通信协议**，定义了 Agent（Client）与工具服务（Server）之间如何通信。它规定了 `tools/list`、`tools/call` 等标准接口，使得任何工具服务只要遵循这个协议，就能被任何支持 MCP 的 Agent 调用。

```text
工具[搜索，文件读取、脚本执行、PDF转换…] <-MCP-> 智能体 <-Function Calling/自然语言-> LLM
```

MCP：把工具从 Agent 主程序里解耦出来，变成独立的服务，用一套标准协议通信，这套协议就是 MCP。

```text
工具服务 <--MCP协议--> Agent <--Function Calling协议--> LLM
  (标准化左边)           (传话筒)        (标准化右边)
```

MCP 里 Agent 叫 **Client**，工具服务叫 **Server**，和你熟悉的 C/S 架构一样：

| MCP Server（工具提供方） | MCP Client（Agent） |
| --- | --- |
| 搜索服务 | 调 `tools/list` 获取有哪些工具 |
| 文件服务 | 调 `tools/call` 执行具体工具 |
| PDF 服务 | 把结果拼进 prompt 发给 LLM |

此时，大模型就像个只会说话不会干活儿的智者，MCP 服务就是一个可以提供各种工具服务的程序集，中间的 Agent 就是个传话筒——把大模型的话转化为调用工具的代码，把工具的调用结果原封不动地传给大模型，同时还要给我们传话。

## 用户交互形式

所以，现在聚焦于我们和 Agent 之间，虽然底层还是对话，但是形式可以丰富多彩，可以是类似于 CLI 的命令行，也可以是可视化的窗口，比如 ClawBoot 等。

> **⚠️ 小修正**
>
> "ClawBoot" 这个名称不太明确，可能是指 **Claude**（Anthropic 的产品）、**ChatGPT**、或者其他 AI 工具界面。如果是泛指可视化 AI 对话窗口，建议改为更通用的表述。

---

## Workflow（工作流）

但是现在，这些智能体还有一些缺点。比如我们想让智能体读取文档，并生成总结，最后保存成一个新的 MD 文档。

如果我们同时提示词输入，此时 Agent 会自己开始策划完成这个任务。但是这个流程其实相对稳定，如果每次 Agent 都重新开始策划完成这些任务，其实很消耗 token 并且输出结果不稳定。这整个流程中，提取 PDF 和保存 MD 文件其实都可以固化成脚本，中间的总结由大模型完成即可，整个流程就不需要智能体插手了。

要固化这样的流程，可以通过编程的方式来实现，这个时候有一个开源的框架叫做 **LangChain**；但是这种的门槛有点高，于是又有一种可视化的低代码平台，叫 **Workflow**。

> **❌ 错误标注**
>
> 这里有两个问题：
> 1. **LangChain** 是一个 Python/JavaScript **编程框架**，不是"门槛有点高"的对比对象——它本身就是通过编程来实现流程固化的工具。
> 2. **"Workflow"** 这个词太泛了，它不是一个具体产品名称。应该是指某个具体的工作流平台。
>
> ✅ **正确理解：** 固化流程的方式有：
> - **编程框架**：如 LangChain、LlamaIndex 等，用代码定义流程
> - **可视化低代码平台**：如 **Dify**、**Coze（扣子）**、**FastGPT** 等，用拖拽方式定义工作流
>
> 这两者是不同层级的工具，不应该放在一起做"门槛高低"的对比。

---

## Skill（技能）

但是如果你不只想要能够提取 PDF，你还想要提取 Word、Excel 各种形式的，输出的格式可能是 MD、HTML、PDF 等。这个时候肯定不能写一堆 if，最好是能够以自然语言的方式触发这个任务。

于是你就把这些转换提取脚本文件放到一起，最后再写个说明文档把整体的流程描述清楚告诉 Agent 根据文件格式灵活选取脚本。所以这个时候你再想要执行上面的任务的时候在任务前面加上一段话，先读取指定文件夹下面的内容，然后再按照要求完成任务，这样就既保证了一定的灵活性，同时也变得可控。

又来了优化空间了——我们可以提前约定一个位置来存储刚才写的内容，然后在 Agent 中写死一段程序在执行前读取这个位置，这样每次就不用加上一堆提示词了。其实看起来就像是把提示词换了个位置存储，这个就叫 **Skill**。

> **⚠️ 小修正**
>
> Skill 的概念描述基本正确，但可以更精确：Skill 不仅仅是"换个位置存储提示词"，它通常是 **提示词 + 工具调用脚本 + 使用说明** 的组合包，让 Agent 能够按需加载和执行特定能力。

---

## SubAgent（子智能体）

但是还有个问题。对于一个复杂的任务，一个 Agent 的上下文会非常大。于是就可以将子任务在一个新的 Agent 里执行，就是 **SubAgent**，其实就是做了上下文隔离。

---

## 总结

```text
工具服务 <--MCP协议--> Agent <--Function Calling--> LLM
                         |
                  Skill（提示词 + 脚本 + 说明文档）
                  Workflow（固定流程，不需要 Agent 规划）
                  SubAgent（上下文隔离，子任务独立执行）
```
